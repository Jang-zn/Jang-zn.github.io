## 사용자를 강제하지 않고 유도하는 것

명령 대신 경로를 설계한다. 이게 넛지 사이클의 핵심이다.

---

## Nudge (넛지)

**정의**: 선택을 강제하지 않으면서 특정 행동을 더 자연스럽게 선택하도록 환경을 설계하는 것.

행동경제학에서 나온 개념. 리처드 탈러가 2008년 저서 *Nudge*에서 체계화했다. 카페테리아에서 건강식을 눈높이에 두고 정크푸드를 낮은 선반에 두면 선택이 바뀐다. 강제가 아니다.

앱 UX에서 넛지는 주로 완료 시점 분기로 구현된다.

```
행동 완료
  → 아무 변화 없음: 그냥 종료
  → 변화 감지됨: "이렇게 저장할까요?" 노출
```

"저장 버튼을 누르세요"가 강제라면, "변경이 있으면 완료 시 알아서 제안"이 넛지다.

---

## Inline Editing (인라인 편집)

**정의**: 별도 편집 페이지로 이동하지 않고, 콘텐츠를 표시하는 그 자리에서 바로 수정할 수 있는 패턴.

컨텍스트 전환을 제거한다는 게 핵심이다. 수정하려고 다른 페이지로 가면 현재 맥락이 끊긴다. 다시 돌아왔을 때 어디까지 봤는지 잃어버린다.

인라인 편집이 적합한 상황:
- 수정 단위가 단순하고 작을 때 (수량 하나, 텍스트 한 줄)
- 수정이 원래 보던 화면의 연장선일 때
- 수정 후 전체 상태를 바로 확인해야 할 때

인라인 편집이 부적합한 상황:
- 수정 항목이 많고 복잡할 때 (폼 전체)
- 현재 화면과 별도 맥락의 수정일 때

---

## Fitts's Law

**정의**: 타겟까지 이동하는 시간은 거리에 비례하고, 타겟 크기에 반비례한다.

```
T = a + b × log₂(D/W + 1)
```

- T: 이동 시간
- D: 타겟까지의 거리
- W: 타겟의 너비 (클수록 빠름)

모바일 터치 UI 함의:
1. **크게 만들어라**: 탭 타겟 최소 44×44px (Apple HIG), 안드로이드는 48×48dp
2. **가까이 배치해라**: 연관된 액션은 콘텐츠 바로 옆에
3. **요리 중에는 더 크게**: 손에 뭔가 든 상태, 서 있는 상태에서 조작한다

Fitts's Law를 어기면 조작 오류가 늘고, 포기율이 올라간다.

---

## 넛지 사이클 설계 방법

1. **트리거 정의**: 언제 넛지를 띄울 것인가 (변경 감지, 일정 시간 경과 등)
2. **Path 분기**: 변화 없음 경로와 변화 있음 경로를 명확히 나눔
3. **넛지 메시지**: 맥락을 포함한 구체적 제안. "저장하시겠습니까?" 대신 "이번에 소금을 줄였는데, 새 버전으로 저장할까요?"
4. **부담 제거**: 넛지는 제안이지 강제가 아님. 거절해도 불이익 없음

## TasteNote 적용

요리 모드가 처음엔 읽기 전용이었다. 요리 중에 레시피를 바꾸고 싶으면 모드를 나가서 수정 페이지로 가야 했다. 흐름이 끊기니까 아무도 안 했다.

인라인 편집으로 바꿨다. 재료를 탭하면 상태가 순환한다 (사용함 → 조정함 → 건너뜀). Fitts's Law 고려해서 체크 타겟을 56px로. 단계에는 메모 토글을 붙였다.

요리 완료 시 변경이 감지되면 "새 버전으로 저장할까요?" 넛지가 뜬다. 변경 없으면 그냥 완료. 이 분기 처리가 넛지 사이클이다.

기존에 있던 `CookingReviewSheet`(완료 후 별점+메모 입력)는 삭제했다. 인라인 편집을 이미 했는데 완료 후 또 메모를 받으면 같은 질문을 두 번 하는 거다.
