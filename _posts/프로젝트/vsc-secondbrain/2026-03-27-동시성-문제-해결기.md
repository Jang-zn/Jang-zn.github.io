## 문제의 시작

VS Code를 여러 개 열면 확장도 여러 개 뜬다. 각 인스턴스가 같은 `~/.claude/projects/`를 감시한다. 같은 대화를 두 번 요약하면 중복 노트가 생기고, Gemini API 호출도 낭비다.

한 창에서 작업하면 문제가 없다. 프로젝트 2~3개 동시에 열어놓는 순간 터진다. 각 확장 인스턴스가 독립 프로세스로 돌아가니까 서로의 존재를 모른다. 같은 냉장고를 여러 명이 정리하는데 서로 말을 안 하는 상황이다.

## inFlight의 한계

처음에는 `inFlight`라는 Set으로 막았다. 파일 경로를 넣어두고, 처리 중이면 스킵.

한 인스턴스 안에서 같은 파일을 두 번 처리하는 건 막을 수 있다. 근데 인스턴스 간에는? `inFlight`는 프로세스 메모리에 있다. A 창의 `inFlight`에 들어간 파일 경로를 B 창이 볼 수 없다.

화장실 칸에 잠금장치가 없는 거다. 안에 사람이 있는지 밖에서 모른다. 문을 열어봐야 안다. 근데 여는 순간 이미 늦었다.

## Claude Code에 보안 감사 요청

"레이스 컨디션 전부 찾아봐"라고 시켰다. 6개 취약점을 찾아왔다.

**1. state.json read-modify-write 비원자적**
state.json을 읽고, 메모리에서 수정하고, 다시 쓴다. 두 인스턴스가 동시에 읽으면 각자 다른 버전을 수정하게 된다. 나중에 쓴 놈이 먼저 쓴 놈의 변경을 덮어쓴다. Lost Update.

**2. shouldProcess() 체크 vs 실제 처리 사이 갭**
shouldProcess()에서 "이 파일 처리해야 함"이라고 판단했다. 근데 실제로 처리를 시작하기 전에 다른 인스턴스가 이미 처리를 끝낼 수 있다. 체크 시점과 처리 시점이 다르다.

**3. markProcessed() Lost Update**
두 인스턴스가 다른 파일을 처리하더라도, 둘 다 state.json에 쓴다. 동시에 markProcessed()를 호출하면 한쪽 기록이 사라진다.

**4. NoteWriter 파일명 충돌 (existsSync TOCTOU)**
처음 구현에서 `existsSync()`로 파일 존재 여부를 체크하고, 없으면 `writeFileSync()`로 생성했다. 체크와 생성 사이에 다른 인스턴스가 같은 파일을 만들 수 있다. 전형적인 TOCTOU(Time-of-Check to Time-of-Use).

**5. stale getPreviousNoteFiles()**
이전 처리에서 생성된 노트 파일 경로를 state.json에서 읽는다. 메모리에 캐싱된 state가 오래됐으면 이미 존재하지 않는 파일을 참조할 수 있다.

**6. inFlight가 프로세스 단위**
위에서 설명한 그 문제. 프로세스 간 가시성이 없다.

리포트가 상당히 정확했다. 6개 중 5개가 실제로 문제를 일으킬 수 있는 것들이었다. Claude Code가 동시성 분석을 잘한다는 걸 여기서 확인했다.

## 해결 1: 파일 락 도입

프로세스 간 동기화가 필요하면 파일 시스템을 쓰는 게 가장 단순하다. `~/.vsc-secondbrain/locks/` 아래에 락 파일을 만들기로 했다.

파일 경로를 SHA256으로 해싱해서 락 파일명으로 쓴다. 락 파일 안에는 PID와 타임스탬프를 JSON으로 기록한다.

```typescript
private lockFileFor(filePath: string): string {
  const hash = crypto.createHash('sha256')
    .update(filePath).digest('hex').slice(0, 16);
  return path.join(LOCKS_DIR, `${hash}.lock`);
}
```

120초가 지나면 stale로 판정한다. 확장이 비정상 종료하면 락이 남을 수 있으니까. PID로 프로세스 생존 여부도 체크한다. `process.kill(pid, 0)`은 시그널을 보내지 않고 프로세스 존재만 확인한다. 죽은 프로세스의 락은 바로 정리된다.

## TOCTOU 버그

여기서 진짜 문제가 나왔다.

처음 구현은 이랬다.

1. `mkdirSync()`로 락 디렉토리 생성 (디렉토리 자체가 락)
2. 디렉토리 안에 `meta.json` 쓰기 (PID, 타임스탬프)

이게 2단계다. 문제가 보이나.

A 인스턴스가 `mkdirSync()` 성공. 아직 `meta.json`을 안 썼다. B 인스턴스가 락을 확인한다. 디렉토리는 있는데 `meta.json`이 없다. "meta.json이 없으니 stale 락이네" 하고 디렉토리를 삭제한다. B가 새로 락을 잡는다. A도 이미 `mkdirSync()`가 성공했으니 자기가 락을 잡았다고 생각한다.

결과: 둘 다 락을 잡았다고 생각한다. 중복 처리 발생.

실제로 창 4개 띄워놓으면 중복 노트가 3개 생기는 현상이 있었다. 매번은 아니다. 레이스 윈도우가 짧아서 타이밍이 맞아야 뚫린다. 10번 중 2~3번. 근데 2~3번이면 충분히 문제다.

해결은 간단했다. `openSync('wx')`로 한 단계로 줄이면 된다.

```typescript
const fd = fs.openSync(lockFile, 'wx');
fs.writeFileSync(fd, JSON.stringify(meta), 'utf-8');
fs.closeSync(fd);
```

`'wx'`가 핵심이다. `O_WRONLY | O_CREAT | O_EXCL`. 파일이 이미 존재하면 `EEXIST` 에러를 던진다. 파일 생성과 존재 확인이 커널 레벨에서 원자적으로 일어난다. 2단계가 1단계가 됐다. TOCTOU가 구조적으로 불가능해졌다.

원자적 연산 하나가 2단계 연산보다 낫다. 당연한 얘기인데, 직접 겪어야 체감된다.

## 해결 2: state.json 전용 락

파일별 처리 락과 별개로, state.json 자체도 보호해야 한다.

상황을 보자. A 인스턴스가 `foo.jsonl`을 처리한다. B 인스턴스가 `bar.jsonl`을 처리한다. 서로 다른 파일이니 파일 락은 안 걸린다. 근데 둘 다 처리가 끝나면 state.json에 기록을 쓴다.

A가 state.json을 읽는다. B도 state.json을 읽는다. A가 `foo.jsonl` 기록을 추가하고 저장한다. B가 `bar.jsonl` 기록을 추가하고 저장한다. B가 저장한 버전에는 A의 `foo.jsonl` 기록이 없다. A의 작업 기록이 사라졌다. Lost Update.

`withStateLock` 헬퍼로 해결했다. state.json 전용 락 키를 만들고, 읽기-수정-쓰기를 원자적으로 묶었다.

```typescript
private async withStateLock(mutate: (data: StateData) => void): Promise<void> {
  const locked = await this.stateLock.acquire(STATE_LOCK_KEY, 5_000);
  if (!locked) return;
  try {
    const fresh = this.load();  // 디스크에서 다시 읽기
    mutate(fresh);              // 메모리에서 수정
    this.data = fresh;
    this.save();                // 디스크에 쓰기
  } finally {
    this.stateLock.release(STATE_LOCK_KEY);
  }
}
```

핵심은 `this.load()`를 락 안에서 호출하는 거다. 메모리에 캐싱된 data가 아니라 디스크에서 fresh하게 읽는다. 락 밖에서 읽은 데이터를 수정하면 여전히 Lost Update가 발생할 수 있다.

save()도 원자적으로 한다. tmp 파일에 먼저 쓰고 `renameSync()`로 교체. 쓰다가 크래시해도 state.json이 깨지지 않는다. rename은 대부분의 파일 시스템에서 원자적이다.

## 해결 3: 락 후 stat 재조회

미묘한 문제가 하나 더 있다.

파일 락을 걸기 전에 `stat`으로 mtime을 읽는다. 이 mtime을 기준으로 `shouldProcess()`를 호출한다. 근데 락을 걸기까지 시간이 걸릴 수 있다. 다른 인스턴스가 이미 처리를 끝내고 state.json에 새 mtime을 기록했을 수 있다.

락 획득 전에 읽은 mtime이 stale인 거다. 그래서 락 획득 후에 stat을 다시 조회하고, state도 reload하고, `shouldProcess()`를 한 번 더 체크한다.

```typescript
// 락 획득 후
this.state.reload();
stat = fs.statSync(filePath);  // 재조회
if (!this.state.shouldProcess(filePath, stat.mtimeMs)) {
  return;  // 다른 인스턴스가 이미 처리함
}
```

이중 체크다. 비효율적으로 보일 수 있다. 근데 Gemini API 호출 한 번 아끼는 게 stat 호출 한 번보다 훨씬 비싸다.

## 해결 4: NoteWriter O_EXCL

NoteWriter에서도 같은 패턴을 적용했다.

원래는 이랬다.

```
if (fs.existsSync(path)) → 다른 파일명 생성 → writeFileSync
```

existsSync와 writeFileSync 사이에 다른 인스턴스가 끼어들 수 있다. TOCTOU.

`openSync('wx')`로 바꿨다. 파일이 이미 있으면 `EEXIST`. 이 경우 이미 생성된 파일 경로를 그대로 반환한다. fallback 파일명 생성 로직을 아예 제거했다. 같은 대화를 다른 인스턴스가 이미 저장했으면, 그냥 그 파일을 쓰면 된다.

```typescript
try {
  const fd = fs.openSync(primaryPath, 'wx');
  fs.writeFileSync(fd, content, 'utf-8');
  fs.closeSync(fd);
} catch (err: any) {
  if (err.code !== 'EEXIST') throw err;
  // 이미 존재 — 다른 인스턴스가 만든 것. 정상.
}
return primaryPath;
```

fallback 파일명(`-2`, `-3` 붙이기)을 쓰면 중복은 막을 수 있지만, 같은 내용의 노트가 다른 이름으로 두 개 생긴다. 차라리 먼저 쓴 놈이 이기고, 나중 놈은 조용히 물러나는 게 낫다.

## Claude Code 피드백 루프

6개 취약점 리포트는 정확했다. 분석 능력은 확실하다. 문제는 수정 과정이었다.

처음 수정에서 `mkdirSync` 기반 2단계 락을 만들어왔다. 논리적으로는 맞는 것 같았다. 디렉토리 생성이 원자적이니까 락으로 쓸 수 있다고. 근데 meta.json 쓰기가 분리되어 있어서 TOCTOU가 남아있었다.

"이거 여전히 창 4개에서 중복 생기는데?"라고 피드백했다. 그제서야 문제를 인식하고 `openSync('wx')`로 수정했다. 한 번에 완벽하게 안 된 거다.

동시성 버그는 원래 그렇다. 코드를 읽어서 찾는 건 쉽다. 수정이 정말로 원자적인지 확인하는 건 어렵다. "이 두 줄 사이에 다른 스레드가 끼어들 수 있나?"를 매 줄마다 따져봐야 한다.

교훈이 있다. Claude Code에게 동시성 분석을 시키면 잘한다. 수정도 시키면 대부분 맞다. 근데 "대부분"이 동시성에서는 부족하다. 한 번은 직접 확인하고 피드백을 줘야 한다. 피드백 루프가 없으면 미묘한 TOCTOU가 남는다.

## 정리

6개 레이스 컨디션에 대한 해결책을 요약하면 이렇다.

| 문제 | 해결 |
|------|------|
| inFlight 프로세스 단위 | 파일 기반 락 (FileLock) |
| state.json Lost Update | state.json 전용 락 + withStateLock |
| shouldProcess stale 체크 | 락 후 stat 재조회 + reload |
| NoteWriter TOCTOU | openSync('wx') 원자적 생성 |
| 락 자체의 TOCTOU | mkdirSync 2단계 → openSync('wx') 1단계 |
| stale mtime | 락 획득 후 stat 재조회 |

전부 같은 원칙이다. **체크와 액션을 분리하지 말 것**. 분리하면 그 사이에 누군가 끼어든다. 원자적 연산 하나로 체크와 액션을 동시에 하는 게 답이다.
