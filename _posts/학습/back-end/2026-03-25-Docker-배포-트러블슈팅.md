## 설정은 됐는데 배포가 안 된다

[홈서버에 Docker 환경을 구축](/posts/홈서버-배포와-CI-CD-구축/)해놓고, [실제 배포](/posts/홈서버-실배포-삽질기/)를 하니까 3개가 연속으로 터졌다. 로컬에서 잘 돌던 게 서버에서 안 되는 이유를 정리한다.

## Docker Compose V1 vs V2

Docker Compose에는 V1(docker-compose)과 V2(docker compose)가 있다.

| | V1 | V2 |
|---|---|---|
| 실행 파일 | `docker-compose` (Python) | `docker compose` (Go 플러그인) |
| 설치 | 별도 설치 | Docker 엔진에 플러그인으로 포함 |
| compose 파일 호환성 | 일부 필드 미지원 | 전체 지원 |
| 상태 | deprecated (2023년 EOL) | 현행 |

V1은 Python으로 만들어져서 느리고, compose 파일의 일부 필드(`ContainerConfig` 등)를 파싱하지 못한다. V2는 Go로 재작성돼서 빠르고 최신 스펙을 지원한다.

Ubuntu 서버에 `apt install docker-compose`로 설치하면 V1이 깔린다. V2는 Docker 공식 저장소에서 플러그인으로 설치해야 한다.

교훈: compose 파일을 작성할 때 V2 기준으로 작성하되, 배포 서버에 V2가 설치되어 있는지 미리 확인해야 한다. 집에서 USB-C 충전기를 사놓고 서버실에는 Micro-USB만 있는 격이다.

## ARM vs AMD64 아키텍처

Mac(M1/M2)에서 Docker 이미지를 빌드하면 ARM 아키텍처(linux/arm64)로 빌드된다. 대부분의 서버는 AMD64(x86_64)다. ARM 이미지를 AMD64 서버에서 실행하면 `exec format error`가 나면서 컨테이너가 즉시 죽는다.

### 해결법 1: 빌드 시 플랫폼 지정

```yaml
# docker-compose.yml
services:
  app:
    build:
      context: .
      platform: linux/amd64
```

또는 CLI에서 `docker build --platform linux/amd64`.

### 해결법 2: CI/CD에서 빌드

GitHub Actions는 기본 `ubuntu-latest` 러너(AMD64)를 사용한다. CI에서 빌드하면 자연스럽게 AMD64 이미지가 된다. 로컬 빌드는 개발용, 실서버 이미지는 CI에서 빌드하는 게 맞다.

### 해결법 3: 멀티플랫폼 빌드

`docker buildx`로 한 번에 ARM + AMD64 이미지를 만들 수 있다. 1인 프로젝트에서는 과하다.

## 환경변수와 컨테이너 라이프사이클

`docker compose restart`와 `docker compose up --force-recreate`는 다르다.

| 명령 | 동작 | 환경변수 |
|------|------|---------|
| `restart` | 기존 컨테이너 재시작 | 반영 안 됨 |
| `up --force-recreate` | 컨테이너 삭제 후 재생성 | 반영됨 |
| `up -d` (변경 감지) | 변경된 것만 재생성 | 반영됨 |

`.env` 파일이나 `docker-compose.yml`을 수정한 후 `restart`를 하면 변경 사항이 적용되지 않는다. 컨테이너가 생성될 때 환경변수가 주입되고, restart는 생성 단계를 건너뛰기 때문이다. 메모지에 새로 적어놨는데 이미 출근한 직원한테는 안 전해지는 거다.

### Nginx envsubst

Nginx 공식 이미지는 `envsubst`를 지원한다. `nginx.conf.template`에 `${DOMAIN}` 같은 변수를 넣으면, 컨테이너 시작 시 환경변수로 치환한 `nginx.conf`를 생성한다.

```nginx
server_name tastenote-api.${DOMAIN};
```

이것도 `restart`로는 재치환이 안 된다. `--force-recreate`가 필요하다.

## server_name 네이밍 전략

처음에 `api.{DOMAIN}`, `admin.{DOMAIN}`으로 설정했는데, 나중에 다른 프로젝트를 같은 서버에 올리면 겹친다. 서비스명을 접두사로 넣는 게 맞다.

```
tastenote-api.code-forge.co.kr  → TasteNote API
ait-admin-api.code-forge.co.kr  → Admin API
```

## Cloudflare Tunnel

보통 서버 배포는 포트를 열어서 외부에서 접근하게 한다. Cloudflare Tunnel은 반대다. 서버가 Cloudflare한테 터널을 파서 내부 서비스를 노출한다.

```
외부 → Cloudflare Edge → Tunnel → cloudflared → Nginx → Spring Boot
```

장점:
- 서버 포트를 열 필요 없다 (UFW에서 80/443을 안 열어도 됨)
- 서버 IP가 외부에 노출되지 않는다
- SSL 인증서를 직접 관리하지 않아도 된다

택배 기사가 우리 집 주소를 모르게 하고, 중간 물류센터(Cloudflare)를 통해서만 배송하는 구조다. 집 주소를 바꿔도 물류센터 주소만 알려주면 된다.

## 정리

| 문제 | 원인 | 해결 |
|------|------|------|
| `ContainerConfig` 에러 | Docker Compose V1 | V2 플러그인 설치 |
| `exec format error` | ARM/AMD64 불일치 | `--platform linux/amd64` |
| 환경변수 미반영 | `restart` vs `recreate` | `--force-recreate` 사용 |

세 문제 모두 "로컬에서는 됐는데 서버에서 안 되는" 류다. 로컬 환경과 서버 환경의 차이를 미리 파악하는 게 핵심이다.
