[앱인토스-통합 Api] Pantry 독립화와 재료 이름 매칭 전환

By 코드대장간
Posted Mar 26 6 min read

오전에 만든 걸 오후에 다시 뜯었다

오전에 V2 스택을 구현했다. Pantry는 ingredient_id를 외래키로 가져서 레시피 재료와 연결하는 구조였다. 저녁에 FE와 소통하면서 문제를 발견했다. Flyway V9으로 수정했다. PR은 오전 V2와 별개.

기존 설계의 문제

1

pantry_item: { id, user_id, ingredient_id }

냉장고에 재료를 추가하려면 시스템에 존재하는 ingredient_id가 있어야 한다. “간장”을 냉장고에 넣으려면 ingredient 테이블에 “간장”이 먼저 있어야 한다는 뜻이다.

문제는 레시피 파싱이 자유롭다는 것. Gemini가 “진간장”, “국간장”, “양조간장”을 각각 다른 재료로 파싱할 수 있다. ingredient 테이블이 정규화되어 있지 않으면, 사용자가 “간장”을 냉장고에 넣어도 레시피의 “진간장”과 매칭이 안 된다.

냉장고 목록에 재료를 넣으려면 드롭다운에서 선택해야 하는 구조도 문제다. 사용자가 “고수”를 입력하고 싶은데 시스템에 없으면 추가가 안 된다.

Pantry 독립화

1
2
3
4
5
6
7

-- V9 마이그레이션
ALTER TABLE pantry_item
  ADD COLUMN name VARCHAR(100) NOT NULL DEFAULT '',
  ADD COLUMN category VARCHAR(50),
  MODIFY ingredient_id BIGINT NULL,
  DROP INDEX uq_pantry_user_ingredient,
  ADD UNIQUE KEY uq_pantry_user_name (user_id, name);

ingredient_id 필수 → nullable. name과 category를 직접 저장. Unique key를 (user_id, ingredient_id)에서 (user_id, name)으로 교체. 같은 이름의 재료를 두 번 못 넣도록.

PantryService에서 IngredientRepository 의존 완전 제거. “간장”을 냉장고에 넣으면 그냥 {name: "간장", category: "조미료"}로 저장된다. ingredient 테이블을 조회할 필요가 없다.

냉장고가 식자재 관리 앱의 냉장고처럼 작동한다. 마트에서 산 것들을 자유롭게 넣고 빼는 거다. 라벨이 없어도 넣을 수 있다.

이름 매칭으로 전환

레시피 재료와 냉장고 재료를 매칭하는 CookableRecipeService도 수정했다.

기존: ingredient.id 매칭. 레시피 재료의 ingredient_id와 냉장고의 ingredient_id를 비교. 변경: ingredient.name 정규화 후 매칭. 공백 제거, 소문자화.

1
2
3

String normalize(String name) {
    return name.replaceAll("\\s+", "").toLowerCase();
}

“진간장”과 “진 간장”은 같다. “Olive Oil”과 “olive oil”도 같다. 완벽한 매칭이 아니라 “합리적인 매칭”. 동의어 처리(간장 ↔ 진간장)는 아직 안 한다. 그건 나중에.

RecipeIngredient 링킹

레시피 재료(recipe_ingredient)에서 ingredient_id 자체가 null일 수 있다. Gemini가 파싱할 때 매칭되는 재료가 없으면 null로 남긴다.

RecipeIngredientLinkService와 RecipeIngredientLinkController를 추가했다.

1
2

PATCH /api/v1/recipe-ingredients/{id}/link
Body: { ingredientId: 123 }

어드민이 미연결 재료를 직접 연결할 수 있게. 나중에 어드민 UI에서 처리할 작업이다.

FE 핸드오프

냉장고 추가 API 요청 body가 바뀌었다.

이전이후
{ ingredientId: 42 }{ name: "간장", category: "조미료" }

FE가 바뀐 스펙에 맞춰 수정이 필요하다. TasteNote FE 쪽에 전달했다.

Claude Code 협업

오전에 만든 설계를 오후에 다시 뜯는 거라, “이렇게 바꿔야 할 것 같다”는 판단은 내가 했다. Claude한테는 “Pantry를 ingredient_id 의존에서 분리하고, name/category로 직접 저장하는 구조로 바꿔”로 지시.

Flyway V9, PantryService 수정, CookableRecipeService 이름 매칭 전환, RecipeIngredientLink 추가까지 한 세션에 처리했다.

“ingredient_id가 null이면 기존 unique 제약 위반이 안 나오는가?” 체크는 내가 직접 했다. MySQL에서 nullable unique 컬럼은 null이 여러 개 들어가도 제약 위반이 아니다.

오늘의 결과

항목상태
Flyway V8: rating nullable완료
Flyway V9: Pantry 독립화완료
PantryService: ingredient 의존 제거완료
CookableRecipeService: 이름 매칭 전환완료
RecipeIngredientLink API완료
FE 스펙 변경 전달완료

설계는 처음부터 완벽하지 않다. 실제로 FE와 이어보면 안 맞는 게 나온다. 뜯고 고치는 게 일이다.
