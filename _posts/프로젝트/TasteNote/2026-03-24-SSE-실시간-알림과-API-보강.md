## 파싱이 끝났는지 알 방법이 없다

레시피 등록(ParseJob)을 제출하면 BE에서 AI 파싱이 돌아간다. 비디오 분석이라 1~3분 걸린다. FE에서 완료 여부를 알 방법이 없었다. 폴링하면 되긴 하는데, SSE가 더 깔끔하다. 서버가 "끝났어"라고 한 번 밀어주면 된다.

## 도메인 이벤트 발행

[Discord Webhook](/posts/Discord-Webhook과-포트-패턴/)에서 `ServerErrorEvent`를 `ApplicationEvent`로 발행하는 패턴을 이미 썼다. 같은 구조를 그대로 적용.

`ParseJobService.complete()`/`fail()`에서 트랜잭션 커밋 후 `ParseJobCompletedEvent`를 발행. `publishEventAfterCommit()`은 기존 `enqueueAfterCommit` 패턴 재사용. 이벤트 발행 실패 시 로그만 남기고 job 처리에 영향 없음.

도메인 모델도 보강했다. `ParseJobStatus`에 `isTerminal()` 메서드 추가(DONE/FAILED 판별). `ParseJobResponse`에 `userId` 필드 추가(SSE 엔드포인트에서 소유자 검증용).

## SSE 엔드포인트

`GET /api/v1/parse-jobs/{id}/events`. JWT 인증 + job 소유자 검증. `SseEmitterManager`에서 `ConcurrentHashMap` 기반으로 emitter 생명주기 관리. 타임아웃 120초, job당 emitter 1개. 완료 후 즉시 종료.

### Race Condition 대응

이미 완료된 job에 SSE를 등록하면 이벤트를 놓친다. 3단계로 처리.

1. 상태 확인 — 이미 완료됨? → 바로 결과 반환
2. emitter 등록
3. 재확인 — 등록하는 사이에 완료됐을 수 있으니 한 번 더 체크

식당에서 번호표를 받으려는데, 이미 주문이 나왔을 수 있으니까 일단 카운터에 물어보고(1), 번호표 받고(2), 혹시 받는 사이에 나왔나 다시 확인(3)하는 거다.

## enqueueAfterCommit

SSE와는 별개로, 파싱 자체가 안 되는 버그가 있었다. `@Transactional` 메서드 안에서 DB save 직후 Redis enqueue를 호출하면, 트랜잭션 커밋 전에 워커가 dequeue해서 DB에서 PENDING 상태를 못 찾는다. "Job already claimed, skipping".

택배를 부치기도 전에 배달원이 와서 "택배 없는데요?" 하는 거다.

`TransactionSynchronization.afterCommit()`으로 커밋 후 enqueue하도록 변경. `createParseJob`, `recoverStuckJobs`, `retryJob` 전부 적용. 트랜잭션이 비활성 상태면 즉시 enqueue하는 fallback도 추가. 실패 시 watchdog(10분 주기)가 orphaned job을 복구.

## RecipeResponse 보강

FE를 실제로 연결하니까 응답 필드가 안 맞는 게 보였다.

| 문제 | 수정 |
|------|------|
| FE가 `versions` 배열을 기대하는데 BE가 안 내려줌 | `RecipeResponse`에 `versions`, `thumbnail` 필드 추가 |
| `stepOrder` 필드명 불일치 | `StepResponse`에서 `stepOrder` → `order`로 변경 |
| soft delete 표시 필요 | `RecipeVersionResponse`에 `deletedAt` 필드 추가 |
| steps/ingredients 정렬 안 됨 | `RecipeVersionResponse.from()`에서 정렬 추가 |

`getRecipe`, `getRecipesByUser`, `createRecipe` 등 모든 반환에 versions 포함. 기존 캐시 키(`recipe:v2:`)는 동일하므로 캐시 자동 갱신.

## Claude Code 협업

SSE 구현은 "ParseJob 완료 시 FE에 실시간 알림 보내고 싶다, ApplicationEvent + SseEmitter로"라고 지시. Race condition 3단계 대응은 Claude가 제안. 잘 잡아줬다.

`enqueueAfterCommit` 버그는 로그를 보다가 발견. "Job already claimed" 로그가 계속 찍히길래 타이밍 문제인 걸 파악하고 Claude에 "트랜잭션 커밋 후로 enqueue를 지연해"라고 지시.

RecipeResponse 보강은 FE 연결하면서 에러가 나올 때마다 하나씩 수정. 이건 FE/BE를 번갈아가며 작업한 결과다.
