## 3월 17일

[광고 넣고](/posts/광고-시스템과-리팩토링/) 정리한 다음날. 세 프로젝트 동시. 여기는 TasteNote FE 얘기만.

**어드민에서 앱을 제어할 수 있게** 만드는 게 목표. 어드민 API가 추가되면서 TasteNote도 그 데이터를 받을 준비가 필요했다.

Claude Code한테 BE 기획서(1,200줄)를 던지고 "이 스펙대로 FE 인프라 깔아"라고 했다. 여러 프로젝트를 왔다갔다 하니까 컨텍스트 설명을 반복해야 하는 게 좀 귀찮았다.

## adminMapper — 8개 Raw 타입 → 도메인 변환

어드민에서 내려오는 데이터 타입이 8개. 공지사항, 배너, 앱 설정, 피처 플래그... 전부 snake_case. **adminMapper**로 일괄 변환.

```typescript
// Raw 타입들 (서버 응답)
RawNotice, RawBanner, RawAppConfig, RawFeatureFlag,
RawPopup, RawTerms, RawReportReason, RawAppVersion

// → 도메인 모델로 변환
adminMapper.toNotice(raw)
adminMapper.toBanner(raw)
adminMapper.toAppConfig(raw)
// ...
```

[처음에 설계한](/posts/TasteNote-프로젝트-시작/) Mapper 패턴 그대로. 구조를 잡아둔 보람이 있다.

## appConfigStore — 피처 플래그 관리

어드민에서 앱 기능을 켜고 끄는 **피처 플래그** 시스템이 필요했다. Zustand로 `appConfigStore`를 만들었다.

```typescript
// 피처 플래그 예시
ENABLE_REPORT: false   // 신고 기능 — 아직 비활성
ENABLE_ADS: true       // 광고 — 활성
```

신고 기능은 만들어두긴 했는데 `ENABLE_REPORT=false`로 숨겨뒀다. 현재 버전에서는 혼자 쓰는 거라 신고 기능이 불필요하다. 근데 나중에 사용자가 늘면 필요하니까 코드는 만들어두고 플래그로 숨김. 어드민에서 플래그 하나만 바꾸면 바로 켜지게.

## MaintenanceGuard — 점검 모드 처리

점검 모드가 가장 고민이었다. 어드민에서 "점검중" 설정하면 점검 화면을 띄워야 하는데, **어드민 API가 죽으면?**

3초 기다려보고 응답 없으면 그냥 앱을 연다. 어드민이 죽었다고 사용자한테 점검 화면 보여주는 건 말이 안 된다.

```typescript
// MaintenanceGuard 핵심 로직
// 1. 앱 시작 시 어드민 API에 점검 상태 확인
// 2. 3초 내 응답 없으면 → 그냥 통과 (안전하게 열어줌)
// 3. 점검중이면 → 점검 화면 표시
// 4. 로딩 중에는 스플래시 화면 (앱 노출 방지)
```

로딩 중에 앱이 잠깐 보이는 버그가 있었다. 스플래시 화면으로 가렸다. Claude가 놓친 부분. "로딩 중에 앱 보이는데?" 하니까 바로 고쳤는데, 이런 UX 디테일은 결국 사람이 눈으로 확인해야 한다.

## AppVersionCheck 훅

강제 업데이트 대비. 어드민에서 최소 버전을 설정하면 그보다 낮은 버전은 업데이트 유도 화면.

## 그 외 작업들

오늘 TasteNote에서 한 작업이 진짜 많았다:

- **홈 화면 강화**: 추천 레시피 섹션, 배너 캐러셀, 팝업
- **콘텐츠 신고 기능**: 만들어두고 피처 플래그로 숨김
- **공지사항, 약관, 설정 페이지**: 라우트 추가
- **공지사항 HTML 보안 처리**: 공지사항에 HTML을 그대로 렌더링하고 있었는데, 악성 스크립트가 끼어들 수 있다. DOMPurify로 위험한 태그를 걸러내게 수정
- **배너 링크 보안**: tabnabbing이라고, 새 창으로 연 링크에서 원래 페이지를 조작하는 공격 방식이 있다고 해서 막았다. `noopener/noreferrer` 추가
- **Firebase Analytics 연동**: 로그인, 홈, 상세, 등록, 배너, 신고 전부 계측
- **Analytics 버퍼링**: Firebase 초기화 전 이벤트 누락 방지

Analytics는 사업자등록 마무리 전에 미리 깔아둬야 한다고 판단해서 먼저 작업했다. 출시 후 사용자 행동 데이터가 없으면 개선 방향을 못 잡으니까. Claude한테 "전체 계측 넣어"라고 했더니 10개 페이지를 한번에 넣어줬다. 반복 작업은 확실히 잘한다. 근데 이벤트가 중복 발화되는 문제가 있었고, `.env.production`을 git에 올리는 실수도 했다. 내가 리뷰에서 잡았다. 또 Analytics 이벤트에 사용자가 입력한 URL이 그대로 들어가는 것도 문제. 토큰이나 식별자가 URL에 포함될 수 있으니까 도메인만 전송하도록 수정했다.
