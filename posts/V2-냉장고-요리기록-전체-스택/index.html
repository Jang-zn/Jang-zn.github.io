[앱인토스-통합 Api] V2 냉장고·요리기록 전체 스택 하루에

By 코드대장간
Posted Mar 26 7 min read

09:09부터 09:11까지 5개 커밋

어제 인앱결제와 홈서버 배포를 끝내고, 오늘은 V2 기능 전체를 구현했다. 냉장고(Pantry)와 요리기록(CookingLog) — DB 스키마부터 도메인, 서비스, API, 테스트까지. 커밋 5개가 09:09~09:11 사이에 쏟아졌다. Claude Code 에이전트 4명을 병렬로 돌린 결과다.

V2가 뭔가

TasteNote V1은 레시피 저장과 버전 관리만 했다. V2는 두 가지를 추가한다.

  1. 냉장고 관리 (Pantry): 가지고 있는 재료를 등록. 레시피에 필요한 재료와 매칭해서 “지금 만들 수 있는 레시피”를 제안
  2. 요리기록 (CookingLog): 레시피를 실제로 만들었을 때 기록. 별점, 소요시간, 메모. Streak(연속 요리 일수) 계산

CREATE TABLE cooking_log (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  recipe_version_id BIGINT NOT NULL,
  rating INT,           -- nullable (별점 없이 기록 가능)
  cooking_time_ms BIGINT,
  memo TEXT,
  cooked_at TIMESTAMP NOT NULL
);

CREATE TABLE cooking_log_ingredient (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  cooking_log_id BIGINT NOT NULL,
  ingredient_id BIGINT NOT NULL,
  action ENUM('USED','ADJUSTED','SKIPPED') NOT NULL,
  adjusted_amount VARCHAR(100)
);

CREATE TABLE pantry_item (
  id BIGINT AUTO_INCREMENT PRIMARY KEY,
  user_id BIGINT NOT NULL,
  ingredient_id BIGINT NOT NULL
);

recipe_version에 source 컬럼 추가. 수동 입력인지(MANUAL), URL 파싱인지(PARSED), 요리기록에서 파생됐는지(FROM_COOKING).

CookingLog에 BaseEntity를 상속하지 않은 이유는 학습 포스트에 정리했다.

도메인 레이어

서비스 4개가 핵심이다.

PantryService

사용자의 냉장고 관리. 재료 추가/삭제, 냉장고 목록 조회. 단순하다.

PantryDeductionService

요리 완료 후 냉장고에서 사용한 재료를 차감. CookingLogIngredient의 action이 USED나 ADJUSTED면 해당 재료를 냉장고에서 차감한다. SKIPPED는 안 씀.

N+1이 여기서 걸렸다. 재료 목록 순회 중 각각 pantryRepo.findByIngredientId()를 호출했다. 학습 포스트에 정리한 배치 조회 패턴으로 해결.

CookingLogService

요리기록 저장 + 냉장고 차감을 단일 트랜잭션으로 묶었다. 기록은 됐는데 재료 차감이 안 된 상태가 나오면 안 된다.

CREATE cooking_log
→ CREATE cooking_log_ingredients (bulk insert)
→ DEDUCT pantry_items (배치 차감)

세 작업이 전부 성공하거나 전부 실패해야 한다.

Streak 계산: DATE(cooked_at)로 날짜 목록을 가져오고 Java에서 연속성 판단. MySQL 함수 인덱스를 적용했다.

CookableRecipeService

냉장고에 있는 재료와 레시피의 필수 재료를 비교해서 만들 수 있는 레시피를 찾는다. 매칭률이 50% 이상이면 “만들 수 있음”으로 분류.

API 레이어 — 4개 컨트롤러

컨트롤러엔드포인트 수주요 기능
PantryController4개냉장고 CRUD
CookingLogController3개요리기록 생성/조회/삭제
CookableRecipeController1개만들 수 있는 레시피 목록
CookingStatsController2개통계(streak, 월별 요리 수)

Admin에도 2개 컨트롤러 추가. 어드민 대시보드에 V2 지표 4개(총 요리기록 수, 오늘 요리 수, 활성 냉장고 사용자, 평균 별점) 추가.

테스트

서비스 4개 단위 테스트 + 컨트롤러 4개 @WebMvcTest. 총 파일 9개, 852줄.

엣지케이스 포함:

  • null amount 재료 차감 (수량 없이 USED 처리)
  • 매칭률 정확히 0.5 (경계값)
  • 7일 연속 → 하루 빠짐 → Streak 리셋

@ConditionalOnProperty 적용으로 admin-api 테스트에서 JWT 빈 충돌 제거. 학습 포스트에서 다뤘던 문제가 여기서도 나왔다.

Fake Repository 9개를 AbstractFakeRepository로 리팩터링. 학습 포스트 내용 그대로 적용.

Claude Code 협업

에이전트 4명 병렬 구성:

  • Domain: 엔티티, Enum, 포트 인터페이스, DTO
  • Persistence: JPA 구현체, @Query
  • App: 서비스, 컨트롤러
  • Test: 단위 테스트, @WebMvcTest

포트 인터페이스를 먼저 정의하면 Persistence와 App이 동시에 개발할 수 있다. 설계도(포트)가 나오는 순간 3팀이 동시 시공.

09:11에 5개 커밋이 몰려있는 건 에이전트들이 동시에 PR 올린 것들을 한꺼번에 머지한 것. 실제 개발 시간은 더 길었다.

Reviewer 에이전트가 잡은 것:

  • PantryDeductionService의 N+1 쿼리
  • CookingLogService에서 CookingLogIngredient bulk insert 누락 (각각 save하고 있었음)
  • 매칭률 임계값 0.5가 “이상”인지 “초과”인지 모호 → >= 0.5로 명시
