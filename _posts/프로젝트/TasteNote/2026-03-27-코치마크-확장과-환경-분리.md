## 코치마크 확장

[이전에 코치마크 시스템을 만들었다](/posts/코치마크-튜토리얼-구현/). 홈/상세/등록/요리 4페이지 튜토리얼. 오늘은 여기에 재료 검색과 조리 단계 탭 튜토리얼을 추가했다.

기존 구조가 잘 잡혀 있어서 확장 자체는 어렵지 않았다. 문제는 확장하면서 드러나는 엣지 케이스들이었다.

## 재료 검색 튜토리얼

[어제 만든 냉장고 페이지](/posts/냉장고와-히스토리-페이지/)의 `AddIngredientSheet`에 튜토리얼을 붙였다. 재료를 추가하는 과정이 3갈래로 나뉜다.

| 단계 | 시나리오 |
|------|----------|
| search | 자주쓰는 재료 → 검색 → 직접 추가 안내 |
| common-detail | 공통 재료 선택 시: 수량/단위 → 추가하기 |
| custom-detail | 직접 추가 선택 시: 카테고리 → 수량/단위 → 추가하기 |

하나의 진입점에서 유저 행동에 따라 다른 튜토리얼이 실행된다. `useAddIngredientCoachMark` 훅 하나로 3개 시나리오를 처리한다.

```tsx
function useAddIngredientCoachMark(
  stage: 'search' | 'details',
  isCustom: boolean,
  open: boolean
) {
  const config = stage === 'search'
    ? ADD_SEARCH_TUTORIAL
    : isCustom ? ADD_CUSTOM_DETAIL : ADD_COMMON_DETAIL;
  useCoachMark(config, open && (stage === 'search' || searchDone));
}
```

`stage`와 `isCustom` 조합으로 config를 결정하고, `open`으로 시트가 열려있을 때만 활성화한다. 처음에는 hook을 3번 호출하는 구조였는데, Zustand 구독이 불필요하게 늘어나서 1번 호출로 줄였다.

## 조리 단계 탭 튜토리얼

[요리모드](/posts/요리모드-v2-구현과-UX-개선/)에는 "재료 확인"과 "조리 단계" 두 탭이 있다. 기존 튜토리얼은 재료 확인 탭에서만 돌아갔다. 조리 단계 탭 안내가 빠져 있었다.

문제는 튜토리얼이 탭을 자동으로 전환해야 한다는 것이었다. 박물관 오디오 가이드처럼 "이쪽으로 오세요" 하면서 유저를 끌고 다녀야 한다.

처음에 Claude Code가 드래그 관련 스텝을 재료 확인 탭에 넣어둔 걸 발견했다. 재료 탭에서 "드래그로 순서를 바꿀 수 있어요"라고 설명하면 유저는 혼란스럽다. 조리 순서 탭으로 넘어가서 설명해야 한다.

"스텝을 제거하지 말고 탭을 전환해서 설명해라"고 요청했다. 탭 자동 전환 로직이 추가됐다.

```tsx
// step 변화를 구독해서 탭을 자동 전환
useEffect(() => {
  if (step >= STEPS_TAB_FIRST && step <= STEPS_TAB_LAST)
    setActiveTab('steps');
  else if (step > STEPS_TAB_LAST)
    setActiveTab('ingredients');
}, [step, setActiveTab]);
```

step이 조리 순서 구간에 들어가면 탭을 전환하고, 끝나면 다시 재료 확인으로 복귀한다.

## 탭 전환 로직 캡슐화

근데 이게 좀 걸렸다. `useCookingCoachMark`가 탭 전환까지 직접 하면 [추상화 누수](/posts/Hooks-규칙과-추상화-캡슐화/)다. 코치마크 훅이 페이지의 탭 상태를 알고 있으면 안 된다. 냉장고가 식당 주문 시스템을 제어하는 꼴이다.

`setActiveTab` 콜백을 외부에서 주입하는 방식으로 해결했다.

```tsx
useCookingCoachMark(enabled, setActiveTab);
```

훅 내부에서는 "몇 번 스텝이면 어떤 탭"이라는 매핑만 알고 있고, 실제 탭 전환은 주입받은 콜백으로 위임한다. 코치마크 훅은 탭의 존재를 모른다. 그냥 "이 스텝에서 이 함수를 호출해라" 수준.

## scrollIntoView 자동 이동

화면 밖에 있는 요소에 스포트라이트를 비추려면 먼저 스크롤해야 한다. [DOM Observer 학습 포스트](/posts/DOM-Observer와-뷰포트-스크롤-패턴/)에서 다룬 패턴이다.

`getBoundingClientRect`로 타겟이 뷰포트 안에 있는지 확인하고, 밖이면 `scrollIntoView({ behavior: 'smooth', block: 'center' })`로 부드럽게 이동시킨다. 스크롤 완료 후 위치를 다시 측정해서 스포트라이트를 배치한다.

조리 단계가 길어지면 아래쪽 스텝은 화면 밖으로 밀려난다. 이때 스포트라이트만 덩그러니 화면 밖을 가리키고 있으면 유저는 아무것도 못 본다. 스크롤이 먼저 움직여야 한다.

## 시트 닫힘 시 자동완료 버그

이게 오늘 가장 짜증나는 버그였다.

BottomSheet가 닫히면 내부 DOM이 통째로 사라진다. `data-coach` 속성이 붙은 요소도 같이 사라진다. MutationObserver가 "타겟이 없어졌다"고 판단하고, 3초 타임아웃 후 `skipStep()`을 호출한다. 모든 스텝이 건너뛰어지면서 튜토리얼이 "완료" 처리된다.

시트를 닫았을 뿐인데 튜토리얼을 다 본 것으로 기록되는 거다. 택배 상자를 뜯기도 전에 "수령 완료" 처리되는 느낌.

해결은 간단했다. `useCoachMark`의 두 번째 effect에 `enabled` 가드를 추가했다.

```tsx
if (!enabled) return; // 시트 닫혔으면 Observer 자체를 안 돌린다
```

`open` 상태가 `false`면 `enabled`가 `false`가 되고, Observer가 아예 시작하지 않는다. DOM에서 요소가 사라져도 관심 없다.

비슷한 문제로 진행 중인 요리가 있을 때 재개 시트와 튜토리얼이 겹치는 것도 있었다. `useCookingCoachMark(!resumeSheetOpen)`으로 시트가 열려있으면 튜토리얼을 차단했다.

## 환경 설정 분리

여기서 코치마크 작업에서 잠깐 벗어났다. .env 파일이 하나뿐이라 local/dev/prod를 구분할 수 없었다.

3개로 분리했다.

| 파일 | 모드 | 용도 |
|------|------|------|
| `.env.localhost` | localhost | 내 노트북 |
| `.env.dev` | dev | 개발 서버 |
| `.env.prod` | prod | 프로덕션 |

npm 스크립트도 3개 추가. `npm run local`, `npm run dev`, `npm run prod`.

한 가지 삽질이 있었다. Vite 7에서 `--mode local`을 쓸 수 없다. "local"이 Vite 내부 예약어라서 충돌한다. `.env.local`도 Vite가 기본으로 읽는 파일이라 겹친다.

그래서 모드명을 "localhost"로 바꿨다. `.env.localhost`, `--mode localhost`. 이름이 좀 길어졌지만 충돌은 없다.

## Debug FAB

코치마크를 개발하면서 매번 localStorage를 수동으로 지우는 게 귀찮았다. 튜토리얼 완료 기록을 리셋하려면 DevTools를 열어야 한다.

Debug FAB를 만들었다. 주황색 벌레 아이콘 플로팅 버튼. local/dev에서만 보인다.

```tsx
// IS_DEBUG_ENV = import.meta.env.MODE !== 'prod'
if (!IS_DEBUG_ENV) return null;
```

프로덕션에서는 컴포넌트 자체가 렌더링되지 않는다. 번들에는 포함되지만 실행되지 않는다.

기능은 일단 하나. "코치마크 초기화" 버튼. 누르면 localStorage에서 튜토리얼 완료 기록을 전부 지우고, 토스트로 "코치마크 초기화 완료"를 띄운다. 새로고침하면 홈 튜토리얼부터 다시 시작된다.

나중에 디버그 기능이 더 필요해지면 여기에 추가하면 된다. 메뉴가 확장 가능한 구조.

처음에 Claude Code가 만든 코드에서 Rules of Hooks 위반이 있었다. 조건부 렌더링 안에서 hook을 호출하는 구조. 내부 컴포넌트로 분리해서 해결했다.

## 플랫폼 체크 중앙화

`navigator.userAgent`로 토스 앱 내부인지 확인하는 코드가 여기저기 흩어져 있었다. `IS_TOSS_PLATFORM` 상수로 한곳에 모았다.

```tsx
export const IS_TOSS_PLATFORM = /tossapp/i.test(navigator.userAgent);
```

같은 파일(`platform.ts`)에 `IS_DEBUG_ENV`도 넣었다. 환경/플랫폼 관련 상수가 한곳에서 관리된다.

## Claude Code 협업

오늘은 크게 두 세션으로 나뉘었다.

첫 번째 세션: 코치마크 확장. "재료 추가 과정을 보여주는 튜토리얼 추가해"로 시작. 재료 검색 튜토리얼은 한 번에 잘 나왔다. 조리 탭 튜토리얼은 처음에 스텝 배치가 엉뚱했다. 재료 탭에서 조리 순서 설명을 하고 있어서 "탭을 전환해서 설명해라"고 수정 요청. 탭 자동 전환 로직이 추가됐다.

`recipe.name`이 `undefined`로 나오는 버그도 Claude Code가 만든 것이다. `Recipe` 타입에는 `title` 필드만 있는데 `name`으로 참조했다. TypeScript strict 모드에서도 `undefined`를 반환할 뿐 컴파일 에러가 안 나는 케이스. 스크린샷을 보여주고 원인을 찾게 했다.

두 번째 세션: 환경 분리와 Debug FAB. "local/dev/prod 환경 분리하고 디버그 FAB 추가해"로 지시. Vite 7의 "local" 예약어 문제는 Claude Code가 먼저 발견하고 "localhost"로 우회했다. Debug FAB에서 Rules of Hooks 위반이 나왔는데 이건 내가 발견하고 수정 지시.

전반적으로 코치마크 확장은 기존 구조 덕분에 수월했다. `useCoachMark` 훅이 `enabled`와 `config`만 받는 단순한 인터페이스라서, 새 튜토리얼을 추가하는 건 config 정의 + data-coach 속성 추가 + 전용 훅 작성 세 단계면 끝난다.
