## 이미 있는 테이블의 제약을 바꾸는 건 다르다

[Flyway 기본 전략](/posts/Flyway-멀티모듈-마이그레이션-전략/)은 새 테이블 생성에 관한 거였다. 이번엔 운영 중인 테이블의 제약을 바꾸는 Flyway V9 마이그레이션. [Pantry 독립화](/posts/Pantry-독립화와-재료-연결/)에서 실제로 적용한 내용이다.

V9에서 한 변경 3가지:
1. `ingredient_id NOT NULL` → `NULL` 허용
2. `name`, `category` 컬럼 추가
3. Unique Key `(user_id, ingredient_id)` → `(user_id, name)` 교체

## NOT NULL → NULL 전환

```sql
ALTER TABLE pantry_item
  MODIFY COLUMN ingredient_id BIGINT NULL;
```

### 주의: 컬럼 정의 전체 재선언

`MODIFY COLUMN`은 컬럼 정의를 통째로 교체한다. 기존에 `DEFAULT 0`이나 `COMMENT '...'`가 있었으면 누락 시 사라진다. 헤어스타일을 바꿀 때 옷도 다 벗어야 하는 구조. 귀찮지만 기존 정의를 복사해서 바꾸는 것만 수정해야 한다.

기존 데이터 값은 유지된다. `NOT NULL`이었던 컬럼이 `NULL`로 바뀐다고 기존 데이터가 null이 되지 않는다.

### 운영 환경 주의

신규 개발 중이라 데이터가 없으니 간단하다. 운영 중이라면:
- 기존 데이터의 `ingredient_id`가 null이 될 수 없으므로 backfill 불필요
- `name` 컬럼 추가 후 기존 데이터의 `name` 값을 채워야 함 (backfill 스크립트)
- backfill 전에 배포하면 `name NOT NULL` 제약이 있으면 INSERT 실패

## Unique Key 교체

```sql
ALTER TABLE pantry_item
  DROP INDEX uq_pantry_user_ingredient,
  ADD UNIQUE KEY uq_pantry_user_name (user_id, name);
```

`DROP INDEX` + `ADD UNIQUE KEY`를 **단일 `ALTER TABLE`에 묶으면** 테이블 락이 1번만 발생한다. 분리하면 2번. 대용량 테이블에서 락 시간이 두 배로 늘어난다. MySQL 5.6 이상에서 적용. 가독성을 위해 분리하는 경우도 있는데, 운영 테이블은 락 최소화가 우선이다.

### nullable unique 컬럼과 중복

MySQL에서 nullable unique 컬럼에 null이 여러 개 들어가도 제약 위반이 아니다. null은 "알 수 없음"이라 서로 같은지 비교할 수 없기 때문이다. `ingredient_id NULL UNIQUE`이면 null 값이 여러 행에 있어도 된다.

## 컬럼 추가와 AFTER 절

```sql
ALTER TABLE pantry_item
  ADD COLUMN name VARCHAR(100) NOT NULL DEFAULT '',
  ADD COLUMN category VARCHAR(50) NULL
    AFTER ingredient_id;
```

`AFTER ingredient_id`는 물리적 컬럼 순서를 지정한다. 없으면 테이블 끝에 추가된다. 개발 중에는 크게 중요하지 않지만, ERD나 `DESCRIBE` 출력에서 가독성이 달라진다.

대용량 테이블에서 컬럼 추가는 InnoDB 내부적으로 테이블을 재빌드한다. `AFTER` 절이 있든 없든. 수백만 건이 넘으면 `pt-online-schema-change`나 `gh-ost` 같은 무중단 마이그레이션 도구를 쓰는 게 맞다.

## 안전성 체크리스트

운영 환경 스키마 변경 전 확인 사항:

| 항목 | 확인 내용 |
|------|----------|
| 기존 데이터 호환 | 변경 후 기존 행이 제약 위반하지 않는가 |
| DEFAULT 존재 여부 | NOT NULL 추가 시 기존 행의 기본값 |
| 인덱스 참조 쿼리 | 드롭할 인덱스를 쿼리가 힌트로 쓰고 있지 않은가 |
| backfill 필요 여부 | 새 컬럼에 기존 데이터를 채워야 하는가 |
| 롤백 가능 여부 | 마이그레이션을 되돌릴 수 있는가 |

backfill 전에는 롤백이 쉽다. backfill 후에는 되돌리면 `name`이 비어있어야 하는데 이미 값이 채워진 상태라 복잡해진다. 롤백 계획을 배포 전에 세워두는 게 맞다.
