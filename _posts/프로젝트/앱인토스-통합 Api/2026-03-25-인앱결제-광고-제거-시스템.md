## 5일 만에 돌아온 BE

[3/20에 CI/CD까지 끝내고](/posts/홈서버-배포와-CI-CD-구축/) 며칠간 FE 쪽 작업을 했다. 오늘은 인앱결제로 광고를 제거하는 기능을 BE에 통째로 구현했다. PR #21 머지.

## 왜 인앱결제인가

TasteNote에 인터스티셜 광고를 넣었다. 레시피를 저장할 때마다 전면 광고가 뜬다. 광고가 불편한 사용자한테 "돈 내면 꺼줄게" 옵션을 주는 거다. 자판기에 동전 넣으면 광고 없이 음료수만 나오는 프리미엄 모드.

## 상품 구조

| 상품 | 가격 | 유효기간 |
|------|------|----------|
| 영구 제거 | 9,900원 | 무제한 |
| 월간 제거 | 2,000원 | 30일 |

`PurchaseStatus`는 4개: ACTIVE, EXPIRED, REFUND_REQUESTED, REFUNDED. 영구 상품은 ACTIVE에서 REFUNDED로만 가고, 월간 상품은 EXPIRED까지 간다.

## 도메인 모델

`Purchase` 엔티티에 도메인 로직을 몰아넣었다.

- `isAdFree()`: ACTIVE 상태인 구매가 있는지. 영구면 무조건 true, 월간이면 만료일 체크
- `isRefundable()`: 영구 상품만 환불 가능. 월간은 기간제라 환불 없음
- `calculateRefundAmount()`: 전액 환불 (부분 환불은 과하다)
- `requestRefund()` / `approveRefund()` / `expire()`: 상태 전이 메서드

Flyway V6 마이그레이션으로 `purchase` 테이블 생성.

## 인프라 어댑터

Port & Adapter 패턴. 도메인이 인프라를 모른다.

- **TossOrderVerificationAdapter**: Toss API로 주문 상태 검증. FE가 "결제했다"고 말해도 서버가 직접 확인한다. 손님이 "돈 냈어요"라고 해도 금고를 열어보는 거다
- **JpaPurchaseRepository**: JPQL로 사용자별 활성 구매 조회, 월간 만료 대상 페이징 조회
- **Discord 알림**: 환불 요청이 들어오면 Discord로 알림. [기존 Discord 모듈](/posts/홈서버-배포와-CI-CD-구축/)에 `sendRefundRequest` 추가

## 서비스 레이어

`PurchaseService`가 핵심이다.

### 구매 부여 — 멱등성 + 검증 + 자동환불

`grantProduct()`가 가장 복잡하다. 결제 완료 후 FE가 서버에 "이 주문 처리해줘"를 보낸다.

1. 같은 `orderId`로 이미 처리된 구매가 있으면 그냥 반환 (멱등성)
2. Toss API로 주문 상태 검증 — 실패하면 자동 환불 처리
3. 동일 상품 타입의 기존 구매가 있으면 연장 처리
4. 없으면 신규 생성

멱등성이 왜 필요한가. FE에서 네트워크 에러로 응답을 못 받으면 같은 요청을 다시 보낸다. 그때 중복 처리가 되면 안 된다. 택배를 두 번 보내면 안 되는 거다. 자세한 건 [학습 포스트](/posts/인앱결제-서버-설계-패턴/)에 정리.

### 광고 제거 상태 조회 — Redis 캐시

`getAdFreeStatus()`는 Redis에 5분 캐시. 광고를 표시할지 말지는 매 화면마다 확인하는데, 매번 DB를 조회하면 낭비다. 냉장고에 우유가 있는지 확인하려고 매번 마트에 갈 필요 없다.

### 월간 구매 만료 스케줄러

`PurchaseExpiryScheduler`가 매시간 돌면서 만료된 월간 구매를 EXPIRED로 변경. 500건씩 페이징 처리. 만료일이 지난 구독을 한꺼번에 갱신하면 DB 부하가 걸리니까 끊어서 처리한다.

### 환불

사용자가 환불 요청 → Discord 알림 → 어드민이 확인 후 승인. 자동 환불이 아니라 수동 승인인 이유는, 1인 개발이라 악용 케이스를 자동으로 판별할 로직을 만들기 어렵기 때문이다.

## API 엔드포인트

| 메서드 | 경로 | 용도 |
|--------|------|------|
| POST | `/api/v1/purchases/grant` | 구매 부여 |
| GET | `/api/v1/purchases/ad-free` | 광고 제거 상태 |
| GET | `/api/v1/purchases/history` | 구매 내역 |
| POST | `/api/v1/purchases/refund-request` | 환불 요청 |

사용자 4개, 어드민 3개. 총 7개.

## Claude Code 협업

이번에는 에이전트 4명을 병렬로 돌렸다. Foundation(도메인 모델), Infra(어댑터), App(서비스/컨트롤러), Reviewer(코드리뷰).

Foundation → Infra → App 순서로 의존성이 있지만, 포트 인터페이스를 먼저 정의하면 세 팀이 동시에 작업할 수 있다. 건물 설계도(포트)를 먼저 확정하고 각 층(모듈)을 동시에 시공하는 방식.

Reviewer가 잡은 문제 4개:

1. **AdminPurchaseService**: `PurchaseStatus.valueOf()`에 잘못된 문자열이 들어오면 터짐 → try-catch 추가
2. **getAdFreeStatus()**: `@Transactional(readOnly=true)`인데 Redis 캐시 쓰기를 함 → readOnly 제거
3. **grantProduct()**: 기존 구매 목록 조회 후 개별 save → `saveAll()`로 배치 처리
4. **expireMonthlyPurchases()**: 만료 대상 전체 조회 → 페이징 조회로 변경

2번이 위험했다. readOnly 트랜잭션에서 캐시 쓰기를 하면 조용히 실패하거나 예외가 날 수 있다. 테스트에서 안 잡히는 류의 버그다.
