## BE도 시작

[TasteNote FE를 끝낸](/posts/TasteNote-프로젝트-시작/) 같은 날, 백엔드 초기 세팅. 이 서버는 TasteNote만을 위한 게 아니다. **앱인토스에 출시할 여러 미니앱이 공유하는 공용 서버**라는 전제로 설계했다. 공용 API 하나, 앱별 API 하나씩.

## 멀티모듈 Gradle — 처음 해봄

솔직히 멀티모듈 Gradle은 처음이다. 지금까지는 MVC 단일 모듈로만 해왔는데, 이번에는 처음부터 멀티모듈로 가기로 했다. 어드민, 워커, 다른 앱이 추가될 거니까. **131개 파일, 11개 모듈**, 초기 커밋 하나.

처음에 많이 헷갈렸다. "이거 모듈들 다 따로 띄워야 돼?" — 단일 서버 API인 줄 알았는데 모듈별로 독립 실행이 가능한 구조였다. MVC에서는 Controller → Service → Repository 흐름이 직관적인데, 클린 아키텍처는 인터페이스를 중간에 두고 구현체를 갈아끼우는 방식이라 흐름이 다르다. Claude한테 "동작 흐름 좀 정리해줘, MVC가 아니라 헷갈려"라고 해서 문서를 만들게 했다.

```
app-in-toss-be/
├── core/
│   ├── core-domain/       ← 공용 도메인 (User, Port 인터페이스)
│   ├── core-exception/    ← 공용 예외 처리
│   └── core-security/     ← 보안 공통
├── domain/
│   └── tastenote-domain/  ← 레시피 도메인 로직
├── infra/
│   ├── infra-mysql/       ← JPA 엔티티, 리포지토리
│   ├── infra-redis/       ← 캐시, 동시 접근 제어
│   ├── infra-s3/          ← 이미지 업로드
│   └── infra-gemini/      ← AI 레시피 파싱
├── app/
│   ├── tastenote-api/     ← 메인 API 서버
│   ├── tastenote-worker/  ← 비동기 작업 처리
│   └── admin-api/         ← 어드민 API (나중에 추가)
```

## 기술 스택 선택

| 기술 | 선택 이유 |
|------|----------|
| **Spring Boot 3** | 익숙함 + 가상 스레드 지원 |
| **MySQL** | 레시피 데이터는 관계형이 맞음 |
| **Redis** | 캐시 + 동시 접근 제어 + 세션 |
| **S3** | 이미지 저장소 (버킷: taste-note-rsc) |
| **Gemini AI** | 레시피 URL/유튜브 쇼츠 파싱용 (무료 계정으로 되는지 확인 필요) |
| **Flyway** | DB 테이블 변경 이력 관리 |
| **UUID v7** | 시간순 정렬 가능한 PK |

### UUID v7

auto increment는 분산 환경에서 충돌 가능. UUID v4는 정렬 불가. UUID v7은 타임스탬프 기반이라 **시간순 정렬 가능**하고 인덱스 성능도 v4보다 낫다.

### 가상 스레드

I/O 바운드 작업이 많으니까 Spring Boot 3 가상 스레드 활성화.

## Claude Code와 으쌰으쌰 구조 잡기

솔직히 11개 모듈 구조를 혼자 잡기는 좀 부담스러웠다. Claude Code랑 같이 논의하면서 "core에는 뭘 넣고, domain에는 뭘 넣고, infra는 어디까지" 같은 경계를 정했다.

여기서 좀 웃긴 에피소드가 있었는데, 프로젝트 폴더 이름이 `app-in-toss be`(공백 포함)이라 Claude가 경로를 `app-in-toss-be`(하이픈)로 추측해서 계속 **"No such file or directory"** 에러를 맞았다. 한참 헤매다가 `find`로 찾아서 겨우 맞은 경로를 잡음. 폴더 이름에 공백 넣지 말자...는 교훈을 얻었다.

그래도 구조 논의 자체는 생산적이었다. "CachePort를 recipe-domain에 둘까 core-domain에 둘까?" 같은 질문을 던지면 의존성 방향을 따져서 답을 줬다. 결론:

핵심 원칙은:
- **core**: 모든 모듈이 의존할 수 있는 공용 코드
- **domain**: 비즈니스 로직만, 인프라 의존 없음
- **infra**: 외부 시스템 연동 (DB, 캐시, 스토리지)
- **app**: 실행 가능한 애플리케이션 (API 서버, 워커)

domain이 infra를 모르게 하는 게 핵심. Port 인터페이스를 domain에 두고, 구현체는 infra에.

131개 파일 초기 세팅을 Claude한테 시키고 나서, build.gradle.kts 의존성이 꼬여있는 부분을 하나하나 잡았다. 생성은 빠른데 검증은 사람이 해야 된다. 빌드폴더도 모듈마다 생겨서 프로젝트 구조가 좀 정신없었다.

그리고 네이밍에서 한참 싸웠다. Claude가 처음에 `recipe-api`라고 만들었는데, 이건 하나의 앱 전용 API가 아니라 **공용 코어 API + 앱별 API** 구조여야 한다. "recipe라고 퉁치지 말고 TasteNote라고 구분을 확실히 해" — 이 구조가 나중에 다른 미니앱 추가할 때 중요하다. 결국 `main-api`로 이름을 바꿨고, 이후에 또 `tastenote-api`로 바꾸게 된다.

이미지 업로드 경로도 고민했다. S3에 날짜별로 폴더를 만드는데 "yyyy-mm-dd 형태로, 뎁스 너무 많이 넣지마" — 년/월/일로 3뎁스 넣으면 관리가 복잡해진다. 압축해서 올리는 것도 확인.
