[앱인토스 Admin] BE-FE API 불일치 전면 수정

By 코드대장간
Posted Mar 25 6 min read

6일 만에 터진 것들

3/19에 camelCase 전환을 하고 6일이 지났다. 그동안 BE 쪽 작업을 하다가 오늘 어드민을 실제로 띄워봤다. 로그인하자마자 에러가 쏟아졌다. PR #3 머지.

문제 1: 로그인이 안 된다

로그인 요청에서 401이 떨어졌다. 이상한 건 아이디/비번이 맞는데도 401이다.

원인은 request interceptor. 모든 요청에 토큰을 붙이는데, 로그인 요청에도 만료된 토큰을 붙이고 있었다. 아직 로그인 전인데 이전 세션의 만료된 토큰을 달고 가니까 서버가 “토큰 잘못됨”으로 거부한 거다. 백화점 입구에서 “어제 지급한 임시 출입증 들고 왜 왔냐”고 막히는 꼴.

PUBLIC_ENDPOINTS 배열을 만들어서 로그인, 토큰 리프레시 요청은 토큰을 스킵하도록 수정. localStorage 직접 접근도 useAuthStore.getState()로 교체해서 상태 관리를 일원화했다.

문제 2: 대시보드 데이터가 안 뜬다

로그인 후 대시보드에서 activities.map is not a function 에러. activities가 배열이 아니라는 뜻.

BE가 Spring의 Page 객체를 그대로 내려주는데, FE 타입이 이걸 몰랐다. Spring Page는 배열이 아니라 객체다.

FE가 기대한 구조BE가 보내는 구조
{ data: [...], page: 1, total: 100 }{ content: [...], number: 0, totalElements: 100 }

data → content, page → number. 이름도 다르고 인덱스도 다르다(1-based vs 0-based). 주소를 “서울시 강남구”로 보내야 하는데 “강남구 서울시”로 보내고 있었다.

PaginatedResponse 타입을 Spring Page 구조에 맞게 전면 수정. hooks 9개, pages 5개, api/types 4개 — 총 18개 파일을 고쳤다.

AuthResponse도 user → admin으로 필드명 변경. BE가 admin이라고 보내는데 FE가 user로 받고 있었다.

자세한 건 학습 포스트에 정리.

스텁 페이지 정리

App.tsx에서 라우팅된 16개 페이지가 “공사 중” 스텁이었다. 근데 서브디렉토리에 실제 구현체가 이미 있었다. 건물은 다 지어놓고 현관문에 “시공 중” 표지판을 안 떼어놓은 거다.

16개 스텁 파일(각 ~17줄)을 1줄짜리 export { default } re-export로 교체. 272줄 삭제, 16줄 추가. Banner, Notice, Report, Maintenance, FeatureFlag, AppVersion, Terms, Featured, AuditLog, SystemStatus, AdminUser, DataExport.

방어 코드 보강

API 에러 시 data?.content.map()에서 content가 undefined면 또 터진다. data?.content?.map()으로 optional chaining 추가. hooks 7개에 catch 블록 state 리셋 추가(setData(null), setUsers([])). SystemStatusPage에서 redis/database/jvm 각 섹션을 조건부 렌더링으로 감쌌다.

안 터지던 코드가 아니라, 정상 응답에서만 안 터지던 코드였다. 에러 응답이 오면 줄줄이 터진다. 비 올 때만 새는 지붕.

Claude Code 협업

“로그인 시 401 에러” → Claude가 request interceptor의 토큰 문제를 짚어냄. “대시보드 activities.map 에러” → BE 응답 구조와 FE 타입을 비교해서 Spring Page 불일치 발견. 디버깅 자체는 Claude가 빠르게 했다.

18개 파일 일괄 수정도 Claude한테 맡겼다. 타입 하나 바꾸면 연쇄적으로 고쳐야 하는 파일이 많은데, 이런 기계적 수정은 Claude가 빠지는 게 없어서 좋다.

16개 스텁 페이지 교체는 worker 2명 + reviewer 1명 구성으로 병렬 처리. operations 8개, 나머지 8개를 나눠서 동시에 처리.

오늘의 결과

항목상태
로그인 interceptor 토큰 스킵 처리완료
PaginatedResponse Spring Page 구조 맞춤완료
AuthResponse.user → .admin 변경완료
18개 파일 일괄 수정완료
16개 스텁 페이지 → re-export 교체완료
optional chaining 방어 처리 (14개 파일)완료
Vite dev proxy 포트 변경 (8081→8083)완료
불필요 API 엔드포인트 제거완료
export 모듈 제네릭 팩토리 리팩터링완료
PR #3 머지완료

FE를 만들 때 BE 응답 구조를 “아마 이렇겠지”로 추정하면 이렇게 된다. 스웨거 문서를 보든 BE를 직접 실행해서 응답을 확인하든, FE 타입 정의는 실제 응답 기준으로 해야 한다.
