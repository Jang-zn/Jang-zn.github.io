## 서비스 레이어에 루프 안에 조회가 있다

TasteNote BE에서 N+1 쿼리를 2곳에서 발견했다. `PantryService`와 `PantryDeductionService`. 재료 목록을 순회하면서 각각 DB를 조회하고 있었다.

## N+1 문제란

목록을 1번 조회하고(N), 각 항목마다 추가 조회 1번씩(+1 × N). 총 N+1번 쿼리. 도서관에서 책 목록을 한번 받고(1), 책 하나마다 서가에 가서 확인하는(N) 거다. 목록에 서가 위치를 같이 적어두면 한번만 가면 된다.

### 탐지 방법

```
repo.findById() 또는 repo.findBy*() 가 for/stream 안에 있으면 N+1
```

이 패턴을 기계적으로 찾을 수 있다. IDE에서 `find` 메서드 호출을 검색하고, 호출 위치가 루프 내부인지 확인.

## 해결: 배치 조회 + Map

```java
// Before: N+1
for (IngredientId id : ingredientIds) {
    Ingredient ing = ingredientRepo.findById(id); // N번 쿼리
}

// After: 1번 쿼리
List<Ingredient> ingredients = ingredientRepo.findByIdIn(ingredientIds);
Map<Long, Ingredient> map = ingredients.stream()
    .collect(Collectors.toMap(Ingredient::getId, Function.identity()));

for (IngredientId id : ingredientIds) {
    Ingredient ing = map.get(id.getValue()); // O(1) 조회
}
```

`findByIdIn()`으로 한번에 가져오고 Map으로 변환. 루프 내부에서는 Map의 O(1) 조회. N+1이 1+0으로 줄어든다.

이미 `CookableRecipeService`에서 이 패턴을 쓰고 있었다. 같은 코드베이스인데 적용된 곳과 안 된 곳이 있었다.

## MySQL 함수 인덱스

요리 연속 기록(Streak) 계산에 `DISTINCT DATE(cooked_at)`이 필요했다. JPQL에서 `CAST(cookedAt AS DATE)`를 쓰면 인덱스를 안 탄다.

### JPQL의 한계

```java
// JPQL — 인덱스 미사용
@Query("SELECT DISTINCT CAST(c.cookedAt AS DATE) FROM CookingLog c WHERE c.userId = :userId")
```

JPQL의 `CAST`는 MySQL의 `DATE()` 함수와 다른 SQL로 변환된다. MySQL 옵티마이저가 함수 인덱스를 인식하지 못한다.

### 네이티브 쿼리 + 함수 인덱스

```sql
-- 함수 인덱스 생성
CREATE INDEX idx_cooking_log_date ON cooking_log ((DATE(cooked_at)));

-- 네이티브 쿼리
@Query(value = "SELECT DISTINCT DATE(cooked_at) FROM cooking_log WHERE user_id = :userId ORDER BY DATE(cooked_at) DESC", nativeQuery = true)
```

`DATE(cooked_at)` 결과를 인덱싱한다. 일반 인덱스는 `cooked_at` 전체 값(날짜+시간)을 인덱싱하는데, 날짜만으로 검색하면 인덱스를 못 탄다. 이름 전체를 색인했는데 성만으로 찾으면 색인이 소용없는 것과 같다.

### Streak 계산은 Java에서

연속 일수 계산을 SQL의 재귀 CTE로 할 수도 있지만, Java에서 날짜 리스트를 순회하며 계산하는 게 더 읽기 쉽다.

```java
// 날짜 리스트가 역순으로 왔다고 가정
int streak = 0;
LocalDate expected = LocalDate.now();
for (LocalDate date : dates) {
    if (date.equals(expected)) {
        streak++;
        expected = expected.minusDays(1);
    } else {
        break;
    }
}
```

SQL에서 복잡한 로직을 짜면 디버깅이 어렵다. DB는 데이터를 가져오는 역할, 비즈니스 로직은 애플리케이션 레이어에서.

## SQL AVG()와 NULL

`AVG(rating)`에서 NULL 행은 자동으로 제외된다. 별점을 안 매긴 요리 기록이 있어도 평균에 영향을 주지 않는다. `COALESCE(AVG(rating), 0)`으로 결과가 NULL일 때(전부 NULL) 0을 반환.
