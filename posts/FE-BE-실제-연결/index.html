[TasteNote] MSW에서 실제 API로 — FE-BE 연결

By 코드대장간
Posted Mar 24 5 min read

MSW를 끄니까 다 깨진다

어제 MSW 토글과 Vite proxy 포트를 맞춰놓고, 오늘 VITE_ENABLE_MSW=false로 실행해서 실제 BE에 연결했다. 예상대로 안 맞는 게 산더미.

MSW 목 데이터와 실제 API 응답 구조가 다르다. 필드명, 중첩 구조, 페이지네이션 형태. 목 데이터로만 개발하면 “잘 되는 것 같은” 환상에 빠진다. 무대 리허설에서는 잘 됐는데 본공연에서 소리가 안 나오는 거다.

페이지네이션 응답 구조

BE가 { items, nextCursor, hasNext } 구조를 반환한다. FE는 data를 바로 배열로 기대하고 있었다. .map() 호출 시 에러. 빈 레시피 목록이 “불러오기 실패”로 표시.

PaginatedResponse 타입을 BE 구조에 맞추고, 접근 경로를 res.data.data → res.data.data.items로 수정. 기존에 content/cursor 필드명을 쓰던 것도 items/nextCursor로 통일.

auth persist

authStore가 메모리에만 저장돼 있었다. 새로고침하면 토큰이 날아가서 로그인 페이지로 리다이렉트. MSW 환경에서는 인증을 생략하니까 이 문제가 안 보였다.

Zustand persist 미들웨어 적용. user, accessToken, refreshToken, expiresIn을 localStorage에 저장. 키는 auth-storage. logout() 호출 시 localStorage도 정리.

등록 플로우 간소화

기존 등록 폼에는 제목, 카테고리, URL 입력이 있었다. 근데 BE에서 AI가 제목/카테고리를 자동 추출한다. 사용자가 입력할 필요가 없다.

제목/카테고리 필드 제거. CategorySelect 컴포넌트 삭제(dead code). useSubmitParseJob에서 title/category 파라미터 제거. 버튼 텍스트도 “등록하기” → “레시피 추출하기”로 변경. URL만 붙여넣으면 끝.

HTTP 메서드 불일치

서버에 PATCH /recipes/:id/versions/:versionId 엔드포인트가 없었다. PUT만 지원. 버전 수정 시 500 에러.

usePatchRecipe에서 client.patch() → client.put()로 변경. MSW 핸들러도 http.patch() → http.put()로 동기화. API 메서드를 바꿨으면 MSW mock도 같이 바꿔야 한다. 안 바꾸면 로컬 개발에서 404.

프록시 포트

로컬 개발환경 분리에서 local API 포트를 8082로 정했다. Vite proxy target도 8180 → 8082로 변경.

코치마크 버그

코치마크에서 skip() 호출 시 완료 저장을 안 해서 매번 튜토리얼이 재시작됐다. coachMarkStorage.markDone() 추가. 타겟 요소가 없어 자동 스킵된 스텝이 totalSteps에 포함되어 “2/4”에서 갑자기 끝나는 문제도 수정. displayIndex를 도입해서 실제 보이는 스텝만 카운트. nextStep()과 skipStep() 중복 로직도 advance() 헬퍼로 통합.

날짜 포맷 통일

RecipeCard에서 ISO 타임스탬프가 그대로 표시되고 있었다. formatDate를 yy-mm-dd 형식으로 통일. formatDate를 사용하는 모든 곳(RecipeCard, HeroImage, Notices)에 일괄 적용.

Claude Code 협업

응답 구조 불일치는 FE에서 에러가 나올 때마다 “BE 응답이 이렇게 오는데 FE 타입 맞춰줘”로 하나씩 수정. auth persist는 “새로고침하면 로그인 풀린다, Zustand persist 적용해”라고 지시.

등록 간소화는 내가 판단. “BE에서 AI가 제목/카테고리를 추출하니까 사용자 입력 필요 없다”고 판단해서 “등록 폼에서 제목/카테고리 필드 제거해, CategorySelect도 삭제”라고 지시.
