⭐2 [학습] 시스템 메시지와 맥락 명시 — Gulf of Execution, Cognitive Load, UX Writing

By 코드대장간
Posted Mar 27 Updated Mar 27 13 min read ⭐2

“이어서 할까요?”는 뭘 이어서 하는 건지 모른다

사용자가 화면을 보고 “지금 무슨 상태인지”, “이 버튼을 누르면 어떻게 되는지”를 예측하지 못하면 불안해한다. 이 불안을 줄이는 게 좋은 시스템 메시지다.

Don Norman의 두 가지 간극

출처: Norman, D.A. (1988). The Design of Everyday Things. Basic Books.

인간-컴퓨터 상호작용에서 사용자가 겪는 어려움을 두 개의 간극으로 설명한다.

실행의 간극(Gulf of Execution) 사용자가 “원하는 결과를 만들려면 어떻게 해야 하는가”를 파악하지 못하는 상태.

  • “이 버튼을 누르면 어떻게 되지?”
  • “내가 원하는 걸 어디서 할 수 있지?”

평가의 간극(Gulf of Evaluation) 사용자가 “행동 후 시스템이 어떤 상태인지”를 인식하지 못하는 상태.

  • “저장이 됐나, 안 됐나?”
  • “내가 한 게 효과가 있었나?”

두 간극이 클수록 UI는 사용하기 어렵다. 시스템 메시지와 버튼 레이블의 역할은 이 두 간극을 좁히는 것이다.

인지 부하 이론(Cognitive Load Theory)

정의: 인간의 작업기억(Working Memory)은 용량이 제한적이며, 인지 부하가 이 용량을 초과하면 학습과 행동이 저해된다.

출처: Sweller, J. (1988). “Cognitive Load during Problem Solving: Effects on Learning”. Cognitive Science, 12(2), 257–285.

세 가지 인지 부하:

유형의미설계 관점
내재적 부하(Intrinsic)과제 자체의 복잡도줄이기 어려움
외재적 부하(Extraneous)나쁜 설계로 인한 불필요한 부하설계로 줄일 수 있음
관련적 부하(Germane)학습/이해에 도움이 되는 부하늘려야 할 때도 있음

시스템 메시지에서 외재적 인지 부하(Extraneous Cognitive Load)가 높아지는 경우:

  • “진행 중인 요리가 있어요” → “어떤 요리인지” 추론해야 함
  • “수정하시겠습니까?” → “뭘 수정하는 건지” 기억해야 함
  • [확인] [취소] → 각 버튼의 결과를 추론해야 함

맥락을 메시지에 명시하면 추론 비용이 사라진다. 이게 외재적 부하를 줄이는 것이다.

기억보다 인식(Recognition over Recall) — Nielsen Heuristic #6

정의: 사용자가 정보를 기억해야 하는 인터페이스보다 화면에서 정보를 인식할 수 있는 인터페이스가 더 좋다.

출처: Nielsen, J. (1994). “10 Usability Heuristics”. Nielsen Norman Group. Heuristic #6.

“Minimize the user’s memory load by making objects, actions, and options visible.”

명령줄 인터페이스가 어려운 이유: 명령어를 기억해야 한다. GUI가 쉬운 이유: 화면에 할 수 있는 것들이 다 보인다.

시스템 메시지에 적용:

  • [이어서] → 뭘 이어서? 기억 필요
  • [된장찌개 이어서 하기] → 화면만 봐도 알 수 있음

기억보다 인식 원칙이 버튼 레이블 설계의 이론적 근거다.

멘탈 모델(Mental Model)

정의: 사용자가 시스템이 어떻게 작동한다고 생각하는 내부 표상(representation).

출처: Norman, D.A. (1988). The Design of Everyday Things. 멘탈 모델은 항상 불완전하고 틀렸을 수 있다. 중요한 건 사용자의 멘탈 모델에 맞는 시스템 언어를 사용하는 것.

시스템 이미지(System Image): 사용자가 UI, 문서, 레이블 등을 통해 시스템에 대해 형성하는 이미지. 이게 멘탈 모델을 만든다.

시스템 이미지와 실제 시스템 모델이 불일치하면 오해가 생긴다.

예: “저장” 버튼이 없는 앱. 개발자 입장에선 “자동 저장”이 당연하지만, 사용자의 멘탈 모델에는 “저장 버튼을 눌러야 저장된다”가 있다. 이 불일치를 해소하려면 화면 어딘가에 “자동 저장됨” 상태 표시가 있어야 한다.

행동 유도성과 표시자(Affordance vs Signifier)

출처: Norman, D.A. (2013). The Design of Everyday Things (개정판). Norman이 초판의 행동 유도성(Affordance) 개념을 표시자(Signifier)로 분리.

행동 유도성(Affordance): 객체가 실제로 어떤 행동을 가능하게 하는가.

  • 문손잡이는 물리적으로 당기거나 밀 수 있음 = 행동 유도성

표시자(Signifier): 가능한 행동을 사용자에게 알려주는 신호.

  • 문에 “당기시오” 스티커 = 표시자

UI 설계에서 실제 중요한 건 표시자다. 버튼이 클릭 가능하다는 행동 유도성이 있어도 표시자(텍스트, 모양, 색)가 없으면 사용자는 모른다.

버튼 레이블은 표시자다. 뭘 할 수 있고, 누르면 어떤 결과가 오는지를 알려줘야 한다.

역방향 설계(Reverse Design)

정의: “무엇을 보여줄 것인가”를 먼저 결정하고, 그에 필요한 데이터/시스템을 역으로 설계하는 방법.

교육학의 Backward Design(Wiggins & McTighe, 1998)에서 온 개념. 원래는 “학습 목표 → 평가 → 수업 설계” 순서. UX에서는 “최종 화면 → 필요 데이터 → 데이터 모델 → 구현” 순서.

1
2

일반적 설계: 기능 구현 → UI 설계 → "이 데이터가 없네?"
역방향 설계: 화면 확정 → 필요 데이터 목록 → 데이터 모델 → 구현

역방향 설계의 장점: 뒤늦은 데이터 모델 수정이 줄어든다. “나중에 보여주려 했는데 저장한 게 없다”가 발생하지 않는다.

좋은 시스템 메시지의 3요소

상황 + 맥락 + 행동

요소의미나쁜 예좋은 예
상황지금 무슨 일인가(없음)“진행 중인 요리가 있어요”
맥락무엇에 대한 이야기인가(없음)“된장찌개를 하다 멈췄어요”
행동각 선택지의 결과[이어서] [취소][된장찌개 이어서 하기] [김치찌개 새로 시작]

셋 중 맥락이 가장 자주 빠진다. 시스템은 “어떤 세션인지” 알지만, 사용자가 모르는 걸 당연하게 생각한다. Norman의 평가의 간극(Gulf of Evaluation)이 생기는 지점이다.

버튼 레이블 원칙

기억보다 인식(Recognition over Recall) + 실행의 간극(Gulf of Execution) 최소화

좋은 버튼 레이블 체크리스트:

원칙나쁜 예좋은 예
동사 포함“수정”“수량 변경하기”
행동 대상 명확“삭제”“이 재료 제외하기”
가역성 암시“삭제”(되돌리기 어려움)“숨기기”(다시 볼 수 있음)
범위 명확“수정”(뭘 수정하는지 모름)“수량 변경”(수량만)
목적어 포함“이어서”“된장찌개 이어서 하기”

UX Writing 4원칙 (Nielsen Norman Group):

  • Clear: 의미가 모호하지 않다
  • Concise: 불필요한 단어가 없다
  • Useful: 사용자가 필요한 정보를 준다
  • Human: 로봇 같지 않다

“성공적으로 완료되었습니다” → “저장했어요” (Clear + Concise + Human)

Microcopy

정의: UI에 있는 짧은 텍스트. 버튼 레이블, 플레이스홀더, 에러 메시지, 툴팁, 확인 메시지 등.

작지만 사용자 경험에 큰 영향을 준다. 같은 행동을 유발하는 버튼이라도 텍스트에 따라 전환율이 달라진다.

에러 메시지 작성 원칙 (Nielsen, 1994 Heuristic #9):

  • 무슨 문제인지 (상황)
  • 왜 문제가 생겼는지 (이유)
  • 어떻게 해결하는지 (행동)

나쁜 에러: “오류가 발생했습니다” 좋은 에러: “인터넷 연결이 필요해요. Wi-Fi나 데이터를 켜고 다시 시도해보세요”

TasteNote 적용

CookingResumeSheet에서 “이어서 할까요?” — 상황은 있지만 맥락이 없었다. 레시피가 1개일 때는 문제없었지만 여러 개가 되면서 평가의 간극(Gulf of Evaluation)이 생겼다.

역방향 설계(Reverse Design): “버튼에 레시피명을 넣어야 한다” → “cookingSessionStore에 recipeName 없다” → 세션 시작 시 recipeName 주입 → 메시지에 명시.

재료 수정 버튼 “수정” → “수량 변경”. “수정”은 범위가 넓다. 실제로는 수량만 바뀌는데 이름도 수정될 것 같아서 눌러봐야 알았다. 표시자(Signifier)가 행동을 잘못 암시하고 있었다.
