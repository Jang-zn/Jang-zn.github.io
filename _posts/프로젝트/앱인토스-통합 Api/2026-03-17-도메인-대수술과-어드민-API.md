## recipe-domain → tastenote-domain 리네임

[초기 세팅 때](/posts/멀티모듈-백엔드-세팅/) `recipe-domain`이라고 이름 지었는데, 이게 계속 거슬렸다. TasteNote 앱은 레시피만 다루는 게 아니라 사용자, 버전 관리, 재료 등 여러 도메인이 있다. "recipe라고 퉁치지 말고 TasteNote라고 구분을 확실히 하라고" — [저번에도 같은 지적을](/posts/멀티모듈-백엔드-세팅/) 했는데, 결국 이번에 전면 리네임.

같은 논리로:
- `main-api` → `tastenote-api`
- `parse-worker` → `tastenote-worker`

이름이 중요하다. 다른 미니앱이 추가되면 `someapp-domain`, `someapp-api`로 확장되니까.

## core-domain 분리

제일 큰 작업. `tastenote-domain`에 있던 **User**랑 **Port 인터페이스**를 `core-domain`으로 추출.

어드민 API에서도 User가 필요한데 `tastenote-domain`에 의존하면 의존성이 꼬인다. 공용 모델은 분리.

```
before: tastenote-domain (User + Recipe + Ingredient + Ports)
after:  core-domain      (User + 공용 Ports)
        tastenote-domain (Recipe + Ingredient + 앱 전용 Ports)
```

덕분에 `admin-api`가 `core-domain`만 의존하고, `tastenote-domain`은 안 건드려도 되게 됐다.

## admin-api 모듈 추가

어드민 API 모듈 추가. **인증/인가** 구조가 핵심.

### 토큰 인증 + 역할별 권한

```
역할 3단계:
- VIEWER: 조회만 가능
- ADMIN: 조회 + 수정
- SUPER_ADMIN: 전체 권한 (계정 관리, 시스템 설정)
```

JWT access token은 15분, refresh token은 Redis에 저장. 어드민은 보안이 중요하니까 토큰 수명을 짧게 잡았다. 기본 어드민 ID/PW랑 토큰 비밀키는 코드에 직접 적으면 안 된다. `.env` 파일에서 환경변수로 빼놨다. 이건 원칙이다.

### 활동 기록

어드민에서 누가 뭘 했는지 전부 기록해야 한다. 데이터 변경 이력은 자동 추적되게 했고, 별도 로그 테이블에도 남긴다.

## DB V3 마이그레이션

어드민용 테이블들을 추가하는 DB 변경 스크립트 V3. admin_users, audit_logs, app_configs, notices, banners 등등.

## 이미지 WebP 지원

WebP를 지원 안 하고 있었다. `twelvemonkeys` 추가해서 WebP → JPEG 변환. 픽셀 카운트 검증도 추가.

## 기타 인프라 개선

오늘 커밋이 정말 많았다:

- **UuidV7Generator를 실제 UUID v7으로 구현** — 기존에 임시 구현이었던 걸 제대로
- **EntityScan 범위를 앱별로 정밀 지정** — 불필요한 엔티티 로딩 방지
- **actuator 엔드포인트 보안 강화** — `/actuator/**` 접근 제한. 코드 리뷰에서 노출 위험 잡음
- **CORS 설정을 Spring Security 레벨로 통합** — `https://tastenote.apps.tossmini.com` 허용 추가. 토스 배포 가이드 준수사항
- **재료 수정 시 중복 체크 + 병합 시 참조 이관**
- **spring.profiles.active 하드코딩 제거**
- **어드민 기본 계정/토큰 비밀키를 코드에서 분리**

## Claude Code "code-forge-toss" 팀 — 삽질과 교훈

오늘 제일 재밌었던(?) 건 Claude Code의 **Team 기능** 시도. 여러 에이전트가 동시에 태스크를 처리할 수 있다길래, **"code-forge-toss"** 팀을 만들었다.

근데 첫 시도부터 막혔다.

> "Already leading team 'code-forge-toss'. A leader can only manage one team at a time."

이전에 만들다 만 팀이 남아있었던 거다. cleanup하고 다시 만들고, 태스크 11개를 생성해서 의존성 관계(blockedBy)도 설정했다. `.gitignore` → `core-exception` → `recipe-domain` → ... 순서대로.

결과적으로 팀 기능이 대규모 모듈 작업을 병렬로 처리하는 데 꽤 도움이 됐다. 하나의 에이전트가 순차적으로 하면 한세월인데, 여러 에이전트가 동시에 돌아가니까 체감 속도가 확 달랐다.

다만 에이전트끼리 충돌하는 경우도 있었고, 한 에이전트가 다른 에이전트의 결과물을 기다려야 하는데 의존성 설정이 완벽하지 않으면 빌드가 깨졌다. 이 부분은 사람이 직접 순서를 관리해줘야 한다.
