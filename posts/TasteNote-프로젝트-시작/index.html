[TasteNote] 프로젝트 시작 — 하루만에 Phase 2까지

By 코드대장간
Posted Mar 13 Updated Mar 18 6 min read

시작

토스 미니앱으로 레시피 공유 앱을 만들기로 했다. 이름은 TasteNote. URL 하나 던지면 레시피를 파싱해서 정리해주고, 나만의 버전으로 수정도 할 수 있는 앱.

혼자 한다. FE, BE, 인프라 전부. 선택지가 없으니까 FE부터 시작.

구조부터 잡았다

나중에 고통받기 싫으면 처음에 구조를 잡아야 한다. 기능 단위로 폴더를 나누는 구조로 잡았다.

1
2
3
4

src/
├── pages/        ← 라우트 단위 페이지
├── features/     ← 기능 단위 (auth, recipe-list, recipe-detail...)
├── entities/     ← 도메인 모델, 매퍼
├── shared/       ← 공용 유틸, UI 컴포넌트, API 클라이언트

문제는 토스 네이티브 환경 vs 웹 로컬 개발 양립.

Adapter 패턴

토스 미니앱이라 toss-bridge를 써야 하는데, 로컬에는 토스 앱이 없다. Adapter 패턴으로 해결.

1
2
3

// NavigationAdapter - 토스 환경이면 toss-bridge, 아니면 react-router
// BridgeAdapter - 토스 API 호출 or 웹 폴백
// AdAdapter - 토스 광고 or 구글 애드몹 (이건 나중에 추가)

로컬에서 npm run dev 하면 웹으로 돌아가고, 토스에 올리면 네이티브로 돌아간다. 출시 전에 토스 웹뷰 호환성도 확인해야 하는데, 이건 나중에 토스 개발자 문서 보고 점검할 예정.

MSW v2로 목 API — BE 없이 FE 먼저

백엔드가 아직 안 되어 있다. 근데 BE 기다리고 있을 여유가 없다. MSW(Mock Service Worker) v2로 전체 API를 목으로 만들었다. 레시피 CRUD, 인증, 검색 전부.

Raw 타입(snake_case)으로 응답 주고 → Mapper에서 camelCase 도메인 모델로 변환하는 구조. 여기서 내가 고민한 건, 나중에 실제 API 붙일 때 얼마나 갈아엎어야 하느냐는 것. 변환기(Mapper)를 중간에 껴두면 서버 응답 형식이 바뀌어도 변환기만 수정하면 된다. 다만 이게 이론대로 될지는 실제 연동할 때 검증해야 한다. “api 서버랑 연결을 해야 phase2 이후를 할 수 있겠네” — 이건 처음부터 알고 있었던 한계점이다.

Phase 0 → 1 → 2, Claude Code랑 으쌰으쌰

혼자 이 양을 하루에 끝내는 건 물리적으로 불가능하다. Claude Code를 썼다.

구조 설계를 주면 Claude가 파일을 생성한다. 근데 그냥 받아쓰면 안 된다. Claude가 파일 쓸 때마다 승인/거부를 반복했다. 방향이 틀어지면 중간에 끊고 다시 시킨다. 가끔 엉뚱한 걸 만들어오는데, 이건 어쩔 수 없다. 대화하면서 의도를 맞춰가는 수밖에.

  • Phase 0: 프로젝트 기본 뼈대 세팅, 디자인 토큰, 도메인 타입, 라우터/브릿지 어댑터, API 인프라, MSW 목, 앱 셸
  • Phase 1: 공유 UI 컴포넌트, 엔티티 유틸, 홈/상세/수정/등록 페이지
  • Phase 2: 통합, 인증 기능, 빌드 수정, 모든 페이지 완성

하루 만에 Phase 2까지 전부 완성. 빌드 결과물 344KB.

그리고 API 엔드포인트에서 버전 경로(/v1/) 제거하는 리팩토링이랑 인증 플로우 개선까지 마무리.

코드베이스 탐색

다 만들고 나서 Claude한테 “코드베이스 전체 탐색해서 정리해줘”라고 시켰다. 내 코드를 리포트로 정리해주니까 머릿속 정리에 도움이 됐다. 디렉토리 트리, 타입 시스템, 매퍼 패턴, 스토어 구조.

근데 탐색만 시켰는데 파일을 수정하려 해서 거부했다. 시키지 않은 걸 자기가 알아서 하려는 건 좀 거슬린다. 탐색/문서화 능력 자체는 쓸만하다.

BE 쪽을 봐야 한다.
