## 캐시 하나로는 부족하다

[BE 프로젝트](/posts/멀티모듈-백엔드-세팅/)에서 레시피 파싱 기능을 만들면서, 캐시가 한 종류로는 부족하다는 걸 느꼈다. URL 중복 체크, AI 응답 재사용, 레시피 조회 — 목적이 전부 다르다. TTL도 다르고 보호하려는 대상도 다르다.

냉장고에 반찬이랑 식재료를 같이 넣으면 관리가 안 된다. 반찬은 3일 안에 먹어야 하고, 냉동 식재료는 한 달 가니까. 용도가 다르면 관리도 달라야 한다. 그래서 **캐시를 2개 레이어**로 나눠서 적용했다.

## 2개 레이어

### Layer 1: URL → recipeId (TTL 7일)

이미 파싱 완료된 URL인지 확인. 이게 있으면 잡 생성 자체를 안 한다. Gemini API 호출 비용을 아끼는 게 목적.

```
요청: "이 URL 파싱해줘"
→ 캐시에 있음 → "이미 했는데, 결과 여기" (API 호출 0번)
→ 캐시에 없음 → 잡 생성 → Gemini 호출
```

### Layer 2: recipeId → 레시피 데이터 (TTL 10분)

일반적인 DB 조회 캐시. 레시피 상세 페이지에서 같은 레시피를 여러 번 조회할 때 DB 부하를 줄인다. [캐시 스탬피드 방지](/posts/캐시-스탬피드/)도 이 레이어에 적용.

## TTL 설계

| 레이어 | TTL | 이유 |
|--------|-----|------|
| URL → recipeId | 7일 | 같은 URL의 레시피가 바뀔 일이 거의 없다 |
| recipeId → 데이터 | 10분 | 사용자가 레시피를 수정할 수 있으니 짧게 |

TTL이 짧으면 데이터가 신선하고 캐시 효과가 적다. 길면 반대. 각 레이어의 데이터 특성에 맞춰 잡았다.

## 캐시 무효화

사용자가 레시피를 수정하면 Layer 2 캐시를 지워야 한다. 안 지우면 수정 전 데이터가 10분간 보인다.

```java
// 레시피 수정 시
recipeRepository.save(recipe);
cachePort.evict("recipe:" + recipeId);  // 캐시 삭제
```

Layer 1은 무효화할 필요가 없다. URL→레시피 매핑은 바뀔 일이 거의 없으니까.

## 캐시 키 설계도 고민이었다

캐시를 나누면 키가 겹치는 문제가 생긴다. 작년 학습 과정에서 여러 서비스가 같은 캐시 키를 쓰면 충돌 난다는 걸 배웠었다. 그때 `CacheKeyConstants`로 키 네이밍 규칙을 정하고, `keyGenerator` 패턴으로 키를 생성하는 방식을 썼는데, 이번에도 비슷하게 접두어로 용도를 구분했다.

```
parse-result:{urlHash}   ← URL 파싱 결과 캐시
recipe:{recipeId}         ← 레시피 상세 캐시
lock:parse-job:{urlHash}  ← 분산 락용 (캐시 아님)
```

접두어가 없으면 나중에 "이 키가 뭐였지?" 하고 헤맨다.

## 작년에는 단층이었다

작년 학습 과정에서 Redis 캐싱을 처음 적용했을 때는 캐시가 하나였다. 상품 랭킹을 Redis Sorted Set에 넣고 TTL 7일로 돌렸는데, 그때는 "캐시 = Redis에 넣고 빼기" 정도의 이해였다.

이번에 레이어를 2개로 나누면서, 같은 Redis라도 **뭘 보호하느냐에 따라** 키 설계랑 TTL이 완전히 달라진다는 걸 체감했다. API 비용을 아끼려는 캐시(7일)랑 DB 부하를 줄이려는 캐시(10분)는 목적이 다르니까 전략도 달라야 한다. 수업 때는 그냥 "캐시 넣으면 빨라진다" 수준이었는데, 실제 프로젝트에서 용도별로 나누는 판단을 하게 되니까 그때 쌓아둔 기초가 도움이 됐다.

## 정리

캐시는 만능이 아니고, 하나로 다 해결하려고 하면 TTL 설정이 어정쩡해진다. 용도별로 나누니까 각각의 TTL과 무효화 전략을 독립적으로 가져갈 수 있었다.
