⭐5 [학습] 이름 기반 매칭과 문자열 정규화

By 코드대장간
Posted Mar 26 Updated Mar 26 5 min read ⭐5

“닭가슴살”과 “닭가슴살 “은 다르다

TasteNote Pantry를 독립화하면서 CookableRecipeService가 ingredient ID 매칭에서 이름 매칭으로 바뀌었다. 냉장고의 재료 이름과 레시피의 재료 이름을 비교해서 만들 수 있는 레시피를 찾는다.

이름 매칭은 간단해 보이지만 함정이 있다.

ID 매칭 vs 이름 매칭

 ID 매칭이름 매칭
정확도완벽표기 불일치 시 실패
사전 작업연결 필요불필요
유연성낮음높음
적합한 경우데이터 정규화된 경우자유 입력 허용 시

Pantry 독립화 후에는 사용자가 자유롭게 재료를 입력한다. “간장”, “국간장”, “진간장”이 모두 다른 항목. ID 매칭은 연결 작업이 선행돼야 하므로 사용 불가.

이름 매칭의 함정 — 공백

1
2
3

// "닭가슴살"과 "닭가슴살 " (trailing space)
"닭가슴살".contains("닭가슴살 "); // false
"닭가슴살 ".contains("닭가슴살");  // true (반대는 됨)

사용자가 재료를 입력할 때 trailing space를 붙이는 경우가 있다. 모바일 키보드에서 특히 자주 발생한다. 공백 하나 차이로 매칭이 실패한다.

strip() vs trim()

1
2

"닭가슴살 ".trim();   // "닭가슴살" — ASCII 공백(0x20 이하)만 제거
"닭가슴살 ".strip();  // "닭가슴살" — 유니코드 공백 포함 제거

trim()은 Java 1.0부터 있던 메서드. ASCII 공백(코드 0x20 이하)만 제거한다. strip()은 Java 11에서 추가. Character.isWhitespace()를 기준으로 유니코드 공백까지 제거한다.

문제가 되는 유니코드 공백:

  • \u00A0 — Non-breaking space (복사-붙여넣기 시 자주 섞임)
  • \u3000 — 전각 공백 (일본어/한국어 입력기)

trim()으로 제거 안 되면 매칭 실패. strip()이 안전하다.

정규화 적용 위치

저장 시점과 비교 시점 모두 정규화해야 한다.

// 저장 시
public PantryItem addItem(String name, ...) {
    String normalized = name.strip();
    // normalized로 저장
}

// 비교 시
boolean matches(String pantryName, String recipeName) {
    return pantryName.strip().equals(recipeName.strip());
}

저장 시에만 정규화하면: 정규화 전에 이미 저장된 데이터와 비교할 때 실패. 비교 시에만 정규화하면: 매번 정규화 연산이 발생 (저장된 데이터가 많으면 성능 저하).

빈 컬렉션 조기 종료

냉장고가 비어있으면 만들 수 있는 레시피가 없다. 빈 냉장고로 쿼리를 날릴 필요가 없다.

사무실에 아무도 없으면 회의실 예약 현황을 확인할 필요 없다.

Fake Repository 방어 코드

인메모리 Fake Repository는 DB의 NOT NULL 제약을 강제하지 않는다. null 이름이 저장될 수 있다.

1
2
3
4
5

// Fake에서 이름 매칭 시
store.values().stream()
    .filter(Objects::nonNull)           // null 엔티티 방어
    .filter(p -> p.getName() != null)   // null 이름 방어
    .filter(p -> p.getName().strip().equals(target))

실제 DB는 NOT NULL이라 null이 들어올 수 없지만, 테스트 Fake는 그 제약이 없다. 테스트가 NullPointerException으로 터지면 원인 찾기가 어려워진다.
