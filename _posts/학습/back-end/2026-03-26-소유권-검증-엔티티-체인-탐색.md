## 소유자 정보가 조상 엔티티에 있다

TasteNote에서 `RecipeIngredient`를 수정할 때 "이 사용자가 이 재료를 수정할 권한이 있는가?"를 검증해야 한다. 문제는 `RecipeIngredient`에 `userId`가 없다는 것.

```
RecipeIngredient → RecipeVersion → Recipe.userId
```

`userId`가 3단계 위에 있다. 어떻게 검증하나.

## 3-hop 체인 탐색

```java
RecipeIngredient ingredient = ingredientRepo.findById(ingredientId)
    .orElseThrow(() -> new AppException(ENTITY_NOT_FOUND));

RecipeVersion version = versionRepo.findById(ingredient.getRecipeVersionId())
    .orElseThrow(() -> new AppException(ENTITY_NOT_FOUND));

Recipe recipe = recipeRepo.findById(version.getRecipeId())
    .orElseThrow(() -> new AppException(ENTITY_NOT_FOUND));

if (!recipe.getUserId().equals(requestingUserId)) {
    throw new AppException(FORBIDDEN);
}
```

단계마다 `orElseThrow`로 없으면 바로 404. 최종 단계에서 소유자 비교 후 403.

비슷한 패턴이 곳곳에 있다:
- `Comment` 삭제 → `Post.userId` 확인
- `OrderItem` 수정 → `Order.userId` 확인
- `RecipeIngredient` 수정 → `Recipe.userId` 확인

## FORBIDDEN(403) vs NOT_FOUND(404)

소유자가 아닌 사람이 접근했을 때 403을 내야 하나, 404를 내야 하나.

| | 403 FORBIDDEN | 404 NOT_FOUND |
|---|---|---|
| 의미 | "있긴 한데 권한 없음" | "모르겠음" |
| 리소스 존재 노출 | O | X |
| 공격자 관점 | 존재는 알 수 있음 | 존재 여부 불명 |
| 적합한 경우 | 내부 관리 API, 본인 데이터 수정 | 공개 API |

SNS에서 내 DM을 다른 사람이 URL로 직접 접근할 때는 404가 맞다. DM이 존재하는지조차 알려주면 안 된다. 하지만 본인 레시피 수정 같은 맥락에서는 403이 명확하다. "네 레시피가 아니야"라고 알려주는 게 UX에서도 낫다.

이 프로젝트는 본인 데이터 수정 맥락이므로 403을 선택.

## 3-hop의 성능 비용

SELECT가 3번 발생한다. JOIN 쿼리 하나로 줄일 수 있다.

```java
// 최적화 버전 — 1번 쿼리
@Query("SELECT r.userId FROM RecipeIngredient ri " +
       "JOIN RecipeVersion rv ON ri.recipeVersionId = rv.id " +
       "JOIN Recipe r ON rv.recipeId = r.id " +
       "WHERE ri.id = :ingredientId")
Optional<Long> findOwnerUserId(Long ingredientId);
```

단, 이 API는 어드민이 레시피 재료를 수동 연결할 때만 사용하는 저빈도 기능이다. YAGNI 원칙 적용 — 실제로 성능 문제가 발생하면 그때 최적화한다. 지금 코드를 복잡하게 만들 이유가 없다.

## 패턴 정리

"X를 수정하고 싶은데 소유자 정보가 X의 조상 Y에 있다"는 상황의 처리 방법:

1. X 조회 → 없으면 404
2. 체인 탐색으로 소유자 엔티티까지 올라감 → 각 단계 없으면 404
3. 소유자 비교 → 불일치면 403

체인이 3단계를 넘으면 설계를 재검토할 타이밍이다. `RecipeIngredient`에 직접 `userId`를 비정규화해서 넣는 것도 방법이다. 성능과 정합성 유지 비용 사이의 트레이드오프.
