## 어제까지의 UX, 오늘 정리

[어제 FAB 메뉴와 UX 개선](/posts/FAB-메뉴와-UX-개선/)을 대거 했다. 오늘은 알림 UX를 통일하고, 인앱결제 구매 화면을 만들었다. PR 2개 머지.

## 캐시 히트 알림 통일 — PR #18

레시피 URL을 등록하면 Gemini가 파싱한다. 근데 이미 파싱된 URL이면 캐시에서 바로 가져온다. 문제는 이 두 경로의 UI가 달랐다.

- SSE 파싱: 홈 화면 하단 스낵바에서 진행률 표시
- 캐시 히트: 등록 페이지에 오버레이가 뜸

같은 "레시피 등록 완료"인데 화면이 다르면 사용자가 헷갈린다. 결혼식장에서 신랑 측 하객은 A 입구, 신부 측은 B 입구로 들어가는데 결국 같은 홀에 앉는 거다. 입구를 하나로 합쳤다.

`parseJobStore`에 `cachedRecipeId` 상태를 추가. 캐시 히트면 이 값을 세팅하고 홈으로 이동. `ParseJobSnackbar`가 캐시 히트를 감지하면 4초 타이머 후 상세 페이지로 이동. 미사용된 `ParseProgress` 컴포넌트는 삭제.

## 토스트 상단 + 색상 분리

토스트가 하단에 떴다. 모바일에서 하단은 엄지 영역이라 토스트가 가려지기 쉽다. 상단으로 옮기고 `safe-area-inset-top`으로 노치 영역 회피.

`showToast`에 `type` 옵션을 추가. 성공은 녹색, 에러는 붉은색. 기존에는 전부 같은 색이라 "성공인지 실패인지" 읽어봐야 알았다. 신호등 색이 다 회색이면 뭔 소용인가.

레시피 수정 완료 시 토스트도 추가. `RecipeEditPage`의 `onAllDone`에 `showToast` 호출 한 줄.

## 인앱결제 Bridge — PR #19

[BE에서 인앱결제 API를 만들었다](/posts/인앱결제-광고-제거-시스템/). FE에서 Toss SDK를 통해 결제하고 서버에 결과를 보내야 한다. 토스 미니앱에서 결제하려면 Bridge를 통해 네이티브 SDK를 호출한다.

`BridgeAdapter`에 3개 메서드 추가:

- `processProductGrant`: Toss 결제 SDK 호출
- `completeProductGrant`: 서버에 결제 완료 알림
- `getPendingOrders`: 미지급 주문 복원

`TossBridgeAdapter`에서 dynamic import로 Toss SDK를 로드. 빌드 타임에 SDK가 없어도 된다.

## 구매 feature 모듈

`features/purchase/`를 새로 만들었다.

### 훅 3개

- `useAdFreeStatus`: 광고 제거 상태 조회. 캐시 5분
- `useGrantPurchase`: 서버에 구매 부여 요청
- `useRequestRefund`: 환불 요청

### 구매 플로우 오케스트레이션

`usePurchaseFlow`가 전체 플로우를 관리한다.

1. Bridge로 Toss 결제 SDK 호출
2. 결제 성공하면 서버에 `grant` 요청
3. 서버 응답 받으면 Bridge에 `complete` 알림
4. 앱 진입 시 `getPendingOrders`로 미지급 주문 복원

3단계가 왜 필요한가. 결제는 됐는데 서버 응답이 안 오면 Toss SDK가 "미지급" 상태로 보관한다. 다음에 앱을 열면 미지급 건을 다시 처리한다. 택배가 문 앞에 왔는데 수취인이 없으면 내일 다시 오는 거다.

## 구매 페이지

`PurchasePage`에 상품 카드 2개. 영구(9,900원), 월간(2,000원). 상단에 현재 상태 배지(활성/비활성). 하단에 `PolicyBottomSheet`로 환불 정책 안내.

설정 페이지에 "광고 제거" 메뉴 항목 추가. MSW에 4개 엔드포인트 핸들러 추가(ad-free, grant, history, refund-request).

analytics 이벤트 6개 등록. blue 컬러 토큰 추가(#3182F6, Toss 브랜드 컬러).

## Claude Code 협업

UX 통일 작업(캐시 히트 알림, 토스트 색상)은 "이런 문제가 있다, 이렇게 바꿔"로 지시. 단순한 상태 관리 변경이라 Claude가 깔끔하게 처리.

인앱결제는 "BE API 가이드 문서를 참고해서 FE 구매 모듈 만들어줘"로 지시. BE 구현 때 FE 참조 문서를 같이 생성해둔 게 여기서 써먹혔다. `usePurchaseFlow`의 3단계 오케스트레이션 구조는 Claude가 제안한 건데, 미지급 주문 복원까지 고려한 게 괜찮았다.
