Windows에서 Claude Code가 죽는 버그

증상: 대시보드를 켜면, 별도 터미널에서 돌고 있던 Claude Code가 종료된다.

대시보드가 활성 세션 목록을 보여주려면 각 Claude Code 프로세스가 살아있는지 확인해야 한다. Unix에서의 표준 방법은 os.kill(pid, 0). signal 0을 보내는 거다. 이 signal은 “아무 것도 안 하지만, 프로세스가 존재하면 성공, 없으면 에러”다. 프로세스 존재 확인용.

Windows에서 signal 0은 다른 의미다. CTRL_C_EVENT가 된다. Ctrl+C.

“살아있니?” 물으려고 주먹을 날리는 셈이다. 대답 대신 죽는다.

이걸 1초마다 한다. 대시보드가 매 리프레시마다 세션 목록을 갱신하니까. Claude Code를 켜놓고 대시보드를 띄우면, 1초 안에 Claude Code가 꺼진다.

Win32 API로 교체

ctypes로 Windows 커널 API를 직접 호출한다.

handle = ctypes.windll.kernel32.OpenProcess(
    _PROCESS_QUERY_LIMITED_INFORMATION, False, pid)
if not handle:
    return False
exit_code = ctypes.c_ulong()
ok = ctypes.windll.kernel32.GetExitCodeProcess(
    handle, ctypes.byref(exit_code))
return bool(ok) and exit_code.value == _STILL_ACTIVE

OpenProcess로 핸들을 열고, GetExitCodeProcess로 종료 코드를 확인한다. STILL_ACTIVE(259)면 아직 돌고 있는 거다. 프로세스에 아무 signal도 보내지 않는다. 조용히 상태만 확인.

PROCESS_QUERY_LIMITED_INFORMATION 권한이면 충분하다. 프로세스를 제어하는 게 아니라 정보만 읽는 거니까. ctypes는 Python 내장 모듈이라 추가 설치도 필요 없다.

플랫폼 분기는 모듈 로드 시점에 한다.

함수 자체가 플랫폼별로 다르게 정의된다. 호출할 때마다 if win32 체크를 안 해도 된다.

/simplify가 잡아낸 것들

Win32 API 교체 후 Claude Code에 /simplify를 돌렸다. 코드 품질 검수 용도. 세 가지를 잡아냈다.

(a) GetExitCodeProcess 반환값 미체크

처음 코드:

1
2

ctypes.windll.kernel32.GetExitCodeProcess(handle, ctypes.byref(exit_code))
return exit_code.value == _STILL_ACTIVE

GetExitCodeProcess가 실패하면(잘못된 핸들 등) exit_code가 초기값 그대로다. c_ulong()의 기본값은 0. 0은 STILL_ACTIVE(259)가 아니니까 “죽었다”로 판정된다. 운이 좋으면 맞는 결과지만, API 실패를 정상 결과로 착각하는 구조다.

수정: bool(ok) 체크 추가. API가 성공했을 때만 exit_code를 신뢰한다.

1
2

ok = ctypes.windll.kernel32.GetExitCodeProcess(handle, ctypes.byref(exit_code))
return bool(ok) and exit_code.value == _STILL_ACTIVE

이게 프로덕션에서 터지면 찾기 어려운 버그다. “가끔 세션이 사라졌다 나타났다 한다” 류의 증상으로 나올 거다.

(b) ctypes import와 상수가 함수 안에

처음에 import ctypes와 _PROCESS_QUERY_LIMITED_INFORMATION = 0x1000 같은 상수 선언이 _is_pid_alive 함수 본문 안에 있었다.

이 함수는 1초마다 호출된다. 세션이 5개면 1초에 5번. Python은 import를 캐싱하긴 하지만, 매번 import 메커니즘을 타는 건 불필요한 오버헤드다. 상수 바인딩도 매번 재실행.

모듈 레벨로 끌어올렸다. if sys.platform == "win32": 블록 안에서 한 번만 선언.

(c) sys.platform 분기가 매 호출마다

초기 구조:

1
2
3
4
5

def _is_pid_alive(pid):
    if sys.platform == "win32":
        # Win32 방식
    else:
        # Unix 방식

sys.platform은 런타임에 안 바뀐다. Windows에서 시작했으면 끝까지 Windows다. 매 호출마다 체크할 필요가 없다.

import 시점에 한 번만 평가하도록, 플랫폼별로 함수 자체를 다르게 정의했다. if sys.platform == "win32": 아래에 Win32 버전 함수를 정의하고, else: 아래에 Unix 버전을 정의. 런타임에는 이미 결정된 함수만 호출.

세 가지 다 “동작은 하는데 비효율적이거나 잠재적 버그”인 것들이다. /simplify 없이 내가 직접 봤으면 (b)와 (c)는 넘어갔을 거다.

1초 리프레시 결정

처음에는 5초 간격이었다. 쓰다 보니 답답하다. 토큰을 쓰면 바로 게이지에 반영되길 원하는데, 5초는 느리다.

성능을 분석했다.

항목비용
파일시스템 I/O (iterdir + stat)~1-2ms
JSONL 증분 파싱~5-10ms
세션 읽기 + PID 체크~3-6ms
집계 루프 (5000~10000 레코드)~10-30ms
총합~20-50ms

1초 예산에서 20-50ms. 5% 미만. CPU도 안 잡아먹는다. 여유 있다. 바꾸기로 했다.

set_interval의 인자를 config.display.refresh_interval에서 읽는다. 기본값 1초. 사용자가 설정 파일에서 바꿀 수도 있다.

_record_cache 폭발 문제

1초 리프레시로 바꾸고 나서 잠재적 문제를 발견했다.

_record_cache는 증분 파싱한 레코드를 계정별로 쌓아두는 딕셔너리다. 매 리프레시마다 새 레코드를 extend한다. 문제는 상한이 없다는 거다.

냉장고에 음식을 넣기만 하고 안 버리면 결국 문이 안 닫힌다.

장시간 대시보드를 켜놓으면(하루, 이틀…) 레코드가 계속 쌓인다. 집계 루프가 전체 레코드를 순회하니까, 레코드가 늘수록 느려진다. 5초 간격이었을 때는 체감이 안 됐지만, 1초 간격에서는 누적 효과가 생긴다.

lookback 기간(30일) 초과 레코드를 trim한다.

1
2
3
4

if new_records:
    _record_cache[key].extend(new_records)
    cutoff = datetime.now(timezone.utc) - timedelta(days=parse_lookback)
    _record_cache[key] = [r for r in _record_cache[key] if r.timestamp >= cutoff]

핵심: if new_records:. 새 레코드가 없으면 trim 자체를 스킵한다. 1초마다 리프레시해도 매번 새 레코드가 있는 건 아니다. Claude Code를 안 쓰고 있으면 새 JSONL 줄이 안 생긴다. 변한 게 없는데 O(n) 리스트 컴프리헨션을 매 초 돌릴 이유가 없다.

평균 토큰/턴 계산 버그

Token Usage 패널에 모델별 평균 토큰/턴을 보여준다. 처음에 API 호출 횟수 기준으로 계산했다.

문제: tool_use 체인. Claude Code에 “이 버그 고쳐”라고 하면, 내부적으로 파일 읽기 → 분석 → 수정 → 테스트 → 확인 같은 과정을 거친다. assistant 메시지가 5-10개 연속으로 나온다. 한 질문에 API 호출 10번.

평균 토큰/턴 = 전체 토큰 / API 호출 수. 분모가 뻥튀기된다. 실제 체감보다 훨씬 낮은 수치가 나온다.

수정: 30초 gap threshold로 턴을 구분한다. 이 내용은 이전 포스트에서 상세히 다뤘다.

세션 기간 뷰 추가

5 키로 “현재 세션” 뷰를 볼 수 있다. 5시간 rolling window 기준.

Claude Code의 5시간 세션 쿼터와 맞춘 거다. “지금 이 세션에서 얼마나 썼는지”를 보고 싶을 때. day/week/month는 달력 기준이라, 지금 당장의 쿼터 상황을 보려면 세션 뷰가 필요했다.

날짜 경계 버그

day/week/month 뷰에서 날짜 필터를 UTC로 계산하고 있었다. 한국 시간(UTC+9) 새벽 3시에 쓰면 UTC 기준으로는 전날이다. “오늘” 뷰에서 방금 쓴 게 안 보이는 상황.

로컬 타임존 기준으로 바꿨다. datetime.now().date()로 오늘 날짜를 잡고, 레코드의 타임스탬프도 .astimezone().strftime("%Y-%m-%d")로 로컬 변환해서 비교한다.

session 뷰는 날짜가 아니라 시간 기반(지금부터 5시간 전)이니까 이 버그에 영향을 안 받았다.

수동 리프레시에 force_oauth

r 키를 누르면 수동 리프레시가 된다. 이때 force_oauth=True를 넘긴다.

자동 리프레시(1초 간격)는 force_oauth=False다. OAuth API를 매 초 호출하면 안 되니까. 하지만 사용자가 명시적으로 r을 누르면 “지금 당장 최신 쿼터 정보를 보고 싶다”는 의도다. OAuth도 강제 재호출한다.

1
2

def action_refresh(self) -> None:
    self.refresh_data(force_oauth=True)

OAuth 모듈에서는 force_refetch()로 _last_fetch를 0으로 초기화한다. 다음 get_oauth_usage 호출 시 _should_refetch가 True를 반환해서 API를 즉시 호출.

Claude Code 협업 정리

이 프로젝트에서 Claude Code 협업은 전반적으로 효율적이었다.

잘된 것

Windows os.kill 버그 진단. 증상만 알려줬다. “윈도우에서 대시보드 켜면 다른 터미널의 Claude Code가 꺼진다.” 이것만으로 os.kill(pid, 0)이 Windows에서 CTRL_C_EVENT를 보낸다는 걸 바로 찾아냈다. 내가 Windows signal 동작을 몰랐으면 한참 삽질했을 거다.

/simplify가 실제 버그 발견. GetExitCodeProcess 반환값 미체크는 프로덕션에서 간헐적으로 터질 수 있는 버그였다. 단순 리팩토링이 아니라 실제 결함을 잡은 거다.

성능 분석 → 구현 분리. 내가 분석 결과와 개선 플랜을 정리해서 넘기면, Claude Code가 그대로 구현했다. 계획과 실행을 분리할 수 있었다. “이 테이블 보고 이렇게 고쳐”라고 하면 된다.

아쉬운 것

/simplify 결과를 영어로 줬다. “한글로 말해”라고 했더니 “앞으로 한국어로 하겠습니다”라고 답함. 아니, 지금 방금 말한 거를 한글로 다시 말하라는 거다. “네 알겠습니다” 하고 넘어가버렸다. 한 번 더 말해야 알아들었다. 두 번 말해야 하는 건 좀 답답하다.

큰 단위 지시와 작은 단위 지시의 차이. 스캐폴드나 위젯 전체 구현 같은 큰 단위는 잘 맞았다. “데이터 모델 전체 만들어”라고 하면 적절한 구조로 만들어준다. 반면 최적화 같은 세밀한 작업은 구체적으로 지시할수록 결과가 좋았다. “1초 리프레시로 바꾸고, 캐시 trim하고, 새 레코드 없으면 스킵해”라고 구체적으로 말해야 했다. 당연한 얘기지만, 맥락이 복잡할수록 지시도 구체적이어야 한다.
