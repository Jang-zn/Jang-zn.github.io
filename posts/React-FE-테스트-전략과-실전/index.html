FE 테스트는 BE 테스트랑 다르다

BE API 테스트는 구조가 단순하다. 입력값을 넣고 반환값을 확인한다. 숫자 넣고 계산 결과 맞추는 것.

FE 테스트의 “출력값”은 DOM이다.

2

사용자 행동 → [컴포넌트] → DOM 변화
  ↑                            ↑
시뮬레이션                   검증

“이 상황에서 화면에 뭐가 보이냐”를 검증한다. 버튼을 눌렀더니 뭐가 뜨는지. 에러가 나면 어떤 메시지가 표시되는지. 접근 방법이 다르니까 도구도 다르고, 쿼리하는 방식도 다르다.

테스트 피라미드(Testing Pyramid)

정의: 테스트를 세 계층으로 나누고, 하위로 갈수록 많이, 상위로 갈수록 적게 작성하는 전략.

계층도구속도검증 대상
단위 테스트Vitest, Jest빠름순수 함수, 커스텀 훅
컴포넌트 테스트React Testing Library보통렌더링, 이벤트, 상태 변화
E2E 테스트Playwright, Cypress느림전체 유저 플로우

피라미드 이유: 위로 올라갈수록 실행 비용이 높다. E2E는 실제 브라우저를 띄워야 하고, 외부 서비스 의존성이 생기고, 실패 원인을 찾기 어렵다. 투자 대비 효과는 컴포넌트 테스트가 제일 높다.

실무 비중: 컴포넌트 테스트 > 단위 테스트 > E2E. E2E는 결제, 로그인처럼 절대 망가지면 안 되는 핵심 경로에만.

단위 테스트 — 순수 함수와 커스텀 훅

순수 함수

입출력이 명확한 함수는 단위 테스트가 제일 간단하다. React 없이 돌아가면 된다.

// formatAmount.test.ts
describe('formatAmount', () => {
  test('수량과 단위 모두 있으면 조합', () => {
    expect(formatAmount(3, 'kg')).toBe('3kg');
  });
  test('수량 없고 단위만 있으면 단위만', () => {
    expect(formatAmount(null, '개')).toBe('개');
  });
  test('둘 다 없으면 빈 문자열', () => {
    expect(formatAmount(null, null)).toBe('');
  });
});

엣지 케이스(null, undefined, 빈 값)를 빠짐없이 넣는다. 이런 함수는 나중에 어디서 쓰일지 모른다.

renderHook — 커스텀 훅 테스트

정의: React 훅을 컴포넌트 없이 테스트할 수 있게 감싸주는 유틸리티. @testing-library/react 제공.

훅은 컴포넌트 안에서만 쓸 수 있다는 React 규칙이 있다. 그냥 함수처럼 호출하면 에러. renderHook이 최소한의 렌더링 환경을 만들어준다.

test('increment하면 count 1 증가', () => {
  const { result } = renderHook(() => useCounter(0));

  act(() => {
    result.current.increment();
  });

  expect(result.current.count).toBe(1);
});

act() — 상태 업데이트 동기화

정의: React의 상태 업데이트(setState)를 테스트 환경에서 동기적으로 완료시키는 래퍼.

setState는 비동기적으로 처리된다. act() 없이 상태를 변경하면 React가 아직 업데이트를 완료하지 않은 상태에서 검증이 돌아갈 수 있다. 경고가 뜨거나, 테스트가 불안정해진다.

규칙: 훅의 상태를 변경하는 함수를 호출할 때는 항상 act()로 감싼다.

test('startSession 후 session이 생성됨', () => {

beforeEach에서 초기화하지 않으면 이전 테스트의 상태가 남아서 다음 테스트에 영향을 준다. 스토어 테스트의 필수 패턴.

컴포넌트 테스트 — React Testing Library

RTL의 철학: 사용자 관점으로 쿼리하라

정의: React Testing Library(RTL)는 사용자가 실제로 보고 상호작용하는 방식을 기준으로 컴포넌트를 테스트하는 라이브러리.

핵심 철학이 있다: “구현 세부사항을 테스트하지 말고, 사용자가 경험하는 것을 테스트하라.”

1
2
3
4

// ❌ 클래스명으로 쿼리 — 리팩토링하면 테스트도 깨짐
container.querySelector('.ingredient-card__check-btn')

// ✅ 사용자가 보는 방식으로 쿼리
screen.getByRole('button', { name: /패스/ })
screen.getByText('고춧가루')
screen.getByPlaceholderText('재료 검색')

클래스명이나 id로 쿼리하면 내부 구현이 바뀔 때마다 테스트가 깨진다. 텍스트나 ARIA 역할로 쿼리하면 UI가 재구성돼도 같은 기능이면 테스트가 통과한다.

쿼리 메서드 3종

메서드없을 때용도
getBy*에러 발생반드시 있어야 하는 요소
queryBy*null 반환“없어야 함”을 검증할 때
findBy*대기 후 에러비동기로 나타나는 요소

findBy*는 비동기다. 기본 1초까지 대기하면서 요소를 찾는다. await가 필요하다.

// 있어야 함 (동기)
expect(screen.getByText('링귀니 까르보나라')).toBeInTheDocument();

// 없어야 함
expect(screen.queryByText('오류')).not.toBeInTheDocument();

// 비동기로 나타남
expect(await screen.findByText('링귀니 까르보나라')).toBeInTheDocument();

  const onSetAction = vi.fn(); // mock 함수

  render(
    <CookingIngredientList
      ingredients={[{ id: 'ing-1', name: '마늘', amount: 3, unit: '개' }]}
      onSetAction={onSetAction}
      {...나머지props}
    />
  );

  fireEvent.click(screen.getByRole('button', { name: /패스/ }));

  expect(onSetAction).toHaveBeenCalledWith('ing-1', 'SKIPPED');
});

vi.fn()으로 mock 함수를 만들어서 “호출됐는지”, “어떤 인자로 호출됐는지”를 검증한다.

test('레시피 목록 로딩 → 데이터 표시', async () => {
  server.use(
    http.get('/api/v1/recipes', () =>
      HttpResponse.json({ data: [{ id: 'r1', name: '링귀니 까르보나라' }] })
    )
  );

  renderWithProviders(<HomePage />);

  expect(screen.getByText('로딩 중...')).toBeInTheDocument(); // 로딩 상태
  expect(await screen.findByText('링귀니 까르보나라')).toBeInTheDocument(); // 완료 상태
});

test('API 에러 시 에러 메시지 표시', async () => {
  server.use(
    http.get('/api/v1/recipes', () =>
      HttpResponse.json({}, { status: 500 })
    )
  );

  renderWithProviders(<HomePage />);
  expect(await screen.findByText('불러올 수 없어요')).toBeInTheDocument();
});

React Query를 쓰는 컴포넌트는 QueryClientProvider로 감싸야 한다. 테스트용 QueryClient는 retry: false로 설정한다. 실패 시 재시도하면 테스트가 느려지고 의도가 흐려진다.

MSW 재사용의 가치

MSW로 BE 없이 FE 선행개발에서 이미 핸들러를 만들어놨다. 개발용으로 작성한 MSW 핸들러가 테스트에서도 그대로 쓰인다.

1
2

개발 환경: MSW 핸들러 → 실제 API처럼 동작
테스트 환경: 같은 MSW 핸들러 → 테스트에서 재사용

API 모킹 코드를 두 벌 관리하지 않아도 된다. 핸들러가 변경되면 개발/테스트 양쪽이 동시에 업데이트된다.

test('요리 시작 → 스텝 체크 → 완료', async ({ page }) => {
  await page.goto('/recipe/recipe-1');
  await page.click('button:has-text("요리 시작")');

  await expect(page).toHaveURL(/cooking/);

  await page.click('[data-testid="step-0-check"]');
  await expect(page.getByRole('progressbar')).toHaveAttribute('aria-valuenow', '1');

  await page.click('button:has-text("요리 완료")');
  await expect(page.getByText('수고하셨어요')).toBeVisible();
});

E2E가 필요한 경우:

  • 결제 플로우
  • 로그인/인증
  • 여러 페이지를 넘나드는 핵심 시나리오

E2E 없어도 되는 경우:

  • 단순 UI 변화 → RTL로 충분
  • 함수 로직 → 단위 테스트로 충분

E2E는 한 번 돌리는 데 오래 걸린다. 커버리지는 낮아도 핵심 경로에만 쓰는 게 맞다.

무엇을 테스트하고 무엇을 안 하나

가치가 높은 것:

  • 조건 분기가 많은 렌더링 (로딩/에러/빈 상태/데이터 4가지가 다 다르게 보임)
  • 복잡한 인터랙션 시퀀스 (수량변경 클릭 → 입력창 → 값 입력 → 확인 → 표시 변경)
  • 비즈니스 로직이 있는 훅 (hasModifications 같은 판단 로직)
  • 엣지 케이스 (null, 빈 배열, 긴 문자열)

안 해도 되는 것:

  • 라이브러리 동작 검증 (React Query가 캐싱을 잘 하는지는 React Query가 책임진다)
  • 순수 스타일 (color가 #C1440E인지)
  • 구현 세부사항 (useState 내부 변수 이름)
  • 스냅샷 테스트 (UI 조금 바꿀 때마다 전부 깨짐, 유지비용 높음)
