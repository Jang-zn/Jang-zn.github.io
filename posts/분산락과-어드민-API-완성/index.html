[앱인토스-통합 Api] 분산락과 어드민 API 완성

By 코드대장간
Posted Mar 18 Updated Mar 20 10 min read

오늘 커밋 28개

어제 어드민 API 모듈 만들고 기획서 1,200줄 작성한 게 밑작업이었다. 오늘은 그걸 실제로 구현하는 날. 분산락 인프라부터 어드민 API 10개 도메인 CRUD, 모니터링 세팅까지 전부 들어갔다.

분산락 인프라 — Port/Adapter 패턴

Redis(Redisson) 기반 분산락을 Port/Adapter 패턴으로 구현했다. 학습 포스트에 정리한 내용을 그대로 적용.

1
2
3

// 포트 인터페이스
boolean tryLock(String key, long waitTime, long leaseTime, TimeUnit unit);
void unlock(String key);

테스트는 Fake 구현체로 돌린다. 통합 테스트에서는 진짜 Redis로 락 획득/해제/타임아웃 3건 검증.

tryLock 에러 처리 — 경합과 인프라 에러는 다르다

처음 구현할 때 tryLock에서 Redis 에러가 나면 false를 반환하게 했다. 이게 문제다. 화장실 문이 잠긴 건지 화장실 자체가 고장 난 건지 구분이 안 된다. 락 경합으로 실패한 건지 Redis가 죽은 건지 구분이 안 된다. Redis 에러면 LockAcquisitionException을 던지도록 바꿨고, ParseWorker에서 이 예외를 잡으면 job을 재큐잉해서 유실을 방지한다. 락 경합(false 반환)은 waitTime 동안 재시도하다가 실패하면 해당 요청을 거절하는 정상 흐름이고, 인프라 에러는 잡을 버리면 안 되는 비정상 상황이다. 이 둘을 구분해야 한다.

ParseJob 동시성 보호 — 3단계

멱등성 학습 포스트에서 정리한 TOCTOU 문제가 여기서 터졌다.

1단계: 잡 생성 중복 방지

같은 유저가 같은 URL을 동시에 제출하면 잡이 2개 생긴다. 분산락으로 잡 생성 구간을 직렬화.

2단계: 완료 URL 재사용

이미 파싱 완료된 URL이면 캐시(TTL 7일)나 DB에서 결과를 가져와서 새 레시피로 복제한다. 여기서 버그가 하나 있었는데, 다른 유저의 recipeId를 그대로 저장하고 있었다. 남의 노트를 원본째 가져간 거다. 복사해서 내 노트로 만들어야 한다. 다른 유저의 레시피를 직접 참조하면 안 된다. forkRecipe로 복제해서 별도 레시피를 만들도록 수정.

3단계: Worker 이중 처리 방지

Worker가 잡을 꺼내서 처리하는 동안 같은 잡이 다시 큐에서 나오면(stuck job 복구 등) 이중 처리된다. 분산락(lease=15분)으로 잡 단위 직렬화, PENDING 상태 검증 추가.

추가: URL-scoped 분산 락

3단계까지 적용하고 나서 한 가지 더 발견했다. ai-parse 캐시를 제거하니까(어차피 히트가 안 되는 캐시였다) in-flight 구간에서 동일 URL에 대한 Gemini 호출이 중복 발생하는 문제. parseRecipe가 .block()으로 워커 스레드를 점유하기 때문에 중복 호출은 처리량 저하로 직결된다. 기존 락이 userId 스코프라 다른 유저가 같은 URL을 넣으면 별도 job으로 처리됐고, stuck job 복구 시에도 동일 job이 재실행될 수 있었다.

parse-url:{urlHash}로 URL 스코프 분산 락을 걸어서 AI 호출~레시피 생성 구간을 직렬화했다. 락 획득 후 parse-result 캐시를 재확인해서, 이미 완료된 URL이면 fork로 바로 빠진다. Gemini API 호출이 동일 URL당 최대 1회로 수렴.

RecipeService 캐시 스탬피드 방지

캐시 스탬피드 학습 포스트에서 정리한 Double-Check Locking 패턴 적용. getRecipe()에 분산락(wait=2초, lease=5초) + 이중 캐시 조회. 락 실패하면 DB 직접 조회로 graceful degradation.

약관 도메인 — 클린 아키텍처 레이어별 구현

약관(TermsPolicy) 기능을 레이어별로 쌓았다.

  1. 도메인: TermsPolicy 엔티티, TermsPolicyType 열거형, TermsPolicyRepository 포트
  2. 인프라: JPA Repository 구현
  3. 어드민 API: CRUD + 활성화/비활성화 (동일 app+type 약관 자동 비활성화)
  4. 클라이언트 API: 인증 불필요한 조회 + Flyway V4로 초기 데이터 시딩

여기서도 버그 하나. 약관 폴백 조회가 appName 필터 없이 돌고 있었다. 다른 앱의 약관이 노출될 수 있는 문제. appName='ALL'로 한정해서 수정. 조회 방식도 List.get(0) 대신 findFirst...OrderByEffectiveAtDesc로 바꿔서 결과를 확정적으로 만들었다.

어드민 API Step 2 — 10개 도메인 CRUD

기획서 스펙대로 어드민 API 전체를 구현했다. 10개 컨트롤러, 10개 서비스, 30개 DTO. Payment 도메인도 이때 같이 추가 — Payment, DailySettlement, ProcessedPaymentEvent 엔티티 3개와 PaymentStatus 상태 머신, V5 마이그레이션까지.

모든 쓰기 작업에 AuditLog 기록. @PreAuthorize로 역할별 권한 검사. 일괄 신고 처리는 @Size(max=1000)으로 DoS 방어.

Client API 7개 추가

어드민만 아니라 클라이언트(TasteNote FE)용 API도 같이 만들었다. 배너/공지 조회, 신고 접수, 앱 설정/버전 조회, 점검 상태, 결제 웹훅. MaintenanceCheckFilter로 점검 중이면 503 반환.

어드민 API Step 3 — 모니터링

Prometheus + Grafana 세팅.

  • admin-api(8081) 스크래핑 타겟 추가
  • Grafana datasource/dashboard 프로비저닝을 Docker volume으로 자동화
  • TasteNote Overview 대시보드: HTTP 요청량, JVM 메모리, 비즈니스 메트릭 11개 패널
  • 커스텀 Micrometer 메트릭: 로그인 성공/실패 counter, 결제취소 counter, 점검모드 gauge
  • HTTP percentile histogram 활성화

SystemStatusResponse에 grafanaUrl 필드도 추가해서 어드민 FE에서 바로 Grafana로 이동 가능.

Claude Code 협업

오늘은 양이 많아서 Claude한테 Step 단위로 지시했다. “Step 2: 이 기획서대로 API 전체 구현해줘” → “Step 3: 모니터링 세팅해줘” 이런 식.

분산락 쪽은 내가 직접 설계하고 Claude가 구현했다. ai-parse 캐시 제거 같은 건 캐시 흐름을 분석하다가 “이거 히트가 안 되는 캐시인데?”라고 내가 발견한 거다. Claude가 만든 코드를 내가 분석하면서 불필요한 걸 쳐내는 과정. 이게 결국 코드 리뷰다.

약관 폴백 조회 버그도 마찬가지. Claude가 초기 구현을 만들고, 내가 “appName 필터 안 거는데?” 하고 짚었다. 생성 속도는 빠른데, 보안이나 데이터 격리 같은 세밀한 부분은 사람이 잡아야 한다.
