## "이어서 할까요?"는 뭘 이어서 하는 건데

[TasteNote 요리모드](/posts/요리모드-v2-구현과-UX-개선/)에서 `CookingResumeSheet`를 개선했다. 기존 메시지: "진행 중인 요리가 있어요. 이어서 할까요?" 버튼은 "이어서" / "새로 시작".

A 레시피를 하다 B 레시피로 진입하면, 어떤 레시피에 대한 질문인지 알 수 없다. 이어서 할 게 A인지 B인지 모른다.

## 좋은 시스템 메시지의 3요소

**1. 상황**: 지금 무슨 일이 일어났나

> "진행 중인 요리가 있어요"

**2. 맥락**: 무엇에 대한 이야기인가

> "**된장찌개**를 하다 멈췄어요"

**3. 행동**: 각 선택지가 어떤 결과를 만드나

> "된장찌개 이어서 하기 / 김치찌개 새로 시작"

셋 중 **맥락**이 가장 자주 빠진다. "이어서 할까요?"는 상황+행동은 있지만 맥락이 없다.

## 버튼 레이블 원칙

### "수정" → "수량변경"

"수정"은 범위가 넓다. 이름도 수정, 순서도 수정, 수량도 수정. 사용자가 버튼을 눌렀을 때 무슨 일이 일어날지 예측하기 어렵다.

"수량변경"은 명확하다. 재료의 수량만 바꾼다는 걸 안다.

좋은 버튼 레이블 체크리스트:
- **동사 포함**: "수정"보다 "수량 변경하기"
- **행동 대상 명확**: "삭제"보다 "이 재료 제외"
- **가역성 암시**: "삭제"(되돌리기 어려움) vs "숨기기"(다시 볼 수 있음)

## 역방향 설계

"뭘 보여줄까?" → "그러면 뭘 저장해야 해?"

요리 재개 시트에 레시피명을 표시하려면, 세션 시작 시점에 레시피명을 함께 저장해야 한다. 나중에 "레시피명을 보여주고 싶은데 저장이 안 돼 있다"가 되면 스토어를 뜯어야 한다.

UI 요구사항에서 역으로 데이터 모델을 설계하는 습관. 화면 기획이 완성되기 전에 어떤 데이터가 필요한지 먼저 파악하면 뒤늦은 수정을 줄일 수 있다.

```typescript
// cookingSessionStore에 추가
recipeName: string | null;  // 재개 시트에 표시하기 위해

// 세션 시작 시
startSession({ recipeId, versionId, recipeName });
```

## 프로젝트 적용

```
Before: "진행 중인 요리가 있어요. 이어서 할까요?"
        [이어서] [새로 시작]

After:  "된장찌개를 하다 멈췄어요."
        "된장찌개 이어서 하기" / "김치찌개 새로 시작"
```

버튼 레이블에 레시피명을 넣었다. 사용자가 선택하기 전에 결과를 예측할 수 있다. 영화관에서 "A관 또는 B관?" 대신 "인셉션(A관) 또는 오펜하이머(B관)?"으로 물어보는 거다.
