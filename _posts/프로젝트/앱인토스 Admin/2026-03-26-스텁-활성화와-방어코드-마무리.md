## 어제 작업의 연장선

## 스텁 페이지 16개 활성화

App.tsx 라우팅에 연결된 페이지 16개가 "준비 중입니다" 스텁이었다. 서브디렉토리에 실제 구현체가 이미 있었는데, 라우팅이 스텁 파일로 연결돼 있었다.

1줄 re-export로 교체.

```typescript
export { default } from './banner/BannerListPage';
```

파일 16개, 272줄 삭제, 16줄 추가. App.tsx 변경 없음. 자세한 원리는 [Re-export와 코드 스플리팅 학습 포스트](/posts/Re-export와-React-Lazy-코드-스플리팅/).

활성화된 페이지들:

| 섹션 | 페이지 |
|------|--------|
| operations | Banner(List/Form), Notice(Form), Report(List/Detail), Maintenance |
| app-config | FeatureFlag, AppVersion, Terms |
| apps/recipe | Featured |
| audit | AuditLog |
| system | SystemStatus, AdminUser, DataExport |

## RevenueStatsPage 크래시 수정

`stats.byMethod.map()`에서 `byMethod`가 undefined일 때 TypeError. optional chaining 2곳 추가.

작은 수정이지만, 결제 통계 페이지가 열릴 때마다 터지면 어드민이 못 쓰는 거다.

## 에러 시 Stale State 전면 처리

API 에러가 나도 이전 성공 데이터가 화면에 남아있는 문제. hooks 7개의 catch 블록에 상태 리셋 추가.

```typescript
catch (e) {
  setUsers([]);   // 이전 데이터 초기화
  setError(e);
}
```

pages 7개에 optional chaining 보강. `SystemStatusPage`의 redis/database/jvm 섹션 조건부 렌더링.

방어적 프로그래밍 패턴은 [학습 포스트](/posts/방어적-프로그래밍과-에러-핸들링/)에 정리.
