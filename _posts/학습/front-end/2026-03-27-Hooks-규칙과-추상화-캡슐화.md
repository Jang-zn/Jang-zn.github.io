## Hooks의 두 가지 철칙

React Hooks에는 반드시 지켜야 하는 규칙이 두 개 있다.

1. **최상위에서만 호출** — 조건문, 반복문, 중첩 함수 안에서 호출하지 않는다
2. **React 함수 내에서만 호출** — 일반 함수에서는 사용할 수 없다

왜 이런 규칙이 있는가. React는 Hook을 **이름이 아니라 순서로 기억**한다.

출석부를 생각하면 된다. 선생님이 "1번 김철수, 2번 이영희, 3번 박민수"로 기억하고 있다. 어느 날 2번 이영희가 안 오면? 3번 박민수가 2번 자리로 밀린다. 선생님은 "2번이 이영희인 줄 알았는데 박민수네?"라고 혼란에 빠진다. React의 Hook 관리가 정확히 이거다.

```
[렌더링 1]              [렌더링 2]
Hook #1: useState       Hook #1: useState       ← 같음
Hook #2: useEffect      Hook #2: useEffect      ← 같음
Hook #3: useCoachMark   (조건부로 생략됨!)       ← 밀림
                        Hook #3: useStore       ← 원래 #4
```

## 조건부 Hook 사용 불가 이유

React 내부에서 Hook은 배열의 인덱스로 관리된다. 첫 번째 렌더에서 `useState`가 인덱스 0, `useEffect`가 인덱스 1. 다음 렌더에서도 같은 순서로 호출되어야 인덱스 0의 state가 올바른 `useState`에 매칭된다.

if문 안에 Hook을 넣으면 조건에 따라 인덱스가 밀린다. React는 "인덱스 2번 Hook"이 뭔지 이름으로 확인하지 않는다. 그냥 2번째로 호출된 Hook이 2번이다.

```typescript
// 이러면 안 된다
if (isAdmin) {
  useAdminPanel(); // 조건부 호출 — 인덱스가 밀림
}
```

ESLint의 `react-hooks/rules-of-hooks` 규칙이 이걸 잡아준다. 다만 이 규칙은 정적 분석이라 빌드타임 상수인지 런타임 변수인지 구분 못 한다. `IS_DEBUG_ENV` 같은 빌드타임 상수로 분기해도 경고를 낸다.

## enabled 패턴 — Hook은 호출하되 동작만 멈추기

그러면 "조건에 따라 Hook의 동작을 켜고 끄고 싶을 때"는 어떻게 하는가.

Hook 자체는 항상 호출하고, 내부에서 `enabled` 플래그로 동작을 제어한다.

```typescript
// Hook은 항상 호출, enabled로 동작 제어
useCoachMark(config, sheetOpen);
```

내부는 이렇게 된다.

```typescript
function useCoachMark(config, enabled = true) {
  const store = useCoachMarkStore((s) => s.activeTutorial);
  useEffect(() => {
    if (!enabled) return; // 동작만 멈춤
    // ... Observer 설정
  }, [config.id, enabled]);
}
```

Hook 호출 순서는 매 렌더에서 동일하다. Rules of Hooks를 위반하지 않는다. `enabled`가 false면 Effect 내부에서 early return할 뿐이다.

이 패턴은 React Query에서도 쓴다.

```typescript
const { data } = useQuery({
  queryKey: ['search', keyword],
  queryFn: () => searchIngredients(keyword),
  enabled: keyword.length >= 1, // keyword 없으면 fetch 안 함
});
```

### 조건부 로직 처리 방법 비교

| 패턴 | Hook 호출 | 구독 생성 | 사이드이펙트 | 적합한 경우 |
|------|:-:|:-:|:-:|------|
| 내부 컴포넌트 분리 | 조건부 | 조건부 | 조건부 | 컴포넌트 전체 ON/OFF |
| enabled 파라미터 | 항상 | 항상 | 조건부 | 부분 기능 ON/OFF |

내부 컴포넌트 분리는 외부에서 조건부 렌더링하고, 내부 컴포넌트에서 Hook을 호출하는 방식이다.

```typescript
function DebugFAB() {
  if (!IS_DEBUG_ENV) return null;
  return <DebugFABInner />; // 이 안에서 Hook 사용
}
```

이렇게 하면 `DebugFABInner`는 렌더될 때만 Hook이 호출되므로 Rules of Hooks를 지킨다. ESLint도 만족한다.

## 추상화 누수(Abstraction Leak)

여기서부터는 Hook 규칙과 다른 이야기다. 코드 설계 문제.

추상화 누수란, 모듈 A가 모듈 B의 **내부 구현을 알아야** 정상 동작하는 상태다.

자동차 운전대를 돌리면 바퀴가 돌아간다. 운전자는 기어비나 조향 각도를 몰라도 된다. 이게 좋은 추상화다. 그런데 "운전대를 37도 돌리면 바퀴가 15도 움직이니까, 바퀴를 30도 돌리려면 운전대를 74도 돌려주세요"라고 하면? 운전대의 추상화가 새는 거다. 내부 구현이 밖으로 흘러나왔다.

## 실전: 탭 전환 매직넘버 노출

TasteNote 코치마크에서 발견한 추상화 누수 사례다.

### Before

```typescript
// useCookingCoachMark.ts
export const COOKING_STEPS_TAB_FIRST = 2; // step 배열의 index
export const COOKING_STEPS_TAB_LAST = 4;
```

```typescript
// CookingPage.tsx — 튜토리얼 내부 구조를 알아야 함
import { COOKING_STEPS_TAB_FIRST, COOKING_STEPS_TAB_LAST } from '...';
import { useCoachMarkStore } from '...';

const step = useCoachMarkStore((s) => /* 현재 step 가져오기 */);
useEffect(() => {
  if (step >= COOKING_STEPS_TAB_FIRST && step <= COOKING_STEPS_TAB_LAST) {
    setActiveTab('steps');
  }
}, [step]);
```

CookingPage가 튜토리얼의 step index를 직접 다루고 있다. "2번 step부터 4번 step이면 조리 순서 탭을 보여줘." 이 "2"와 "4"는 매직넘버(Magic Number)다. step 배열을 열어봐야 뭔 뜻인지 안다.

문제는 step을 추가하거나 순서를 바꾸면 이 숫자도 바뀐다는 거다. 그러면 `useCookingCoachMark.ts`와 `CookingPage.tsx` 둘 다 수정해야 한다.

### After

```typescript
// useCookingCoachMark.ts — 탭 전환까지 내부에서 처리
export function useCookingCoachMark(enabled = true, setActiveTab?) {
  useCoachMark(COOKING_TUTORIAL, enabled);
  // 탭 전환 로직도 여기서 — 상수는 export하지 않음
}
```

```typescript
// CookingPage.tsx — 한 줄
useCookingCoachMark(!resumeSheetOpen, setActiveTab);
```

CookingPage는 step index를 모른다. store에 직접 접근하지 않는다. 매직넘버도 없다. "코치마크 켜줘, 탭 바꿔야 하면 이 함수 써"라고 콜백만 넘긴다. 운전대만 돌리면 되는 상태다.

## Shotgun Surgery — 한 곳 고치면 여러 곳 터지는 안티패턴

Shotgun Surgery(산탄총 수술)는 하나의 변경이 여러 파일에 흩어진 수정을 요구하는 코드 냄새(code smell)다.

Before 상태에서 "조리 순서 탭에 step 1개 추가"를 하면:

1. `useCookingCoachMark.ts` — step 배열에 추가
2. `useCookingCoachMark.ts` — `COOKING_STEPS_TAB_LAST` 수정
3. `CookingStepCards.tsx` — `data-coach` 속성 추가
4. `CookingPage.tsx` — useEffect 범위 확인 (바뀌지 않기를 기도)

After 상태에서 같은 변경:

1. `useCookingCoachMark.ts` — step 배열에 추가 + 상수 수정
2. `CookingStepCards.tsx` — `data-coach` 속성 추가

CookingPage는 건드릴 필요가 없다. 수정 범위가 줄었다.

### 캡슐화 판단 기준

| 구분 | 결합도 | 예시 |
|------|--------|------|
| 높은 결합 | 외부가 내부 index/상수에 의존 | CookingPage가 STEPS_TAB_FIRST 사용 |
| 중간 결합 | 외부가 콜백을 전달 | CookingPage가 setActiveTab 전달 |
| 낮은 결합 | 외부는 enable/disable만 | CookingPage가 enabled만 전달 |

항상 낮은 결합이 좋은 건 아니다. 콜백 전달이 필요하면 중간 결합이 적절하다. 중요한 건 "내부 구현이 바뀌었을 때 외부 파일도 수정해야 하는가?"다. 답이 yes면 캡슐화를 고려한다.

### 언제 캡슐화하고 언제 놔두는가

캡슐화가 필요한 경우:
- 같은 변경이 2개 이상 파일에 영향을 줄 때
- 내부 구현(index, 범위, 상수)이 외부에 노출될 때
- "이거 왜 여기서 이걸 알아야 하지?"라는 의문이 들 때

놔둬도 되는 경우:
- 변경 빈도가 매우 낮을 때
- 한 파일 내에서만 사용되는 상수일 때
- 캡슐화로 오히려 복잡도가 올라갈 때

## TasteNote 적용

TasteNote 코치마크에서 실제로 위 두 가지를 같이 적용했다. Rules of Hooks를 지키면서 `enabled` 패턴으로 코치마크를 조건부 제어하고, 탭 전환 로직은 hook 내부로 캡슐화해서 CookingPage의 의존성을 줄였다. step을 4단계에서 6단계로 늘렸을 때 CookingPage는 수정하지 않았다.

## 정리

| 개념 | 핵심 | 실전 가이드 |
|------|------|------------|
| Rules of Hooks | Hook은 순서로 관리됨 | 조건문/반복문 안에서 호출 금지 |
| enabled 패턴 | Hook은 호출, 동작만 제어 | `if (!enabled) return` in Effect |
| 내부 컴포넌트 분리 | 외부에서 조건부 렌더링 | Hook 없는 Wrapper → Hook 있는 Inner |
| 추상화 누수 | 외부가 내부 구현에 의존 | "왜 이 상수를 export하지?" |
| Magic Number | 의미 없는 하드코딩 숫자 | 명명된 상수로, 가능하면 유도값으로 |
| Shotgun Surgery | 한 변경 → 여러 파일 수정 | 수정 영향 범위를 한 파일로 좁히기 |
| 캡슐화 | 내부 로직을 모듈로 감싸기 | 콜백 전달 > 상수 export |
