⭐5 [학습] MSW로 BE 없이 FE 선행 개발

By 코드대장간
Posted Mar 26 Updated Mar 26 4 min read ⭐5

BE가 안 끝났는데 FE를 만들어야 한다

TasteNote 요리 모드는 BE API(POST /api/v2/cooking-logs)가 아직 구현되기 전에 FE를 먼저 만들었다. MSW(Mock Service Worker)로 API 응답을 흉내 내서 FE를 독립적으로 개발.

MSW란

Service Worker를 사용해서 네트워크 요청을 가로채고 목 응답을 반환하는 라이브러리. 실제 HTTP 요청이 발생하지만 서버에 도달하기 전에 가로채진다. 공항 보안 검색대에서 가방을 열어보고 “통과” 도장을 찍어주는 거다. 비행기(서버)까지 안 가도 검사(응답)가 된다.

MSW 핸들러는 “이 엔드포인트에 이런 요청을 보내면 이런 응답이 온다”를 코드로 기술한다. API 문서를 글로 쓰는 대신 실행 가능한 코드로 작성하는 거다.

API 스펙 합의가 먼저

MSW로 선행 개발하려면 BE-FE가 API 스펙에 먼저 합의해야 한다.

1
2
3
4
5

1. 설계 문서(v2-be.md, v2-fe.md) 작성
2. 엔드포인트, 요청/응답 필드명, 타입 합의
3. FE: MSW 핸들러 작성 + 개발
4. BE: API 구현
5. 통합 테스트

스펙 합의 없이 MSW를 작성하면, BE가 구현한 실제 응답과 FE의 목 응답이 다른 문제가 생긴다. Spring Page 불일치가 정확히 이 케이스다.

설계 문서가 계약

TasteNote v2에서는 v2-fe.md와 v2-be.md를 먼저 작성했다.

1
2
3
4
5

## POST /api/v2/cooking-logs
### Request
{ recipeId, versionId, rating?, memo?, ingredientLogs[] }
### Response
{ id, cookedAt, streak }

이 문서를 기반으로 FE는 MSW 핸들러와 타입을 작성하고, BE는 컨트롤러와 서비스를 구현한다. 문서가 “계약”이다.

MSW의 장점

장점설명
FE 독립 개발BE 완성 전에 UI/로직 구현 가능
네트워크 현실적실제 fetch/axios 호출이 발생
에러 시나리오500, 401, timeout 등 시뮬레이션 가능
Storybook 연동컴포넌트 스토리에서도 사용

fetch mock이나 axios interceptor mock과 다른 점은, MSW는 네트워크 레이어에서 가로챈다는 것. FE 코드에 목 관련 코드가 섞이지 않는다.

  return res(ctx.json({
    id: Date.now(),
    cookedAt: new Date().toISOString(),
    streak: 3,
  }));
});

인앱결제 핸들러 4개도 같은 방식으로 추가. BE API가 완성되면 MSW 핸들러를 끄고 실서버로 연결하면 된다. 핸들러 코드는 나중에 테스트에도 재사용할 수 있다.
