## 문제

같은 요청을 두 번 보내면 결과도 두 번 생긴다. `DELETE /recipes/123`은 두 번 보내도 결과가 같다(이미 삭제됨). 근데 `POST /parse-jobs`는 두 번 보내면 잡이 2개 생긴다.

중복 요청이 생기는 이유는 많다:
- 사용자가 버튼 더블 클릭
- 네트워크 타임아웃으로 클라이언트가 재시도
- 로드밸런서가 다른 서버로 재전송
- [Watchdog가 아직 처리 중인 잡을 재큐잉](/posts/분산-락/)

## Check-then-Act의 함정

"이미 있는지 확인하고, 없으면 생성" — 단순하지만 함정이 있다. 확인(Check)과 생성(Act) 사이에 시간 차이가 있다.

```
Thread A: [DB 조회: 없음] ──────────── [생성] ✅
Thread B:      [DB 조회: 없음] ──────── [생성] ✅ ← 중복
              ↑ A가 생성하기 전에 조회해서 "없다"고 판단
```

확인과 생성 사이의 틈에 다른 요청이 끼어들면 둘 다 "없다"고 판단하고 둘 다 생성한다. 마트에서 우유가 없는 걸 보고 사러 갔는데, 같은 타이밍에 배우자도 "우유 없네" 하고 사러 가는 상황이다. 둘 다 우유를 사온다.

## 적용한 방법

### 분산 락으로 틈 없애기

Check-then-Act 전체를 [분산 락](/posts/분산-락/)으로 감싸서 한 번에 하나의 요청만 처리하게 했다. 같은 URL에 대한 요청은 URL 해시값을 키로 잠금을 건다.

```java
// 잠금 안에서 확인+생성을 한번에
lock(urlHash) {
    if (이미_있으면) throw 중복;
    생성();
}
```

### 3단계 중복 체크

잠금만으로 끝이 아니다. 비용을 줄이기 위해 단계별로 체크한다:

1. **캐시 확인**: 이미 완료된 URL이면 잠금 없이 바로 반환
2. **잠금 획득 + DB 확인**: 진행 중인 잡이 있는지 확인
3. **완료된 잡 재사용**: 다른 사용자가 같은 URL을 이미 파싱했으면 결과 재사용

잠금이 비용이니까, 잠금 전에 캐시로 먼저 걸러내는 게 핵심.

## 작년에 다뤘던 내용

작년 학습 과정에서 동시성 제어를 다루면서 멱등성 개념이 계속 나왔었다. "같은 요청이 여러 번 와도 결과가 같아야 한다" — 멘토링에서도 재시도 로직 설계할 때 이 원칙이 강조됐다. 네트워크 타임아웃으로 클라이언트가 재시도하는 건 흔한 일이고, 서버는 그걸 감당할 수 있어야 한다고.

그때 Check-then-Act의 Race Condition 문제도 배웠다. DB 조회하고 생성하는 사이에 다른 요청이 끼어드는 것. 수업 때는 "아 이론적으로 그렇구나" 했는데, 이번에 URL 파싱에서 정확히 이 문제를 마주쳤다. 같은 URL을 빠르게 두 번 보내면 둘 다 "없다"고 판단하고 Gemini API를 두 번 호출한다. 비용이 이중으로 나간다. 그때 배운 게 이렇게 실전에서 써먹히니까 비로소 내 것이 된 느낌이다.

## 정리

멱등성은 "실수로 두 번 눌러도 괜찮다"를 보장하는 것이다. 사용자 입장에서는 당연한 건데, 서버 입장에서는 의외로 신경 쓸 게 많다.
