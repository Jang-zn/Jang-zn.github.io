[앱인토스 Admin] 스텁 활성화와 방어 코드 마무리

By 코드대장간
Posted Mar 26 3 min read

어제 작업의 연장선

어제 BE-FE API 불일치를 수정했다. 오늘은 남은 정리 작업 3개. PR #4 머지.

스텁 페이지 16개 활성화

App.tsx 라우팅에 연결된 페이지 16개가 “준비 중입니다” 스텁이었다. 서브디렉토리에 실제 구현체가 이미 있었는데, 라우팅이 스텁 파일로 연결돼 있었다.

1줄 re-export로 교체.

1

export { default } from './banner/BannerListPage';

파일 16개, 272줄 삭제, 16줄 추가. App.tsx 변경 없음. 자세한 원리는 Re-export와 코드 스플리팅 학습 포스트.

활성화된 페이지들:

섹션페이지
operationsBanner(List/Form), Notice(Form), Report(List/Detail), Maintenance
app-configFeatureFlag, AppVersion, Terms
apps/recipeFeatured
auditAuditLog
systemSystemStatus, AdminUser, DataExport

RevenueStatsPage 크래시 수정

stats.byMethod.map()에서 byMethod가 undefined일 때 TypeError. optional chaining 2곳 추가.

작은 수정이지만, 결제 통계 페이지가 열릴 때마다 터지면 어드민이 못 쓰는 거다.

에러 시 Stale State 전면 처리

API 에러가 나도 이전 성공 데이터가 화면에 남아있는 문제. hooks 7개의 catch 블록에 상태 리셋 추가.

1
2
3
4

catch (e) {
  setUsers([]);   // 이전 데이터 초기화
  setError(e);
}

pages 7개에 optional chaining 보강. SystemStatusPage의 redis/database/jvm 섹션 조건부 렌더링.

방어적 프로그래밍 패턴은 학습 포스트에 정리.

Claude Code 협업

오늘 작업은 전부 기계적인 수정이라 Claude한테 통째로 맡겼다. “스텁 16개 re-export로 교체”, “hooks 7개 catch 블록 state 리셋”, “pages 7개 optional chaining 보강” — 단순 반복 작업. 빠지는 파일 없이 깔끔하게 처리했다.

오늘의 결과

항목상태
스텁 16개 → re-export 교체완료
RevenueStatsPage byMethod 크래시 수정완료
hooks 7개 catch 리셋완료
pages 7개 optional chaining 보강완료
SystemStatusPage 조건부 렌더링완료
PR #4 머지완료

어드민 기능이 전부 활성화됐다. BE V2 API가 완성되면 어드민 페이지들도 실제 데이터를 표시할 수 있다.
