## 도메인을 샀다

[TasteNote 홈서버 배포](/posts/로컬-개발환경-분리와-도메인/)에서 도메인을 서버에 연결하려니까 DNS 레코드를 알아야 했다. 도메인은 전화번호부에 이름을 등록하는 거다. "김영수"를 치면 "010-1234-5678"이 나오듯, "api.도메인.com"을 치면 서버 IP가 나온다.

## DNS 조회 과정

브라우저가 `api.도메인.com`을 입력하면:

1. DNS Resolver(ISP나 Cloudflare)에 물어본다
2. Root DNS → `.com` TLD DNS → 도메인 권한 DNS 순서로 추적
3. 최종 IP(또는 CNAME)를 반환

한 번 조회하면 캐시된다. TTL(Time To Live) 값에 따라 캐시 유지 시간이 결정. Cloudflare 프록시를 쓰면 TTL은 자동 관리.

## A 레코드

가장 기본. 도메인을 IP 주소에 직접 매핑한다.

```
도메인.com    A    116.33.6.29
```

문제는 가정용 인터넷의 유동 IP. IP가 바뀌면 수동으로 A 레코드를 수정해야 한다. [홈서버](/posts/홈서버-배포-인프라/)에서는 이게 불편하다.

## CNAME 레코드

도메인을 IP가 아니라 다른 도메인으로 연결한다.

```
api.도메인.com    CNAME    터널-uuid.cfargotunnel.com
```

Cloudflare Tunnel이 이 방식이다. 서브도메인을 터널 주소로 CNAME 연결하면, 내 서버 IP가 바뀌어도 터널이 자동 재연결. IP 변경을 신경 쓸 필요 없다.

| 방식 | IP 변경 시 |
|------|-----------|
| A 레코드 | 수동으로 레코드 수정 |
| CNAME + Tunnel | 자동. 터널이 알아서 재연결 |

## 서브도메인과 Nginx

하나의 도메인을 사면 서브도메인은 무제한이다. `api.`, `admin.`, `grafana.` 전부 같은 서버, 같은 Cloudflare Tunnel CNAME을 가리킨다. 그럼 어떻게 구분하는가?

DNS가 "어디로 갈지" 결정하고, Nginx가 "무엇을 보여줄지" 결정한다. 모든 요청이 Nginx 80포트로 들어오면, Nginx가 HTTP Host 헤더를 보고 적절한 서비스로 프록시. 자세한 건 [도메인과 Nginx 리버스 프록시](/posts/도메인-DNS-Nginx-리버스-프록시/) 포스트에.

## TTL

DNS 레코드의 캐시 유지 시간.

| TTL | 의미 | 용도 |
|-----|------|------|
| 300 (5분) | 자주 갱신 | 유동 IP (DDNS) |
| 3600 (1시간) | 보통 | 일반 서비스 |
| 86400 (1일) | 거의 안 바뀜 | 안정적인 서비스 |

Cloudflare 프록시(주황색 구름)를 켜면 Cloudflare가 자동 관리하므로 직접 설정할 필요 없다.

## 나머지 레코드

| 타입 | 역할 | 용도 |
|------|------|------|
| AAAA | 도메인 → IPv6 | A 레코드의 IPv6 버전 |
| MX | 메일 서버 지정 | 이메일 수신 경로 |
| TXT | 텍스트 정보 | 도메인 소유권 인증, SPF |

서버 배포에는 A와 CNAME만 알면 된다.
