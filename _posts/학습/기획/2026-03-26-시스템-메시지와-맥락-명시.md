## "이어서 할까요?"는 뭘 이어서 하는 건지 모른다

시스템 메시지의 가장 흔한 실수. 상황은 설명하는데 맥락이 빠진다.

---

## 시스템 메시지의 3요소

좋은 시스템 메시지는 세 가지가 다 있다.

**1. 상황 (Situation)**: 지금 무슨 일이 일어났나

> "진행 중인 요리가 있어요"

**2. 맥락 (Context)**: 무엇에 대한 이야기인가

> "된장찌개를 하다 멈췄어요"

**3. 행동 (Action)**: 각 선택지가 어떤 결과를 만드나

> "된장찌개 이어서 하기 / 김치찌개 새로 시작"

셋 중 **맥락**이 가장 자주 빠진다. 시스템 입장에서는 "진행 중인 세션"이지만 사용자 입장에서는 "어떤 레시피인지"가 맥락이다.

---

## Reverse Design (역방향 설계)

**정의**: "무엇을 보여줄까" → "그러면 무엇을 저장해야 하나" 순서로 설계하는 방법.

일반적인 설계 순서는 기능 → UI다. 기능을 만들고 나서 어떻게 보여줄지 고민한다. 이러면 UI를 구성할 때 필요한 데이터가 없어서 뒤늦게 스토어나 API를 수정하는 일이 생긴다.

역방향으로 하면:
1. "사용자에게 어떻게 보일 것인가"를 먼저 확정
2. 그 화면에 필요한 데이터 목록 추출
3. 그 데이터를 어디서 어떻게 저장/조회할지 설계

```
UI 확정 → 필요 데이터 목록 → 데이터 모델 설계 → 구현
```

이 순서는 뒤늦은 수정을 줄인다. "나중에 보여주려 했는데 데이터가 없다"가 발생하지 않는다.

## 버튼 레이블 원칙

버튼 텍스트는 사용자가 **누르기 전에** 결과를 예측할 수 있어야 한다.

**나쁜 레이블**: "수정", "확인", "진행", "계속"
**좋은 레이블**: "수량 변경", "이 재료 제외", "된장찌개 이어서 하기"

체크리스트:
- **동사 포함**: 무슨 행동인지 명확히
- **행동 대상 명확**: "삭제"보다 "이 재료 제외"
- **가역성 암시**: "삭제"(되돌리기 어려움) vs "숨기기"(다시 볼 수 있음)
- **범위를 암시**: "수정"은 너무 넓다. 수량만 바뀐다면 "수량 변경"

영화관 비유: "A관 또는 B관?"이 아니라 "인셉션(A관) 또는 오펜하이머(B관)?"

---

## 모달 메시지 작성 가이드

다이얼로그, 바텀 시트, 알림에도 동일한 3요소 원칙이 적용된다.

```
제목: [상황] + [맥락]
본문: (선택사항) 상세 설명
버튼: [행동 결과를 암시하는 레이블]
```

| Before | After |
|--------|-------|
| "이어서 할까요?" | "된장찌개를 하다 멈췄어요" |
| [이어서] [새로 시작] | [된장찌개 이어서 하기] [김치찌개 새로 시작] |
| "수정하시겠습니까?" | "저장하지 않은 변경사항이 있어요" |
| [확인] [취소] | [변경사항 저장] [저장 없이 나가기] |

## TasteNote 적용

`CookingResumeSheet`에서 "이어서 할까요?"가 레시피가 1개일 때는 문제없었다. V2에서 레시피가 여러 개가 되면서 A 레시피 세션이 있는데 B 레시피 상세에서 시트가 뜨는 상황이 생겼다.

Reverse Design 적용: "레시피명을 보여줘야 한다" → "`cookingSessionStore`에 `recipeName` 저장" → 세션 시작 시 `recipeName` 주입.

버튼도 "이어서" / "새로 시작" → "A 이어서 하기" / "B 새로 시작"으로 맥락이 들어갔다.
