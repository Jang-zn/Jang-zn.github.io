## 두 가지

오늘 TasteNote FE에서 한 건 크게 두 가지. Firebase Analytics 버그 수정이랑 약관 기능 구현. PR 2개 머지.

## Firebase Analytics 버그 5건 — PR #8

[어드민 연동 준비](/posts/어드민-연동-준비/)에서 Firebase Analytics 전체 계측을 넣었는데, 실제로 돌려보니 문제가 여러 개 나왔다.

### .env.production이 git에 올라가 있었다

플레이스홀더 값이 들어간 `.env.production`이 번들에 포함되면서 프로덕션 Firebase 초기화가 실패하고 있었다. `git rm --cached`로 추적 해제하고 `.env.production.example` 템플릿으로 대체. `isFirebaseConfigValid()` 함수를 추가해서 플레이스홀더나 누락 값을 런타임에 검증한다.

### api_error 이벤트에 객체가 들어가고 있었다

`error_type` 필드에 `{code, message}` 객체가 통째로 들어가고 있었다. string 타입이어야 하는 필드에. `.code`만 추출하도록 수정.

### URL 검증 실패 이벤트에 전체 URL 전송

사용자가 입력한 URL에 토큰이나 식별자 같은 민감 정보가 포함될 수 있다. 전체 URL 대신 도메인(hostname)만 전송하도록 변경.

### 초기화 전 이벤트 유실

Firebase SDK 초기화가 끝나기 전에 `page_view` 같은 이벤트가 발생하면 그냥 사라졌다. 버퍼에 저장해뒀다가 초기화 완료 시 flush하도록 수정.

### Node 엔진 제약 명시

Firebase SDK가 Node 20+를 요구하는데 명시가 안 되어 있었다. `package.json`에 `engines` 필드 추가.

## 약관 기능 구현 — PR #9

[BE에서 약관 도메인을 만들었으니](/posts/분산락과-어드민-API-완성/) FE도 연동.

### type 기반 조회로 전환

처음에는 약관 목록을 API로 가져와서 `termId`로 상세 조회하는 구조였다. 근데 약관은 종류가 2개(이용약관, 개인정보처리방침)로 고정이다. 메뉴가 2개뿐인 식당에서 매번 메뉴판을 서버에서 받아오는 꼴이다. 목록 API 호출이 불필요하다. 정적 2-item 리스트로 바꾸고, 상세 조회를 `termType` 기반으로 전환.

```
before: GET /terms → 리스트 → termId로 상세 조회
after:  정적 리스트 → termType으로 상세 조회
```

`useTerms` 훅 삭제, `useTermByType` 신규 작성. mock 핸들러도 API 가이드 스펙에 맞춰 type enum 값 변경(`SERVICE` → `TERMS_OF_SERVICE`). `Term.type`도 `string`에서 `'TERMS_OF_SERVICE' | 'PRIVACY_POLICY'` 리터럴 유니온으로 강화해서 컴파일 타임에 잘못된 type 값을 잡는다.

### 라우팅 변경 + id 기반 접근 경로 추가

라우트 파라미터를 `termId`에서 `termType`으로 바꿨다. 약관에 인증이 필요 없으니 AuthGuard도 제거.

근데 `termType` 기반으로만 접근하게 바꾸니까 `termId`로 직접 접근하는 경로가 사라졌다. BE나 다른 곳에서 id로 링크할 수 있으니까 `/terms/id/:termId` 경로도 추가. `TermDetailPage`에서 양쪽 파라미터를 분기 처리한다.

### queryKey 충돌

`useTermByType`과 `useTermDetail`이 같은 queryKey 패턴을 쓰고 있었다. 캐시 키가 겹치면 잘못된 데이터가 서빙된다. `['terms', 'byType', type]`과 `['terms', 'byId', termId]`로 분리.

### 런타임 타입 검증

URL 파라미터로 들어오는 `termType`이 유효한 값인지 검증이 없었다. `isTermType()` 가드 함수 추가. 유효하지 않으면 쿼리 비활성화 + 404 UI.

## Claude Code 협업

Firebase 버그 수정은 대부분 내가 발견하고 Claude한테 수정을 시킨 케이스. 특히 `.env.production` 건은 [6일 리뷰](/posts/6일간의-기록-정리와-앞으로/)에서도 언급한 대로 Claude가 git에 올린 걸 내가 리뷰에서 잡은 거다.

약관 쪽은 "type 기반으로 바꿔"라고 방향만 잡아주면 Claude가 훅 교체, 라우팅 변경, mock 수정까지 한번에 처리해줬다. 다만 queryKey 충돌은 Claude가 놓쳤고, id 기반 접근 경로가 사라진 것도 내가 "BE에서 id로 링크할 수 있는데?"라고 짚어서 추가한 거다.
