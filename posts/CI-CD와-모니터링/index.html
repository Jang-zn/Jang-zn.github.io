⭐4 [학습] CI/CD 파이프라인과 모니터링 — GitHub Actions, Prometheus, 인프라 실수

By 코드대장간
Posted Mar 20 Updated Mar 26 5 min read ⭐4

CI/CD가 필요한 시점

앱인토스-통합 Api에서 Docker 환경을 구축하고 나니 배포 자동화가 필요해졌다. 로컬에서 빌드 → 이미지 푸시 → SSH 접속 → docker pull → restart를 매번 수동으로 하면 실수가 난다. 택배를 보낼 때마다 직접 포장하고 차 몰고 가서 전달하는 거다. 컨베이어 벨트를 깔아야 한다.

GitHub Actions + GHCR

GitHub Actions로 빌드하고 GHCR(GitHub Container Registry)에 이미지를 올리는 구조.

왜 GHCR인가

Docker Hub 대신 GHCR을 쓴 이유:

기준Docker HubGHCR
무료 private repo1개무제한
GitHub 통합별도 인증같은 토큰
이미지 가시성별도 UIrepo Packages 탭

개인 프로젝트라 private repo가 필수고, GitHub 안에서 모든 걸 관리하고 싶었다.

브랜치별 환경 분리

1
2

master → latest, {SHA} 태그 → prod 환경 배포
dev    → dev-latest, dev-{SHA} 태그 → dev 환경 배포

같은 서버에서 prod/dev를 분리하되, 배포는 브랜치별로 자동. 배포 Job에서 브랜치를 보고 prod docker-compose만 재시작할지 dev만 재시작할지 결정한다. 교차 배포 방지.

배포 흐름

  1. push → GitHub Actions 트리거
  2. Gradle bootJar (matrix: tastenote-api, admin-api)
  3. Docker build → GHCR push
  4. SSH → docker compose pull → 선택적 restart
  5. docker image prune (미사용 이미지 정리)

Prometheus + Grafana

어제 포스트에서 모니터링 세팅을 했는데, Docker 환경으로 옮기면서 설정을 다시 잡았다.

Prometheus — pull 모델

Prometheus가 주기적으로 대상 서버의 /actuator/prometheus 엔드포인트를 긁어간다. 서버가 메트릭을 push하는 게 아니라 Prometheus가 pull한다. 환자가 “저 아파요”라고 연락하는 게 아니라 의사가 정기적으로 회진을 도는 방식이다.

1
2
3
4
5
6
7

scrape_configs:
  - job_name: tastenote-api-prod
    static_configs:
      - targets: ['tastenote-api:8080']
  - job_name: tastenote-api-dev
    static_configs:
      - targets: ['dev-tastenote-api:8180']

Docker 네트워크 안에서 서비스명으로 접근한다. prod/dev job을 분리해서 Grafana에서 환경별 필터링이 가능하다.

Grafana 변수

대시보드에 environment 변수를 만들어서 prod/dev를 드롭다운으로 전환할 수 있게 했다. PromQL에서 job=~"$environment" 패턴으로 필터링.

인프라 코드리뷰에서 잡은 실수 3가지

Claude가 만든 인프라 설정을 리뷰하다가 3개를 잡았다.

1. MySQL healthcheck 인증 누락

1
2
3

# 잘못된 설정
healthcheck:
  test: ["CMD", "mysqladmin", "ping"]

인증 정보 없이 ping만 하면 MySQL이 실제로 준비되지 않았어도 healthy로 뜬다. 문 두드려서 “살아있어요?” 하는 건데, 정작 안에서 짐 정리도 안 끝난 상태인 거다. -u root -p$$MYSQL_ROOT_PASSWORD를 추가하고, CMD를 CMD-SHELL로 바꿔야 환경변수 치환이 된다. YAML에서 $는 $$로 이스케이프해야 docker-compose가 안 먹는다.

2. Nginx server_name 와일드카드

1
2

# 잘못된 설정
server_name api.*;

Nginx에서 api.*는 유효하지 않다. 와일드카드는 *.example.com처럼 맨 앞에만 쓸 수 있다. nginx.conf.template + envsubst로 api.${DOMAIN} 형태로 바꿨다.

3. GHCR 로그인 하드코딩

1
2

# 잘못된 설정
username: ${{ github.actor }}

github.actor는 PR을 올린 사람이다. fork에서 PR이 오면 다른 사용자의 이름으로 GHCR에 로그인을 시도한다. secrets.GHCR_USERNAME으로 교체.

정리

CI/CD와 모니터링은 “있으면 좋은 것”이 아니라 “없으면 배포할 때마다 불안한 것”이다. 특히 인프라 설정은 한 글자 실수로 전체가 안 돌아가는데, 그걸 로컬에서는 발견하기 어렵다. 코드리뷰가 중요한 영역이다.
