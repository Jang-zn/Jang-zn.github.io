## 디바운스의 실패

처음 스케줄링은 디바운스였다. 파일이 변경되면 타이머를 걸고, 타이머가 만료되면 처리. 중간에 또 변경되면 타이머 리셋.

30초 디바운스로 시작했다. 바로 문제가 드러났다.

Claude Code 대화는 한 번의 응답에도 JSONL을 여러 번 수정한다. tool_use 한 번에 한 줄 추가. 파일 읽기 한 번에 한 줄. thinking 한 번에 한 줄. 대화가 진행되는 동안 디바운스가 계속 리셋된다. 30초 안에 다음 수정이 안 일어나는 순간이 거의 없다.

600초(10분)로 늘렸다. 이번엔 다른 문제.

짧은 대화는 3분이면 끝난다. 근데 디바운스 타이머는 마지막 수정 후 10분을 기다린다. 3분짜리 대화의 노트가 13분 뒤에 생긴다. 사용자가 "방금 한 거 정리해줘"라고 기대하는 시점에 아직 안 돼있다.

반대로 긴 대화도 문제다. Claude Code Teams에서 에이전트를 돌리면 30분 넘게 걸린다. 10분 디바운스는 중간에 터진다. 10분 동안 수정이 없는 구간이 생기거든. 대화가 안 끝났는데 미완성 세션을 요약한다.

디바운스 시간을 올리는 건 근본 해결이 아니다. 짧은 대화에는 너무 느리고, 긴 대화에는 여전히 부족하다. 골디락스 값이 없다.

## 고정 슬롯 전환

문제의 본질을 다시 봤다. "대화가 끝나는 시점을 모른다." JSONL 파일 수정이 멈추는 시점이 대화 종료인지, 사용자가 잠깐 생각 중인지, 에이전트가 긴 작업을 돌리는 중인지 구분할 방법이 없다.

감지를 포기했다. "끝나는 시점을 알겠다"는 전제를 버렸다.

대신 주기적으로 폴링한다. 매 시간 `:00`, `:20`, `:40`. 20분마다 "그동안 dirty에 쌓인 파일 있으면 처리"하는 방식이다.

버스 시간표다. 승객이 언제 오든 다음 정류장 시간에 태운다. 2분 전에 오든 19분 전에 오든 다음 버스를 타면 된다. 승객이 "제가 도착했으니 바로 출발해주세요"라고 해도 버스는 시간표대로 간다.

```typescript
function msUntilNextSlot(): number {
  const now = new Date();
  const minutes = now.getMinutes();
  let targetMinute: number;
  if (minutes < 20) targetMinute = 20;
  else if (minutes < 40) targetMinute = 40;
  else targetMinute = 60;
  return (targetMinute - minutes) * 60_000
    - now.getSeconds() * 1000 - now.getMilliseconds();
}
```

스케줄러는 재귀적으로 다음 슬롯을 예약한다. `setTimeout`으로 다음 슬롯까지 남은 밀리초만큼 대기. 처리가 끝나면 다시 다음 슬롯을 계산해서 예약.

이렇게 바꾸니 짧은 대화든 긴 대화든 상관없어졌다. 최대 20분 기다리면 처리된다. 대화가 아직 진행 중이어도 괜찮다. 증분 처리가 있으니까.

## 새 세션 감지

새 JSONL 파일이 생기면 새 대화가 시작된 거다. 이건 사용자가 이전 대화를 끝냈을 가능성이 높다는 신호다.

```typescript
private onFileAdd(newFilePath: string): void {
  const projectDir = path.dirname(newFilePath);
  const siblings = findJsonlFiles(projectDir)
    .filter(f => f !== newFilePath && !isSubagentFile(f));
  for (const sibling of siblings) {
    this.dirtyFiles.add(sibling);
  }
  this.dirtyFiles.add(newFilePath);
}
```

새 파일이 생기면 같은 프로젝트 디렉토리의 기존 파일들을 전부 dirty에 추가한다. 다음 슬롯에서 처리된다. 이전 대화가 완전히 끝났을 테니 최종 상태로 요약할 수 있다.

새 파일 자체도 dirty에 추가한다. 아직 대화가 진행 중일 수 있지만, 다음 슬롯에서 증분 처리가 알아서 판단한다.

## 증분 처리와 processedMessageCount

20분 슬롯에서 대화가 절반만 진행됐을 수 있다. 그 상태로 처리된다. 다음 슬롯에서 나머지가 추가됐을 수 있다. 전체를 다시 처리하면 중복이다.

`processedMessageCount`를 state.json에 기록한다. "이 파일에서 메시지 47개까지 처리했음." 다음에 파싱하면 메시지가 72개다. 48번째부터 72번째까지만 추출해서 Gemini에 넘긴다.

```typescript
const alreadyProcessed = this.state.getProcessedMessageCount(filePath);
const newMessages = session.messages.slice(alreadyProcessed);
```

이전에 만들어진 노트 내용도 context로 넘긴다. Gemini가 "이전에 이런 작업을 했고, 이번에 이어서 이런 작업을 했다"는 맥락을 파악할 수 있게. 안 그러면 매번 독립적인 노트가 생긴다. 이전 대화와의 연결이 끊긴다.

`loadPreviousContext()` 함수가 이전 노트 파일들을 읽어서 요약 섹션을 추출한다. Full Conversation 섹션은 잘라낸다. 요약만 넘기면 충분하고, 전체 대화를 넘기면 토큰 낭비다.

## incomplete 세션과 pendingContext

Gemini가 대화를 분석해보니 "아직 진행 중"이라고 판단하는 경우가 있다. Claude가 "분석하겠습니다"만 말하고 실제 결과가 없으면 incomplete다. 모든 주제가 incomplete이면 `allIncomplete` 플래그가 선다.

이때 노트를 만들지 않는다. 대신 `pendingContext`에 대화 텍스트를 저장한다.

```typescript
if (allIncomplete) {
  await this.state.markPending(filePath, stat.mtimeMs, conversationText);
  return;  // 노트 생성 없이 종료
}
```

다음 처리에서 같은 프로젝트의 다른 세션이 처리될 때, `getPendingSiblingContext()`로 pending된 대화를 가져온다. Gemini에 이전 맥락으로 넘겨서 합쳐서 요약한다.

여기서 문제가 하나 있다. `getPendingSiblingContext()`는 `entryPath !== filePath` 조건으로 **같은 파일의 pendingContext를 제외**한다. 형제 세션의 context만 가져오겠다는 의도다.

근데 같은 파일이 다음 슬롯에서 재처리되면? 새 메시지가 추가돼서 dirty에 올라왔다. 이번엔 메시지가 충분해서 incomplete가 아니다. 근데 이전에 저장한 pendingContext는 이 파일 자체의 것이라 sibling context에서 빠진다. 이전 맥락이 소실된다.

아직 고치지 않은 버그다. 발생 빈도가 낮아서 우선순위에서 밀렸다. 같은 파일에서 "첫 번째 처리에서 incomplete → 두 번째 처리에서 complete"가 되는 경우에만 발생한다. 대부분의 대화는 한 번에 충분한 내용이 있다.

## 서브에이전트 필터링

Claude Code Teams를 쓰기 시작하면서 새로운 노이즈가 생겼다.

메인 세션의 JSONL에는 "에이전트를 호출했다 → 결과를 받았다" 정도만 남는다. 실제 서브에이전트의 대화는 `subagents/` 폴더에 별도 JSONL로 생긴다.

서브에이전트 JSONL을 처리하면 안 된다. 메인 세션에 요약이 이미 있고, 서브에이전트 내부 대화는 "파일 3개 읽었다" 수준의 짧은 교환이다. 노트로 만들 가치가 없다.

두 곳에서 필터링한다.

```typescript
// chokidar 설정
this.watcher = chokidar.watch(globPattern, {
  ignored: '**/subagents/**',
  // ...
});

// 파일 검색
findJsonlFiles(dir).filter(f => !isSubagentFile(f));
```

chokidar에서 `**/subagents/**` 패턴으로 감시 대상에서 제외한다. `findJsonlFiles()`에서도 `isSubagentFile()` 체크로 걸러낸다. 이중 필터다.

Gemini 프롬프트에도 맥락을 설명했다. "팀, team, teammate는 Claude Code Teams의 서브에이전트를 의미한다. 사람 팀원이 아니다." 이걸 안 넣으면 Gemini가 "팀 리드에게 보고함", "팀원과 협의" 같은 요약을 만든다. 서브에이전트한테 보고하는 건데.

user 메시지에서 `<teammate-message`로 시작하는 XML 블록도 필터링한다. 이건 팀 에이전트 간 내부 통신이다. 사용자가 직접 입력한 게 아니다.

## 남은 버그들

완벽하지 않다. 알고 있는 버그가 세 개 있다.

**1. Team agent 이후 최종 summary drop**
Claude Code Teams 에이전트가 끝나면 마무리 메시지가 짧다. "작업 완료했습니다" 정도. 이전 처리에서 서브에이전트 호출 전까지 이미 처리됐고, 마무리 부분의 text-bearing 메시지가 `minMessages`(기본값 3) 미달이면 노트가 생성되지 않는다. 서브에이전트가 한 실제 작업의 최종 정리가 빠질 수 있다.

**2. partial 세션 타임스탬프**
증분 처리에서 새 메시지만 추출할 때 `firstTimestamp`를 새 메시지 중 첫 번째 text 메시지가 아니라 첫 번째 JSONL 레코드의 타임스탬프로 쓴다. system이나 progress 레코드가 먼저 나오면 실제 대화 시작 시각과 다르다. 노트 파일명에 영향을 준다.

**3. pendingContext 소실**
위에서 설명한 문제. 같은 파일이 재처리될 때 자기 자신의 pendingContext를 sibling context로 가져오지 못한다.

세 개 다 edge case다. 정상적인 사용 패턴에서는 거의 안 걸린다. 근데 알면서 안 고치는 건 찝찝하다. 나중에 고칠 거다. 아마.

## 설계 판단은 사람 몫

디바운스에서 고정 슬롯으로 전환한 건 내 결정이다.

Claude Code에게 "디바운스가 잘 안 되는데 어떻게 하면 좋겠냐"고 물으면, 디바운스 시간을 늘리거나 adaptive 디바운스(파일 크기에 따라 시간 조절)를 제안한다. 기존 접근법 안에서의 개선이다.

"디바운스 자체가 이 문제에 맞지 않는다"는 판단은 내가 했다. "대화 끝나는 시점을 감지하겠다"는 전제가 틀렸다는 걸 인식한 거다. 전제를 바꾸면 해법도 달라진다. 감지를 포기하면 폴링이 자연스럽게 나온다.

Claude Code는 "주어진 프레임 안에서 최선의 해법"을 잘 찾는다. 프레임 자체를 바꾸는 건 사람이 해야 한다.

디바운스 시간을 30초 → 600초 → 1800초로 올리는 건 Claude Code도 할 수 있다. "디바운스를 버리고 고정 슬롯으로 가자"는 결정은 내가 했다. 30초에서 1800초로 올리는 건 같은 종류의 변경이다. 디바운스를 버리는 건 다른 종류의 변경이다.

이 프로젝트에서 Claude Code가 못한 건 딱 두 가지다. 스케줄링 전략 전환과 TOCTOU 한 번 놓친 것. 나머지는 전부 잘했다. 특히 동시성 분석, VS Code API 연동, JSONL 구조 파악. 코드 작성 속도는 혼자 하는 것의 5배는 빠르다.

중요한 건 속도가 아니라 피드백이다. "이거 해봐" → "안 되는데" → "다시 해봐"가 빨라야 한다. Claude Code와 일하면 이 루프가 분 단위로 돈다. 혼자 디버깅하면 시간 단위다.
