⭐6 [학습] 캐시 스탬피드 — 캐시 만료되면 DB가 죽는다

By 코드대장간
Posted Mar 18 Updated Mar 26 6 min read ⭐6

문제 상황

인기 레시피 페이지가 있다고 치자. 캐시 TTL이 10분이고 초당 100개 요청이 들어온다. 캐시가 살아있을 때는 DB 조회 0회. 근데 캐시가 만료되는 그 순간, 100개 요청이 전부 DB를 때린다.

캐시 만료 순간:
Client 1 → [캐시 없음] → DB 조회
Client 2 → [캐시 없음] → DB 조회
...
Client 100 → [캐시 없음] → DB 조회
→ DB 과부하

100개 중 1개만 DB 조회하면 되는데 전부가 간다. 식당 문이 열리자마자 100명이 동시에 주방에 주문하러 가는 거다. 1명만 가서 주문하고 나머지는 자리에서 기다리면 되는데. 이걸 캐시 스탬피드라고 한다.

적용한 해결법: 잠금

분산 락을 여기서도 썼다. 캐시 미스가 나면 잠금을 건다. 잠금을 잡은 1개 요청만 DB를 조회하고 캐시를 채운다. 나머지는 잠금이 풀릴 때까지 기다렸다가 채워진 캐시를 읽는다.

1
2
3

Client 1 → [캐시 없음] → 잠금 획득 → DB 조회 → 캐시 저장 → 응답
Client 2 → [캐시 없음] → 잠금 대기... → 캐시 있음 → 응답
Client 3 → [캐시 없음] → 잠금 대기... → 캐시 있음 → 응답

DB 조회가 100번에서 1번으로 줄어든다.

다른 방법도 있다

  • 확률적 사전 갱신: 만료 전에 랜덤하게 미리 캐시를 갱신. 잠금 없이 동작하지만 100% 보장은 안 됨
  • 논리적 만료: 실제 TTL은 길게 잡고 데이터 안에 만료 시각을 넣어두는 방식. 만료돼도 일단 옛날 데이터를 주고 백그라운드에서 갱신

프로젝트에서는 잠금 방식을 택했다. 이미 분산 락 인프라가 있으니까 추가 비용 없이 적용할 수 있었다.

잠금 실패하면?

잠금을 못 잡아도 서비스가 죽으면 안 된다. 잠금 실패 시에는 그냥 DB를 직접 조회하게 했다. 캐시 보호가 안 되는 거지 서비스가 멈추는 건 아니다.

boolean locked = distributedLockPort.tryLock(lockKey, 2, 5, TimeUnit.SECONDS);
if (!locked) {
    // 잠금 실패해도 DB 직접 조회로 폴백
    return recipeRepository.findById(recipeId);
}

작년 멘토링에서 들었던 것들

작년 학습 과정에서 캐시를 집중적으로 다뤘었다. 멘토링에서 들은 내용 중 기억나는 것들:

  • “TTL 만료를 일괄로 하지 말고 랜덤값을 줘야 한다” — 모든 캐시가 동시에 만료되면 스탬피드가 터진다. TTL에 약간의 랜덤 편차를 줘서 만료 시점을 분산시키라는 얘기
  • “캐시의 적용/수정은 항상 metric 기반으로 수치 보고 판단하자” — 감으로 TTL 정하지 말고 모니터링 데이터를 보라는 것
  • “캐시는 빨리 지우면 빨리 지울수록 용량 확보가 되어서 좋다” — 필요 이상으로 오래 잡고 있을 이유가 없다
  • “락 경합이 엄청나게 길게 잡히면 뭔가 문제다” — 스탬피드 방지용 잠금이 오래 걸린다면 설계를 다시 봐야 한다

그때는 과제 치느라 정리만 해두고 넘겼는데, 이번에 실제로 적용하면서 하나씩 떠올랐다. Cache-Aside, Write-Through 패턴도 그때 배운 건데, 지금 프로젝트에서 레시피 조회(Cache-Aside)랑 URL 파싱 결과(Write-Through 비슷하게)를 나누는 판단 근거가 됐다.

특히 TTL 랜덤화는 이번 프로젝트에서는 아직 적용 안 했는데, 사용자가 늘면 고려해야 할 부분이다. 지금은 트래픽이 적어서 괜찮지만, 멘토링에서 들은 토스 사례(하루 600만 접속, 피크 시 초당 200건)를 생각하면 나중에는 필요해질 수 있다.

정리

캐시는 “있으면 좋고 없으면 좀 느린” 정도여야 한다. 캐시가 없다고 서비스가 터지면 그건 캐시가 아니라 의존성이다.
