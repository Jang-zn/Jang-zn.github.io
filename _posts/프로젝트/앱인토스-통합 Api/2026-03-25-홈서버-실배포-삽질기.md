## CI/CD 만들어놓고 직접 배포는 처음

[3/20에 Docker 환경과 CI/CD를 구축](/posts/홈서버-배포와-CI-CD-구축/)했다. 근데 그건 설정만 만든 거고, 실제 서버에 올려본 적은 없었다. 오늘 홈서버에 처음 배포했다. 삽질 3연속.

## 서버 세팅

Ubuntu 홈서버에 SSH 키 설정, SSHD 보안 강화, UFW 방화벽, Fail2Ban 설치. Docker도 설치. 도메인은 `code-forge.co.kr`로 구매해서 Cloudflare에 연결. Cloudflare Tunnel로 서버를 외부에 노출.

Tunnel을 쓰면 서버의 포트를 직접 열 필요가 없다. 서버가 Cloudflare한테 "나 여기 있어"라고 터널을 파는 거다. 외부에서는 Cloudflare를 통해서만 접근 가능. 자세한 건 [학습 포스트](/posts/Docker-배포-트러블슈팅/)에 정리.

## 삽질 1: Docker Compose V1 호환성

파일을 서버에 scp로 올리고 `docker-compose up -d`를 실행했더니 `ContainerConfig` 에러가 났다. 서버에 설치된 게 docker-compose 1.29.2(V1)였다. V1은 compose 파일의 일부 필드를 지원하지 않는다.

Docker Compose V2 플러그인을 설치해서 해결. V2부터 `docker compose`(하이픈 없음)로 명령어가 바뀐다. `docker-compose`와 `docker compose`는 다른 프로그램이다.

## 삽질 2: ARM/AMD64 아키텍처 불일치

V2로 올렸더니 이번엔 `exec format error`. 컨테이너가 시작하자마자 죽는다.

원인은 Mac에서 빌드한 이미지가 ARM 아키텍처인데, 서버는 AMD64(x86_64). ARM으로 구운 빵을 AMD64 토스터에 넣은 거다. 들어가지도 않는다.

`docker-compose.yml`에 `--platform linux/amd64`를 추가해서 빌드 시 아키텍처를 명시. CI/CD에서 빌드할 때도 동일하게 설정해야 한다.

## 삽질 3: Nginx 환경변수

컨테이너가 다 떴는데 도메인으로 접근이 안 됐다. Nginx 설정을 확인해보니 `server_name`이 `localhost`로 박혀 있었다. `.env`에 `DOMAIN=code-forge.co.kr`을 설정했는데 반영이 안 된 거다.

`docker compose restart`는 기존 컨테이너를 재시작만 한다. 환경변수를 다시 읽지 않는다. 컴퓨터를 절전 모드에서 깨우는 것과 재부팅하는 건 다르다. `docker compose up --force-recreate`로 컨테이너를 새로 만들어야 환경변수가 반영된다.

추가로 `server_name`도 수정했다. 원래 `api.${DOMAIN}`이었는데 `tastenote-api.${DOMAIN}`으로, `admin.${DOMAIN}`은 `ait-admin-api.${DOMAIN}`으로 변경. 나중에 다른 서비스가 추가될 수 있으니 서비스명을 명시적으로 넣었다.

## 배포 확인

```bash
curl -H "Host: tastenote-api.code-forge.co.kr" http://localhost/api/v1/maintenance/status
```

200 응답. 서버가 살아있다.

## Claude Code 협업

오늘 배포 삽질은 대부분 Claude Code SSH 세션에서 진행했다. 에러 메시지를 보여주면 원인 분석과 해결 방법을 제시해줬다. ARM/AMD64 불일치는 Claude가 에러 로그 보자마자 짚었다. `--force-recreate` 필요성도.

다만 서버 보안 설정(SSH 키, UFW, Fail2Ban)은 Claude가 제안한 걸 그대로 쓰지 않고 각 항목을 직접 확인했다. 보안 설정은 한 줄 빠져도 문이 열린 채로 남는다.
