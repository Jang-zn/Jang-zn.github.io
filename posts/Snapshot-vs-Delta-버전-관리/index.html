⭐6 [학습] Snapshot vs Delta — 데이터 버전 관리 전략

By 코드대장간
Posted Mar 25 Updated Mar 26 4 min read ⭐6

레시피에 “버전”이 있다

TasteNote는 레시피 버전 관리 앱이다. 같은 요리를 만들 때마다 재료나 단계를 바꿔서 새 버전을 만든다. 이 버전 데이터를 어떻게 저장할 것인가.

두 가지 선택지가 있다: Snapshot과 Delta.

Snapshot 방식

매 버전마다 전체 데이터를 복사해서 저장한다.

1
2

v1: {재료: [소금 1g, 설탕 2g], 단계: [볶기, 끓이기]}
v2: {재료: [소금 1g, 설탕 3g], 단계: [볶기, 끓이기, 식히기]}

v2는 v1의 전체 복사본 + 변경분이다. v2만 읽으면 전체 레시피를 알 수 있다. 사진첩에 매번 전체 사진을 넣는 거다.

Delta 방식

변경된 부분만 저장한다.

1
2

v1: {재료: [소금 1g, 설탕 2g], 단계: [볶기, 끓이기]}
v2: {변경: 설탕 2g→3g, 추가: 식히기}

v2를 보려면 v1을 먼저 읽고 v2의 변경분을 적용해야 한다. Git의 diff와 같다. 저장 공간은 적게 들지만 조회가 복잡하다.

비교

 SnapshotDelta
저장 공간많음 (매번 전체 복사)적음 (변경분만)
조회 속도빠름 (한번에 읽기)느림 (이전 버전부터 재구성)
구현 복잡도낮음높음 (diff/merge 로직 필요)
버전간 비교두 스냅샷 비교어차피 diff가 있음
적합한 경우소규모 데이터, 빈번한 조회대규모 데이터, 저장 효율 중요

왜 Snapshot을 선택했나

레시피 하나의 데이터가 크지 않다. 재료 10~20개, 단계 5~10개. 전체 복사해도 1KB 미만이다. 사용자가 만 명이고 평균 5개 레시피에 3개 버전이면 15만 스냅샷. 각 1KB면 150MB. 현대 DB에서 이건 아무것도 아니다.

Delta로 가면 레시피 상세 조회 시 매번 버전 체인을 따라가며 재구성해야 한다. 조회가 압도적으로 많은 서비스에서 조회 성능을 희생하면서까지 저장 공간을 아낄 이유가 없다.

실제 성능 고려 사항

Snapshot을 선택하면 저장 전략보다 조회 전략이 중요해진다.

문제설명해결
N+1 쿼리레시피 목록 조회 시 각각의 재료/단계를 별도 쿼리fetch join 또는 batch 조회
인덱스 누락recipe_id + version 복합 인덱스 없으면 전체 스캔복합 인덱스 추가
delete+recreate버전 생성 시 기존 재료/단계 삭제 후 재삽입변경 감지보다 비용이 적은 경우만

delete+recreate가 왜 문제인가. 재료 20개를 수정할 때, JPA의 변경 감지(dirty checking)로 각각 UPDATE를 날리면 20개 쿼리. 전부 DELETE 후 INSERT하면 2개 쿼리(bulk delete + bulk insert). 상황에 따라 delete+recreate가 더 효율적이다.

규모별 우선순위

사용자 규모우선 최적화
~1,000명인덱스만 잘 걸면 됨
~10,000명N+1 해결, 쿼리 최적화
~100,000명캐시 레이어 도입
100,000명~읽기 전용 replica, CQRS 검토

TasteNote는 아직 첫 번째 구간이다. 과최적화보다는 정확한 동작이 우선.
