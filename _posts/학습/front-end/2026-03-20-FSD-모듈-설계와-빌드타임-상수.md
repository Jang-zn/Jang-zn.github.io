## FSD — Feature-Sliced Design

[TasteNote](/posts/피드백-기능과-메인-컬러-변경/)에서 피드백 기능을 추가하면서 FSD(Feature-Sliced Design) 구조를 다시 정리했다. 코드를 기술 역할(components, hooks, utils)이 아니라 비즈니스 기능(features) 단위로 묶는 아키텍처다. 옷장 정리에 비유하면, "상의/하의/속옷" 칸으로 나누는 게 기술 역할별 구조고, "출근복/운동복/잠옷" 세트로 나누는 게 FSD다. 출근복을 빼려면 한 칸만 비우면 된다.

### 레이어 규칙

```
app → pages → features → entities → shared/ui
```

상위 레이어는 하위 레이어만 참조할 수 있다. 단방향. features끼리는 직접 참조하지 않는다. 합성은 pages에서 한다.

### feature 내부 구조

```
features/feedback/
├── api/         ← 서버 통신 (useSubmitFeedback)
├── lib/         ← 유틸리티 (collectDeviceInfo, feedbackCategories)
├── ui/          ← 컴포넌트 (FeedbackPage)
└── index.ts     ← barrel export
```

[코치마크](/posts/코치마크-튜토리얼-구현/)도 같은 구조다. `features/coach-mark/`에 model, store, lib, ui, hooks가 있다. 기능 단위로 폴더가 자급자족한다. 기능을 삭제할 때 폴더 하나만 지우면 된다.

### 어디에 놓을지 판단

- features: 재사용 가능하거나 비즈니스 로직이 있는 코드
- pages: 해당 페이지에서만 쓰는 코드
- shared: 기능과 무관한 공용 유틸, 컴포넌트, 상수

피드백의 `collectDeviceInfo`는 features에 뒀다. 디바이스 정보 수집은 피드백 기능의 일부다. `createFlag` 같은 localStorage 팩토리는 shared에 있다. 기능과 무관한 공용 유틸이니까.

## Vite define — 빌드타임 상수 주입

피드백에서 앱 버전을 수집해야 한다. `package.json`의 version을 런타임에 읽을 수는 없다(번들에 포함 안 됨). Vite의 `define` 옵션으로 빌드 타임에 주입한다.

```typescript
// vite.config.ts
define: {
  __APP_VERSION__: JSON.stringify(pkg.version),
}
```

C의 `#define`과 같은 원리다. 인쇄소에서 원고의 `{{버전}}`을 `"1.2.3"`으로 찍어서 내보내는 거다. 빌드 시 코드에서 `__APP_VERSION__`이 `"1.2.3"` 문자열로 치환된다. `JSON.stringify()`를 빼먹으면 따옴표 없이 치환되어 `1.2.3`이 변수명으로 해석된다.

TypeScript 타입 선언도 필요하다.

```typescript
// vite-env.d.ts
declare const __APP_VERSION__: string;
```

### import.meta.env와의 차이

| | define | import.meta.env |
|---|---|---|
| 용도 | 코드 상수 | 환경 설정 |
| 시점 | 빌드 타임 텍스트 치환 | 런타임 객체 접근 |
| 타입 | declare const | ImportMetaEnv |
| 예시 | 앱 버전, 빌드 해시 | API URL, Firebase 키 |

앱 버전은 환경에 따라 바뀌는 게 아니라 빌드마다 고정이다. `define`이 맞다.

## 정리

FSD는 "기능을 추가/삭제할 때 영향 범위가 명확하다"는 게 핵심이다. 피드백 기능을 통째로 빼려면 `features/feedback/` 폴더를 지우고, 라우트 등록만 해제하면 된다. 기술 역할별 구조였다면 components, hooks, utils, api 폴더를 전부 뒤져야 한다.
