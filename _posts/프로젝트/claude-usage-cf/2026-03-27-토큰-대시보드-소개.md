## 왜 만들었나

Claude Code Max 플랜을 쓰고 있다. 월정액 내고 토큰 한도 안에서 쓰는 구조다. 문제는 그 한도가 얼마나 남았는지 볼 방법이 없다는 거다.

`/usage` 명령이 있긴 하다. 순간 스냅샷 하나 보여주고 끝이다. 5시간 윈도우 쿼터가 얼마나 찼는지, 어떤 프로젝트에서 토큰을 많이 먹는지, 모델별 비중은 어떤지 — 다 안 보인다. 그냥 "지금 이만큼 썼다"가 전부.

가스 게이지 없는 차를 모는 기분이다. 연료가 바닥나야 아는 거다. 갑자기 쿼터가 찼다는 메시지가 뜨면 그때서야 "아 많이 썼구나" 하게 된다. 늦다.

직접 만들기로 했다. 터미널에서 항상 띄워놓고 실시간으로 보는 대시보드. `pip install`하고 `claude-usage` 치면 끝.

## 기술 스택 선택

### 왜 Python인가

빠른 프로토타이핑이 목적이다. 이틀 안에 쓸 수 있는 걸 만들어야 했다. Claude Code의 로그 파일이 JSONL 형식인데, Python은 `json.loads` 한 줄이면 파싱이 된다. 파일 I/O도 편하고, 바이트 오프셋 기반 증분 읽기도 `seek`/`tell` 두 줄이면 끝난다.

Java나 Go로 하면 더 빠르게 돌겠지만, 만드는 데 이틀이 아니라 일주일이 걸린다. 이건 유틸이다. 만드는 속도가 실행 속도보다 중요하다.

### 왜 Textual인가

터미널 UI 프레임워크를 골라야 했다. 후보가 몇 개 있었다.

| 프레임워크 | 장점 | 단점 |
|-----------|------|------|
| curses | 표준 라이브러리 | 날것. 위젯 없음. 직접 다 그려야 함 |
| Rich | 예쁜 출력 | 단방향. 인터랙션 없음 |
| Textual | Rich 기반 + CSS 스타일링 + 위젯 시스템 | 배우는 데 좀 걸림 |
| Blessed/Urwid | 오래된 TUI 라이브러리 | 문서 부족. 커뮤니티 작음 |

Textual로 갔다. Rich 위에서 동작하니 출력이 예쁘고, CSS로 레이아웃을 잡는다. 웹 개발자한테 익숙한 방식이다. 위젯 시스템이 있어서 패널을 컴포넌트로 분리할 수 있다. `compose()` 메서드에서 위젯을 yield하면 화면이 구성되는 구조.

pyfiglet은 헤더 ASCII 아트용으로 넣었다. "CLAUDE USAGE"가 큼지막하게 뜨면 터미널에서 존재감이 있다. 장식이지만, 대시보드는 보는 맛이 있어야 한다.

PyYAML은 설정 파일 파싱용. 계정 정보, 모델별 한도, 갱신 주기 같은 설정을 YAML로 관리한다. 별도 설정 없이도 기본값으로 동작하지만, 다계정이나 커스텀 한도가 필요하면 YAML 파일 하나 만들면 된다.

의존성이 이 세 개(Textual, pyfiglet, PyYAML) + 표준 라이브러리뿐이다. 일부러 가볍게 유지했다. 유틸리티 도구가 의존성 지옥에 빠지면 본말전도다.

## 6개 패널 상세

터미널에서 실행하면 대시보드가 뜬다. 6개 패널이 한 화면에 들어간다.

### Token Usage

모델별 토큰 게이지 바. Opus, Sonnet, Haiku 세 모델이 기본으로 표시된다. 각 모델 옆에 수평 게이지 바와 토큰 수치가 나온다.

핵심은 평균 토큰/턴(turn) 수치다. "이 모델 한 번 질문하면 평균 얼마나 쓰는가"를 보여준다. Opus가 턴당 15만 토큰을 먹으면, 남은 쿼터에서 "앞으로 몇 번 더 물어볼 수 있는가"를 가늠할 수 있다.

게이지 바는 상대적 스케일이다. 가장 많이 쓴 모델이 풀 바, 나머지는 비례해서 줄어든다. 절대적 한도 대비 퍼센트가 아니라, 모델 간 비율을 보는 용도.

### Quota Status

5시간 윈도우 쿼터와 7일 쿼터. 퍼센트로 표시된다. "지금 세션에서 73% 썼다", "이번 주 전체 모델 42% 썼다" 식.

리셋 카운트다운도 보여준다. "2h 34m (resets 15:42)" — 언제 쿼터가 풀리는지. 리셋 직전이면 노란색으로 바뀐다. 리셋 시간이 지나면 "리셋 중..."으로 표시되고, API 재호출해서 새 윈도우 정보를 가져온다.

Sonnet 전용 7일 쿼터도 별도로 표시한다. Anthropic이 모델별로 주간 한도를 다르게 잡아놓은 경우가 있어서다.

### Project Usage

프로젝트별 토큰 소비 수평 바 차트. 토큰을 많이 먹은 프로젝트 순으로 정렬된다. "이번 주에 어디서 토큰을 다 썼지?"에 대한 답.

프로젝트 이름은 JSONL 파일 경로에서 추출한다. `~/.claude/projects/-Users-jang-projects-app-be/` 같은 디렉토리명을 파싱해서 `app-be`를 뽑아낸다. `-projects-`나 `-workspace-` 같은 알려진 경로 세그먼트로 자른다.

### Daily Usage

일별 사용량 바 차트. 선택한 기간(day/week/month) 동안의 일별 추이. 어떤 날에 많이 썼는지 한눈에 보인다. 모델별로 색상이 다르다.

week 뷰면 7일치, month 뷰면 30일치 바 차트가 나온다. 토요일에 많이 쓰고 일요일에 적게 쓰는 패턴 같은 게 보인다. 사용 패턴을 파악하면 쿼터 관리 전략이 나온다. "주 초반에 다 쓰면 주 후반에 쿼터가 없다" 같은.

### Cost Estimate

모델별 예상 비용. Anthropic 공개 가격표 기준으로 계산한다.

| 모델 | input (1M) | output (1M) | cache_read (1M) |
|------|-----------|------------|----------------|
| Opus | $15.00 | $75.00 | $1.50 |
| Sonnet | $3.00 | $15.00 | $0.30 |
| Haiku | $1.00 | $5.00 | $0.10 |

input/output 단가가 다르고, 캐시 토큰은 또 다르다. `cache_read` 토큰은 input 단가의 1/10이니까, 캐시를 많이 탈수록 비용이 준다.

캐시 절감액도 보여준다. "캐시 안 썼으면 input 단가로 계산됐을 금액 - 실제 캐시 비용 = 절감액." Max 플랜은 정액제라 실제로 돈이 나가진 않지만, "API로 썼으면 이만큼 나왔을 거다"는 감각을 준다. Opus를 많이 쓰면 이 금액이 꽤 크다. 정액제의 이점을 체감할 수 있다.

### Active Sessions

현재 실행 중인 Claude Code 세션 목록. PID, 프로젝트 경로, 경과 시간, 모델 정보. `~/.claude/sessions/*.json`에서 세션 파일을 읽고, 각 PID가 살아있는지 확인한다.

세션 파일에는 PID, sessionId, cwd(작업 디렉토리), startedAt(시작 시각)이 들어있다. startedAt으로 경과 시간을 계산하고, cwd에서 프로젝트 경로를 보여준다. "지금 몇 개의 Claude Code가 돌고 있고, 각각 어디서 얼마나 오래 쓰고 있는가"를 한눈에.

이 PID 확인이 Windows에서 문제를 일으켰는데, 이건 [세 번째 포스트](/posts/Windows-버그와-최적화/)에서 다룬다.

## 키바인딩과 기간 전환

대시보드 하단에 키 안내가 떠 있다.

| 키 | 동작 |
|----|------|
| `q` | 종료 |
| `r` | 수동 리프레시 (OAuth도 강제 재호출) |
| `a` | 계정 전환 |
| `1` | 오늘 (Day) |
| `5` | 현재 세션 (5시간 rolling window) |
| `7` | 이번 주 (Week) |
| `3` | 이번 달 (Month) |

`5`가 세션인 이유? 5시간 윈도우니까 5. 숫자가 의미에 매핑된다. `1`일, `7`일, `3`0일, `5`시간.

기간을 바꾸면 모든 패널이 해당 기간 기준으로 재집계된다. "오늘 Opus를 얼마나 썼나"에서 "이번 달 전체는 얼마인가"로 뷰가 전환된다. Textual의 `@work(thread=True)` 데코레이터로 백그라운드 스레드에서 집계하고, `call_from_thread`로 UI를 갱신한다. 메인 스레드를 안 건드니까 키 입력이 안 밀린다.

`r` 키는 수동 리프레시다. 자동 갱신은 1초 간격이지만 OAuth API는 30분 주기다. `r`을 누르면 `force_oauth=True`로 OAuth도 즉시 재호출한다. "방금 쿼터가 리셋된 것 같은데 수치가 안 바뀌네" 할 때 쓴다.

## 다계정 지원

YAML 설정 파일에 accounts 배열을 넣는다.

```yaml
accounts:
  - name: "Personal"
    claude_home: "~/.claude"
  - name: "Work"
    claude_home: "~/.claude-work"
```

`a` 키를 누르면 다음 계정으로 전환된다. 계정마다 `claude_home` 경로가 다르고, 각각 별도의 JSONL 로그와 세션 파일을 갖는다.

왜 다계정이 필요한가. 개인 계정과 회사 계정이 다른 경우가 있다. 각각의 쿼터가 별개로 관리된다. 한 대시보드에서 둘 다 보고 싶은 거다.

모델별 주간 한도도 계정별로 설정할 수 있다. 회사 계정이 Team 플랜이면 한도가 다를 수 있으니까.

```yaml
limits:
  - model: "opus"
    weekly_tokens: 45_000_000
  - model: "sonnet"
    weekly_tokens: 45_000_000
```

설정 파일 없이도 동작한다. 기본값은 `~/.claude` 경로에 계정 하나. 설정이 없으면 기본 한도 45M 토큰으로 추정한다.

계정 전환 시 모든 패널이 해당 계정 기준으로 재집계된다. JSONL 로그, 세션 파일, OAuth 인증 — 전부 `claude_home` 경로 기준이니까, 경로만 바꾸면 나머지는 동일한 파이프라인을 탄다.

## 데이터 흐름

전체 데이터 파이프라인은 이렇다.

```
JSONL 파서 → 캐시 리더 → 세션 리더 → Aggregator → OAuth → 위젯
```

1. **JSONL 파서**: `~/.claude/projects/`의 JSONL 파일을 증분 파싱. `UsageRecord` 리스트 반환.
2. **캐시 리더**: Claude Code가 자체적으로 만드는 `stats_cache.json`에서 과거 일별 데이터 읽기. JSONL이 없는 기간의 데이터를 보완.
3. **세션 리더**: `~/.claude/sessions/*.json`에서 활성 세션 정보 읽기. PID 생존 여부 확인.
4. **Aggregator**: 위 세 소스를 합쳐서 기간별 필터링, 모델별 집계, 프로젝트별 집계, 일별 집계. `AggregatedUsage` 모델 하나로 뭉친다.
5. **OAuth**: Anthropic API로 실제 쿼터 사용률 가져오기. 한도 역산.
6. **위젯**: `AggregatedUsage`를 받아서 각 패널이 자기 영역을 그린다.

`AggregatedUsage`가 핵심 모델이다. 7개 위젯이 전부 이 하나의 데이터 객체를 받는다. 위젯은 데이터를 받아서 그리기만 하고, 직접 파싱이나 API 호출을 하지 않는다. 관심사 분리.

앱의 `refresh_data` 메서드가 1초마다 Aggregator를 호출하고, 결과를 `update_widgets`로 넘긴다. 각 위젯의 `update_*` 메서드가 자기 패널을 다시 그린다. 한 위젯이 에러가 나도 다른 위젯은 영향 없다. try/except로 감싸놨다.

위젯 7개가 전부 같은 패턴이다. `update_*` 메서드 하나 받으면 자기 영역을 다시 그린다. 데이터를 스스로 가져오지 않는다. 수동적이다. 뷔페에서 접시만 들고 있는 사람. 음식은 Aggregator가 가져다 놓는다.

Quota Status 패널만 예외가 하나 있다. 리셋 카운트다운을 1초마다 갱신해야 해서, 자체 `set_interval(1.0)`로 re-render를 돌린다. 데이터를 다시 가져오는 게 아니라, 이미 받은 `resets_at` 시각과 현재 시각의 차이를 다시 계산하는 거다.

## 이틀, 21커밋

스캐폴드부터 최적화까지 이틀. 21커밋.

대략 이런 흐름이다.

1. **스캐폴드** — 프로젝트 구조, `pyproject.toml`, Textual 앱 뼈대
2. **데이터 레이어** — 모델 정의, JSONL 파서, 증분 파싱 구현
3. **위젯 전체** — 게이지 바, 차트, 프로젝트 랭킹, 세션 목록, 비용 패널. 한 커밋에 다 넣었다
4. **앱 통합** — 위젯 조합, 키바인딩, 기간 전환. 32개 단위 테스트
5. **OAuth 연동** — API 호출, 한도 역산, 재동기화
6. **버그 수정** — Windows `os.kill`, 평균 토큰 계산, 리셋 직후 0%
7. **최적화** — 1초 리프레시, 캐시 trim, 모듈 레벨 호이스팅

Claude Code와 둘이서 했다. 큰 단위로 지시했다. "데이터 모델 전체 만들어", "위젯 전체 만들어" 식. 이 프로젝트가 Claude Code 협업에 잘 맞았던 이유는, 단독 유틸이라 컨텍스트가 작기 때문이다. 파일 수가 적고, 외부 의존성이 Textual + PyYAML 두 개뿐. Claude Code가 전체 코드를 한 번에 볼 수 있는 규모.

테스트도 32개 만들었다. 파서, Aggregator, 위젯 렌더링 각각 단위 테스트. "이거 테스트 만들어"라고 하면 알아서 엣지 케이스까지 커버한다. 빈 JSONL, 잘못된 JSON, 타임스탬프 없는 레코드, 알 수 없는 모델명 등. 테스트 코드 품질도 괜찮았다. fixture 분리, parametrize 사용, 경계값 테스트.

처음부터 끝까지 이틀. 하루째에 코어 + 위젯까지 동작하는 상태를 만들었고, 둘째 날에 OAuth 연동 + 버그 수정 + 최적화를 했다. 코드 전량을 Claude Code와 같이 썼지만, 설계 결정은 내가 했다. "이런 구조로 만들어"가 아니라 "이런 구조여야 하는데, 구현해"라는 차이.
