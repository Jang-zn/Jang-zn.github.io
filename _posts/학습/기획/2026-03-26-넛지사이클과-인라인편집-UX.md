## 요리하다가 자연스럽게 레시피를 개선하게

TasteNote의 핵심은 "레시피 버전 관리"다. 요리할 때마다 조금씩 바꿔서 나만의 레시피를 완성하는 것. 문제는 [진입점 분석](/posts/진입점-통합과-선택피로/)에서 발견했듯이, "새 버전" 버튼을 직접 누르는 사용자가 드물다는 것.

넛지 사이클로 요리 → 편집 → 저장을 자연스럽게 유도한다.

## 넛지 사이클

```
요리 시작 → 재료 체크 → 인라인 편집 → 요리 완료
  → 변경 없음: "수고했어요" → 끝
  → 변경 있음: "새 버전으로 저장할까요?" → 넛지
```

요리 중에 "소금을 좀 줄여봤다", "단계를 하나 추가했다" 같은 변경이 발생하면, 완료 시 자동으로 "새 버전 저장" 넛지가 뜬다. 사용자가 직접 "새 버전" 버튼을 찾아 누르지 않아도 된다.

Path A(변경 없음)는 최소 단계로 종료. Path B(변경 있음)는 넛지로 새 버전 저장 유도.

## 인라인 편집

기존 요리 모드는 "읽기 전용"이었다. 레시피를 보면서 요리만 한다. 변경하려면 요리 모드를 나가서 수정 페이지로 가야 한다. 이건 안 한다.

인라인 편집은 요리 중에 바로 수정하는 것.

### 재료 액션 사이클

재료를 탭하면 상태가 순환한다.

```
사용함(USED) → 조정함(ADJUSTED) → 건너뜀(SKIPPED) → 사용함
```

한 번의 탭으로 상태를 변경. Fitts's Law — 큰 타겟(56px 체크박스)을 가까이(재료 바로 옆) 배치하면 조작이 빠르다. 요리 중에 정교한 터치를 할 수 없다.

각 상태에 액션 뱃지가 표시된다. ADJUSTED는 수량 수정 입력이 나타나고, SKIPPED는 취소선.

### 단계 인라인 메모

각 단계에 메모 토글 버튼. 누르면 인라인 텍스트 입력이 나타난다. "다음에는 3분 대신 2분만 볶기" 같은 메모. 단계 추가/삭제도 가능.

## 완료 시 분기

`CookingCompleteSheet`가 변경 감지를 한다.

```typescript
const hasModifications = useMemo(() => {
  return Object.values(ingredientActions).some(a => a !== 'USED')
    || stepMemos.some(m => m.length > 0)
    || addedSteps.length > 0
    || deletedSteps.length > 0;
}, [ingredientActions, stepMemos, addedSteps, deletedSteps]);
```

- 변경 없음 → "수고했어요! 요리 완료" 버튼만
- 변경 있음 → "새 버전으로 저장" 버튼이 추가로 표시

"새 버전 저장"을 누르면 [location.state](/posts/location-state와-페이지간-데이터-전달/)로 변경 데이터를 넘기고 NewVersionPage로 이동. 폼이 요리 중 변경사항으로 프리필된다.

## 프리필 전략

NewVersionPage는 두 가지 진입 경로가 있다.

| 진입 | 프리필 | 출처 |
|------|--------|------|
| 요리 모드에서 | 변경된 재료/단계/메모 | location.state |
| 직접 접근 | 현재 버전 복사 | API 조회 |

`fromCooking` 플래그로 분기. 요리 모드에서 왔으면 `cookingEdits`로 폼을 채우고, 아니면 기존 레시피를 복사.

## ReviewSheet 삭제

원래 요리 완료 시 별점 + 메모를 입력하는 `CookingReviewSheet`가 있었다. 인라인 편집을 넣으면서 삭제했다. 요리 중에 이미 메모를 했는데 완료 후 또 메모를 받으면 중복이다. 같은 설문을 두 번 하면 짜증난다.
