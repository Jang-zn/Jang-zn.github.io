⭐3 [학습] Jackson 네이밍 전략 변경 — 조용히 null이 되는 필드들

By 코드대장간
Posted Mar 19 Updated Mar 26 5 min read ⭐3

발단

앱인토스-통합 Api에서 Jackson의 글로벌 네이밍 전략을 SNAKE_CASE에서 기본값(camelCase)으로 바꿨다. 애초에 이 SNAKE_CASE 설정은 내가 한 게 아니다. Claude가 BE 초기 세팅할 때 지맘대로 넣어둔 거다. 나는 서버도 FE도 다 개발하는데, FE는 당연히 camelCase를 쓴다. 서버가 snake_case를 내려주면 프론트엔드 두 곳에서 매번 변환해야 한다. 불필요한 작업이 3겹으로 쌓여있는 거다. camelCase로 통일하면 변환 로직 자체가 필요 없다.

바꾸고 나서 세 가지 문제가 터졌다.

1. DTO 바인딩이 조용히 실패한다

PropertyNamingStrategy를 SNAKE_CASE로 설정하면 Jackson이 자바 필드명을 자동으로 snake_case로 변환한다. mainImgUrl → main_img_url. 양방향이다. 직렬화(응답)와 역직렬화(요청) 둘 다.

이걸 제거하면? 클라이언트가 main_img_url로 보내던 요청이 mainImgUrl 필드에 바인딩되지 않는다. HTTP 200은 정상으로 내려온다. 필드만 조용히 null이 된다. 택배 라벨에 영어로 주소를 적었는데 배달부가 한글만 읽을 줄 아는 상황이다. 배달부는 “모르겠으니 빈 상자로 전달합니다”라고 하고, 수령인은 빈 상자를 받고 나서야 문제를 알게 된다.

1
2
3

// 요청: {"main_img_url": "https://..."}
// 서버 DTO: private String mainImgUrl;
// 결과: mainImgUrl = null  ← 에러 없이 null

테스트에서도 안 잡힌다. mock으로 any()를 쓰면 null이 들어와도 통과한다. 실제 JSON 키로 테스트를 돌려야 잡을 수 있다.

2. ObjectMapper를 직접 만들면 안 되는 이유

처음에 JacksonConfig에서 @Bean @Primary로 ObjectMapper를 직접 생성하고 있었다. Spring Boot의 JacksonAutoConfiguration은 @ConditionalOnMissingBean으로 동작한다. 커스텀 빈이 있으면 자동 설정이 통째로 꺼진다. 회사에서 제공하는 사내 식당이 있는데 도시락을 싸오면 “아 직접 해결하시는군요” 하고 식당 자체를 닫아버리는 거다.

꺼지면 뭐가 빠지냐:

자동 등록 항목상태
JavaTimeModule미등록
Jdk8Module, ParameterNamesModule미등록
application.yml 설정무시
springdoc, jjwt-jackson 커스터마이저미적용

application.yml에 write-dates-as-timestamps: false를 써놔도 안 먹는다. ObjectMapper가 별도로 만들어졌으니까.

해결은 간단하다. JacksonConfig를 삭제하고 application.yml만 쓴다. 커스터마이징이 필요하면 Jackson2ObjectMapperBuilderCustomizer로 한다. ObjectMapper를 직접 만드는 건 자동 설정 전체를 포기하겠다는 선언이다.

3. 네이밍 결정 우선순위

Jackson이 JSON 키 이름을 정하는 순서가 있다.

우선순위설정범위
1@JsonProperty("name")필드 단위
2@JsonNaming(SnakeCaseStrategy)클래스 단위
3PropertyNamingStrategyObjectMapper 전역
4자바 필드명 그대로기본값

@JsonProperty가 붙은 필드는 글로벌 전략이 바뀌어도 영향이 없다. 그래서 토스 API 같은 외부 연동 DTO에 @JsonProperty를 걸어두면 내부 전략 변경에 안전하다.

전역 전략을 바꿀 때 위험한 건 아무 어노테이션도 없는 필드들이다. 이 필드들은 전역 전략에 100% 의존하고 있다가 전략이 바뀌면 일괄로 깨진다. 양방향으로.

마이그레이션에서 한 것

  1. application.yml에서 property-naming-strategy: SNAKE_CASE 제거
  2. JacksonConfig.java 삭제 (Spring Boot 자동설정 복원)
  3. 테스트 JSON 키 전부 camelCase로 변경
  4. 외부 API DTO는 @JsonProperty로 보호 확인
  5. Redis 캐시 키 버저닝으로 역직렬화 충돌 방지

배운 것

Spring Boot 자동설정을 우회하면 편해 보이지만, 그 자동설정이 해주던 일을 전부 직접 관리해야 한다. 모르는 사이에 의존하고 있던 것들이 많다. @ConditionalOnMissingBean은 “대체하겠다”는 뜻이지 “추가하겠다”는 뜻이 아니다.

네이밍 전략 변경은 API Breaking Change다. 테스트가 실제 JSON 키를 쓰지 않으면 잡히지 않는다.
