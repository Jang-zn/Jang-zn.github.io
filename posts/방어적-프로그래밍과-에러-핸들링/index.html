⭐4 [학습] 방어적 프로그래밍 — Optional Chaining과 Stale State

By 코드대장간
Posted Mar 26 Updated Mar 26 4 min read ⭐4

TypeScript 타입이 있어도 런타임에서 터진다

앱인토스 Admin에서 API 에러 시 줄줄이 터지는 문제를 고치면서 정리한 방어적 프로그래밍 패턴들. TypeScript는 컴파일 타임에만 타입을 보장한다. 런타임에서 서버가 뭘 보낼지는 모른다.

Optional Chaining (?.)

1
2
3
4
5

// Bad: content가 undefined면 터짐
data.content.map(item => ...)

// Good: content가 없으면 undefined 반환
data?.content?.map(item => ...) ?? []

체인의 모든 단계에서 null/undefined 체크. 한 곳이라도 빠지면 그 지점에서 TypeError: Cannot read property 'map' of undefined.

!?.length 패턴

1
2
3

// 이 두 가지는 다르다
if (!items?.length)    // undefined → true, [] → true, [1,2] → false ✅
if (items?.length === 0) // undefined → false, [] → true, [1,2] → false ❌

!items?.length는 “비어있거나 없으면”을 한번에 체크한다. items?.length === 0은 undefined일 때 false를 반환해서 “있지만 비어있는” 경우만 잡는다.

Nullish Coalescing (??) vs OR (||)

1
2
3
4
5

0 ?? 'default'   // 0    (null/undefined만 걸러냄)
0 || 'default'   // 'default'  (falsy 전부 걸러냄)

'' ?? 'default'  // ''   (빈 문자열은 통과)
'' || 'default'  // 'default'  (빈 문자열도 걸러냄)

??는 null과 undefined만 체크한다. ||는 0, '', false도 걸러낸다. 결제 금액이 0원일 때 ||를 쓰면 기본값으로 바뀌어버린다.

Stale State 문제

API 에러가 났는데 이전 성공 데이터가 그대로 화면에 남아있다. 사용자는 “정상”으로 오해한다. 은행 앱에서 잔액 조회가 실패했는데 어제 잔액이 보이면 위험하다.

해결: catch에서 상태 리셋

// Before: 에러 시 이전 데이터 유지
try {
  const data = await fetchUsers();
  setUsers(data);
} catch (e) {
  setError(e);
  // users는 이전 값 그대로
}

// After: 에러 시 초기화
try {
  const data = await fetchUsers();
  setUsers(data);
  setError(null);
} catch (e) {
  setUsers([]);    // 상태 리셋
  setError(e);
}

DB의 ROLLBACK과 같다. 트랜잭션이 실패하면 이전 상태로 되돌린다. UI 상태도 마찬가지.

7개 hooks에 적용: useUsers, useReports, usePayments, useBanners, useNotices, useTerms, useAppConfig.

React 18 자동 배칭

위 코드에서 setUsers([]), setError(e) 두 번의 setState가 연속으로 호출된다. React 18부터는 이벤트 핸들러, Promise, setTimeout 안의 연속 setState를 자동으로 배칭한다. 리렌더가 한 번만 발생.

React 17에서는 Promise 안의 setState가 배칭되지 않았다. 두 번의 리렌더가 발생해서 중간에 “데이터는 없는데 에러도 없는” 상태가 순간적으로 보일 수 있었다.

조건부 렌더링

SystemStatusPage에서 redis/database/jvm 정보가 각각 null일 수 있다.

1
2
3

{status?.redis && (
  <RedisSection data={status.redis} />
)}

데이터가 없으면 섹션 자체를 안 그린다. “에러” 표시보다 낫다. 시스템 상태 페이지에서 redis 섹션이 안 보이면 “아 redis 정보가 없구나”로 이해하고, “redis: Error” 보다 덜 당황한다.
