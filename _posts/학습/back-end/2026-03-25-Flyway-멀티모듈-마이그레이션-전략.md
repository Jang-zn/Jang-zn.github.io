## DB 스키마도 버전 관리가 필요하다

[TasteNote BE](/posts/인앱결제-광고-제거-시스템/)에서 purchase 테이블을 추가하면서 Flyway V6 마이그레이션을 작성했다. 멀티모듈 프로젝트에서 마이그레이션 파일을 어디에 두고 어떻게 관리하는지 정리한다.

## Flyway란

DB 스키마 변경을 SQL 파일로 관리하는 도구. Git이 코드 변경 이력을 관리하듯, Flyway는 DB 변경 이력을 관리한다. 앱이 시작할 때 자동으로 미적용 마이그레이션을 실행한다.

## 네이밍 규칙

```
V{버전}__{설명}.sql
```

`V1__init.sql`, `V2__add_recipe_table.sql` 같은 식이다. 버전 번호는 순차 증가. 밑줄 두 개(`__`)가 구분자다. 한 개면 인식 못 한다.

## 멀티모듈에서 마이그레이션 위치

프로젝트 구조가 이렇다:

```
core-domain/       → 엔티티, 포트
infra-persistence/ → JPA, Repository, 마이그레이션
tastenote-api/     → API 서버
admin-api/         → 어드민 서버
```

마이그레이션 파일은 `infra-persistence/src/main/resources/db/migration/`에 둔다. DB 스키마는 persistence 모듈의 관심사다. API 모듈에 두면 tastenote-api와 admin-api가 각각 마이그레이션을 가지게 되는데, 두 API가 같은 DB를 쓰면 충돌한다. 우체국이 하나인데 우편함을 두 개 세우면 편지가 섞인다.

## 절대 지켜야 하는 규칙

### 적용된 마이그레이션은 수정 금지

Flyway는 파일의 체크섬을 기록한다. 이미 적용된 V1을 수정하면 체크섬이 달라져서 앱 시작이 실패한다. 이미 보낸 편지의 내용을 바꿀 수는 없다. 새 편지(V2)를 보내야 한다.

### 관련 변경은 하나의 파일에

purchase 테이블 생성, 인덱스 추가, 초기 데이터 삽입이 한 기능에 속하면 V6 하나에 묶는다. 파일을 쪼개면 V6은 성공했는데 V7에서 실패해서 반쯤 적용된 상태가 된다.

### 롤백은 수동

Flyway Community 버전은 자동 롤백을 지원하지 않는다. 롤백이 필요하면 V7에서 V6의 변경을 되돌리는 SQL을 직접 작성한다.

## MySQL 함수 인덱스

V6 마이그레이션에서 함수 인덱스를 사용했다. MySQL 8.0+에서 지원.

```sql
CREATE INDEX idx_purchase_user_active
ON purchase ((CASE WHEN status = 'ACTIVE' THEN user_id END));
```

일반 인덱스는 컬럼 값 그대로 인덱싱한다. 함수 인덱스는 연산 결과를 인덱싱한다. "이름순 정렬"이 일반 인덱스라면, "성만 따서 정렬"이 함수 인덱스다. ACTIVE 상태인 구매만 빠르게 조회할 수 있다.

## 프로젝트 적용

| 버전 | 내용 |
|------|------|
| V1 | 초기 스키마 (recipe, ingredient, step) |
| V2 | category 테이블 |
| V3 | recipe_image |
| V4 | feedback |
| V5 | pantry |
| V6 | purchase (인앱결제) |

V6이 [인앱결제 시스템](/posts/인앱결제-광고-제거-시스템/)에서 추가한 마이그레이션이다. 테이블 생성 + 인덱스 + 제약조건을 한 파일에 담았다.
