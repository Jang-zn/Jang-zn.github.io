## 왜 필요했나

[BE 프로젝트](/posts/멀티모듈-백엔드-세팅/)에서 레시피 URL 파싱 기능을 만들었는데, 문제가 하나 있었다. 사용자가 같은 URL을 빠르게 두 번 제출하면 **잡이 2개 생긴다**. Gemini API가 두 번 호출되고 비용이 이중으로 나간다.

동시 접근을 막는 방법은 여러 가지가 있다.

**synchronized** — Java에서 제일 간단한 잠금. 메서드나 블록에 붙이면 한 번에 하나의 스레드만 실행된다. 간단하지만 세밀한 제어가 안 된다. 타임아웃도 못 걸고, 잠금을 못 잡으면 무한정 기다린다.

**ReentrantLock** — `synchronized`보다 유연하다. `tryLock()`으로 "못 잡으면 포기"가 가능하고, 타임아웃도 걸 수 있다. 같은 스레드가 여러 번 잡아도 된다(재진입). 단일 서버에서는 이걸로 충분하다.

**Redis 분산 락** — 위 두 개는 같은 JVM 안에서만 효과가 있다. 서버가 여러 대면? 서버 A의 Lock은 서버 B한테 아무 효과가 없다. Redis 같은 외부 저장소에 잠금을 걸어서 서버 간 공유하는 방식이다.

| 방식 | 범위 | 타임아웃 | 자동 해제 | 서버 확장 |
|------|------|---------|----------|----------|
| synchronized | 같은 JVM | 불가 | 불가 | 불가 |
| ReentrantLock | 같은 JVM | 가능 | 불가 | 불가 |
| Redis 분산 락 | 서버 간 | 가능 | 가능 (TTL) | 가능 |

이 외에도 작년 학습 때 다뤘던 락 종류들이 있다. 참고용으로 정리:

- **DB 비관적 락 (Pessimistic Lock)** — `SELECT ... FOR UPDATE`로 행 자체를 잠근다. 다른 트랜잭션이 해당 행을 읽지도 못하게 막는 방식. 확실하지만 대기가 길어질 수 있다
- **DB 낙관적 락 (Optimistic Lock)** — `@Version` 컬럼을 두고, 업데이트할 때 버전이 바뀌었으면 충돌로 판단. 잠금을 안 거니까 빠르지만 충돌 시 재시도 로직이 필요하다
- **DB 제약조건 (Unique Index, @Check)** — 아예 DB 레벨에서 중복이나 무결성을 막는다. `stock >= 0` 같은 조건. 마지막 방어선
- **InMemory 락 (ConcurrentHashMap)** — `putIfAbsent()`로 메모리에서 잠금. 제일 빠르지만 서버 재시작하면 날아간다. 테스트용으로는 충분
- **스핀락 vs Pub/Sub** — 잠금 대기 방식의 차이. 스핀락은 계속 확인하면서 기다리고, Pub/Sub은 풀릴 때 알림을 받는다. 멘토링에서 "경합이 심하면 Pub/Sub이 낫지만, 애초에 경합이 심한 설계가 문제"라는 얘기를 들었다

이번 프로젝트에서는 Redis 분산 락(Redisson)을 선택했다. 솔직히 당장은 서버를 여러 대 띄울 일이 없다. `ReentrantLock`으로도 충분한 상황이다. 근데 작년 학습 과정에서 InMemory 락 → DB 락 → Redis 분산 락으로 단계별 전환을 해본 경험이 있고, 멘토링에서도 분산 락 관련 내용을 꽤 깊게 다뤘었다. 그때는 과제로 후다닥 해치운 거라 제대로 내 것이 안 됐는데, 이번에 실제 프로젝트에 적용해보면서 제대로 경험해두고 싶었다. Redis도 이미 캐시용으로 쓰고 있으니 인프라 추가 비용도 없고.

## 어떻게 동작하는지

Redis에 키를 하나 잡는다. "이 키가 없으면 내가 선점, 있으면 이미 누가 쓰는 중" — 이게 전부다.

Redisson이라는 라이브러리를 썼는데 핵심 파라미터가 두 개다:

- **waitTime**: 잠금 획득을 얼마나 기다릴지. API 요청은 3초 정도 기다려도 되고, Worker는 0초(못 잡으면 바로 포기)
- **leaseTime**: 잠금을 얼마나 유지할지. 프로세스가 죽어도 이 시간이 지나면 자동으로 풀린다

## 적용한 곳

### 1. 같은 URL 동시 파싱 방지

같은 URL로 요청이 동시에 들어오면 URL 해시값으로 잠금을 건다. 먼저 온 요청이 잡을 만들고, 뒤에 온 요청은 잠금 대기하다가 "이미 있네" 하고 돌아간다.

```java
String lockKey = "lock:parse-job:" + urlHash;
boolean locked = distributedLockPort.tryLock(lockKey, 3, 10, TimeUnit.SECONDS);
// 3초 대기, 10초 유지
```

### 2. Worker 이중 처리 방지

Watchdog가 10분 넘은 잡을 "멈춘 것 같으니 다시 돌려"하고 재큐잉하는데, 실제로는 아직 AI 응답을 기다리고 있을 수 있다. 잠금 유지 시간을 Watchdog 기준(10분)보다 길게(15분) 잡아서, 재큐잉된 잡을 다른 Worker가 가져가도 잠금에 걸려서 중복 처리가 안 되게 했다.

```java
boolean locked = distributedLockPort.tryLock(lockKey, 0, 15, TimeUnit.MINUTES);
// 0초 대기(못 잡으면 포기), 15분 유지(>Watchdog 10분)
```

## Port/Adapter로 추상화

[클린 아키텍처](/posts/멀티모듈-백엔드-세팅/)를 쓰고 있으니까 도메인 코드가 Redisson을 직접 몰라야 한다. `DistributedLockPort` 인터페이스를 만들고, 테스트할 때는 `ConcurrentHashMap`으로 구현한 Fake를 쓴다. Redis 없이도 테스트 가능.

## 정리

| 상황 | waitTime | leaseTime |
|------|----------|-----------|
| API 요청 (사용자가 잠깐 기다려도 됨) | 3초 | 10초 |
| Worker (못 잡으면 포기) | 0초 | 15분 |

잠금 해제할 때 `isHeldByCurrentThread()` 체크는 필수다. leaseTime이 만료된 후에 unlock하면 다른 프로세스가 잡은 잠금을 풀어버리는 사고가 난다.

그리고 중요한 거 하나. **잠금은 트랜잭션 시작 전에 걸어야 한다.** 트랜잭션 안에서 잠금을 걸면 다른 트랜잭션이 이미 읽은 데이터를 기반으로 작업해서 Lost Update가 발생할 수 있다.

## 작년에 배운 거랑 연결

작년에 백엔드 학습 과정에서 동시성 제어를 꽤 깊게 다뤘었다. InMemory 락 → DB 비관적 락 → DB 낙관적 락 → Redis 분산 락 순으로 단계별로 전환하는 과제를 했는데, 그때는 수업 듣고 과제 치느라 정신없이 지나갔다. 멘토링에서 "잠금 순서 틀리면 데드락 난다", "잠금 범위는 최대한 좁게" 같은 얘기를 들었는데, 그때는 "아 그렇구나" 하고 넘겼다.

근데 이번에 실제 프로젝트에 적용하면서 그때 들었던 내용들이 하나씩 떠올랐다. "아 이걸 고려했어야 했지" 하는 순간이 계속 온다. 학습 때 쌓아둔 내용들이 기술부채처럼 머릿속에 남아있다가, 실제로 써먹을 때 비로소 내 것이 되는 느낌. 이번이 그런 기회다.
