## MSW → 실제 API 전환 계획

가장 큰 남은 작업. 현재 MSW로 전체 API를 목으로 돌리고 있다. BE가 준비되면:

1. `.env` 파일에서 `VITE_USE_MOCK=false`로 전환
2. Mapper 단위 테스트로 실제 응답 형식 검증
3. 에러 핸들링 실제 상황 대응 (4xx, 5xx)
4. 로딩/에러 상태 UI 실사용 확인

Mapper 패턴의 보상이 여기서 온다. 서버 응답 형식만 맞으면 나머지는 자동 변환.

## Claude Code 6일 실전 리뷰 — 잘된 것 vs 안 된 것

6일 내내 Claude Code랑 붙어서 작업했으니까 솔직한 리뷰를 좀 하겠다.

### 잘된 것

- **초기 세팅 속도**: "이 구조로 프로젝트 세팅해줘" 하면 131개 파일이든 50개 컴포넌트든 한번에 쭉 만들어준다. 이건 사람이 못 따라간다
- **반복 작업 자동화**: Analytics 계측 코드 10개 페이지에 넣기, CRUD API 엔드포인트 전부 만들기, 타입 정의 뽑기 같은 건 Claude가 훨씬 빠르다
- **아키텍처 논의 상대**: "CachePort를 어디에 둘까?" 같은 설계 질문에 의존성 방향을 따져서 답해줌
- **코드베이스 탐색/문서화**: 내 코드를 구조화된 리포트로 정리해줌
- **Team 기능**: "code-forge-toss" 팀으로 여러 모듈을 병렬 작업

### 안 된 것 & 삽질

- **원하지 않는 수정 시도**: 탐색만 시켰는데 파일을 수정하려 해서 여러 번 거부/중단함
- **경로 인식 실패**: 프로젝트 폴더 이름에 공백이 있으면 하이픈으로 추측해서 "No such file" 에러 반복
- **팀 상태 충돌**: "Already leading team" 에러로 cleanup → 재생성 필요
- **환경 파일 실수**: `.env.production`을 git에 올려버림 → 내가 리뷰에서 발견
- **analytics 중복 발화**: 이벤트 의존성 관리를 놓쳐서 추가 수정 필요
- **UX 디테일 누락**: MaintenanceGuard 로딩 중 앱 노출 같은 건 결국 사람이 눈으로 확인해야 함

### 결론

생산성은 확실히 3~4배 올라간다. 근데 **"다 시켜도 되는 도구"가 아니라 "빠른 팀원"**으로 봐야 한다. 방향 설정은 내가, 속도는 Claude가. 기획서 같은 명확한 스펙이 있을수록 결과물 퀄리티가 올라가고, "알아서 해줘"하면 삽질이 늘어난다.

## 6일간 고민한 것들

돌아보면 기술적 결정보다 **"이거 지금 해야 하나?"** 판단이 더 많았다.

- **신고 기능**: 만들어두되 숨겼다. 혼자 쓰는데 신고가 왜 필요한가. 나중에 사용자 붙으면 플래그로 켠다
- **보상형 광고**: 리워드 줄 게 없으면 넣으면 안 된다. 기술적으로 가능하다고 다 넣는 건 아니다
- **네이밍**: recipe로 퉁치면 나중에 다른 앱 추가할 때 다 엎어야 한다. 처음부터 앱 이름으로 구분
- **어드민 분리**: 코어에 붙이면 의존성이 꼬인다. 별도 앱서버로 분리
- **디자인 시스템 분리**: 사용자 앱은 Toss TDS, 어드민은 Carbon. 목적이 다르면 디자인도 달라야 한다
- **보안**: 비밀키는 코드에 안 적고, 관리자 페이지 접근 제한, 위험한 HTML 필터링, URL에서 민감정보 제거. 이건 타협 없이
- **MSW 선택**: BE 없이 FE를 먼저 만들 수 있게. 한계는 알고 있다

사업자등록 끝나면 토스 client ID 발급받고, 그때부터 실제 API 연동이랑 배포 작업이 시작된다.
