⭐6 [학습] React 렌더링 최적화 실전

By 코드대장간
Posted Mar 26 Updated Mar 26 3 min read ⭐6

매 렌더마다 새 배열이 만들어진다

TasteNote 요리 모드를 만들면서 발견한 렌더링 함정들. 코드가 “동작”은 하는데 불필요한 렌더가 반복되는 패턴이 여럿 있었다.

인라인 fallback 객체

1
2

// Bad: 매 렌더마다 새 빈 배열이 생성됨
const ingredients = recipe?.ingredients ?? [];

?? []는 매 렌더마다 새로운 빈 배열 참조를 만든다. 하위 컴포넌트에 props로 넘기면 “값은 같은데 참조가 다르니까” 리렌더가 발생한다. 같은 내용의 편지를 매번 새 봉투에 넣어 보내면 받는 사람은 “새 편지 왔나?” 열어보게 된다.

1
2
3
4
5
6
7
8
9

// Good: 안정적인 참조
const EMPTY_ARRAY: Ingredient[] = [];
const ingredients = recipe?.ingredients ?? EMPTY_ARRAY;

// 또는 useMemo
const ingredients = useMemo(
  () => recipe?.ingredients ?? [],
  [recipe?.ingredients]
);

모듈 상수로 빈 배열을 정의하거나, useMemo로 참조를 안정화.

Set vs Array — O(1) vs O(n)

요리 모드에서 완료한 스텝을 체크한다. 스텝 카드를 렌더할 때마다 “이 스텝이 완료됐는가?”를 확인한다.

1
2
3
4
5
6
7

// Bad: Array.includes() = O(n)
const completedSteps = [1, 3, 5];
completedSteps.includes(stepId); // 매번 배열 순회

// Good: Set.has() = O(1)
const completedSteps = new Set([1, 3, 5]);
completedSteps.has(stepId); // 해시 조회

스텝이 30개고 각 스텝 렌더마다 includes를 호출하면 30 × 30 = 900번 비교. Set이면 30번. 도서관에서 책을 찾을 때 처음부터 훑는 것(Array)과 색인을 보는 것(Set)의 차이.

단축 평가 (Short-circuit)

비싼 연산을 조건부로 건너뛸 수 있다.

1
2
3
4
5

// Bad: isActive가 false여도 expensiveCalc 실행
const result = isActive && expensiveCalc(data) || defaultValue;

// Good: isActive가 false면 바로 defaultValue
const result = isActive ? expensiveCalc(data) : defaultValue;

연산이 필요 없는 경우를 먼저 걸러내는 것. 가게에 손님이 없으면 주문서를 작성할 필요가 없다.

CSS shorthand/longhand 충돌

1
2
3
4
5
6

/* Bad: shorthand가 longhand를 덮어씀 */
.card {
  border: 1px solid gray;
  border-bottom-color: red;  /* 의도: 아래만 빨간색 */
  border: 1px solid gray;    /* 다시 선언하면 border-bottom-color도 리셋 */
}

CSS에서 border(shorthand)와 border-bottom-color(longhand)를 섞으면 선언 순서에 따라 의도치 않게 덮어써진다. 인라인 스타일에서 특히 주의. React의 style 객체에서 border와 borderBottomColor를 같이 쓰면 같은 문제.

프로젝트 적용

요리 모드 컴포넌트에서 위 패턴들을 적용:

  • CookingIngredientList: EMPTY_ARRAY 상수로 fallback 안정화
  • CookingStepCards: completedSteps를 Set으로 관리
  • CookingPage: useShallow + useMemo 조합으로 불필요한 리렌더 제거
