[앱인토스 Admin] API 클라이언트 인프라 구축

By 코드대장간
Posted Mar 18 Updated Mar 20 6 min read

API 연동 시작

어제 “API 연동 미시작”이었다. 오늘 시작했다. 뼈대만 있던 어드민에 API 클라이언트 인프라를 쌓는 작업.

snake_case ↔ camelCase 자동 변환

BE가 JSON 응답을 snake_case로 내려준다. Spring 기본은 camelCase지만, BE에서 Jackson PropertyNamingStrategies.SNAKE_CASE로 명시 설정한 것. FE는 TypeScript니까 camelCase로 써야 한다. 매번 수동으로 변환하면 실수가 나온다. Axios 인터셉터에서 자동 변환하도록 구현.

// Request: camelCase → snake_case
request.interceptors.request.use(config => {
  config.data = toSnakeCase(config.data);
  return config;
});

// Response: snake_case → camelCase
request.interceptors.response.use(response => {
  response.data = toCamelCase(response.data);
  return response;
});

caseConverter 유틸은 재귀적으로 중첩 객체, 배열 내부까지 전부 변환한다. 토큰 리프레시 흐름도 마찬가지 — refresh 요청의 body와 응답 모두 변환이 적용되어야 한다. 이걸 인터셉터 레벨에서 한번 깔아놓으면 이후 API 코드에서 케이스 변환을 신경 쓸 일이 없다.

ApiResponse 래퍼 처리

BE의 모든 응답이 ApiResponse<T> 래퍼로 감싸져 온다. 택배 상자 안에 또 상자가 있는 거다. 매번 .data.data로 꺼내는 건 번거롭다. Response 인터셉터에서 자동 언래핑.

처음에는 ApiResponse<T>를 단순 인터페이스로 만들었는데, success 필드로 성공/실패를 구분하는 discriminated union으로 강화했다. 타입 레벨에서 성공 응답과 에러 응답을 구분할 수 있게.

1
2
3

type ApiResponse<T> =
  | { success: true; data: T; error: null }
  | { success: false; data: null; error: ApiErrorPayload };

에러 페이로드 인터페이스 이름도 ApiError에서 ApiErrorPayload로 바꿨다. 클라이언트 쪽 ApiError 클래스랑 이름이 겹쳐서 혼동이 오니까.

에러 인터셉터

에러 처리를 한 곳에서 잡아야 한다. 세 가지 케이스를 처리.

  1. success: false인 응답: error가 null이어도 UNKNOWN_ERROR로 throw. 실패 응답이 조용히 성공으로 넘어가는 걸 방지
  2. ApiResponse 래퍼가 아닌 에러: 구조화된 에러 페이로드({ message })도 ApiError로 변환
  3. hooks에서의 에러 표시: err.message가 백엔드가 보낸 메시지를 그대로 표시하도록

이전에는 hooks에서 에러 추출 방식이 제각각이었다. err instanceof Error 패턴으로 통일.

API 엔드포인트 정리

BE에서 만든 Admin API 가이드에 맞춰 엔드포인트 상수를 전면 수정.

변경 전변경 후이유
/dashboard/revenue-summary/dashboard/revenue간결화
/payments/settlement/payments/settlements복수형 통일
/app-config/app-configs복수형 통일
/apps/recipe/featured/featured-recipes독립 경로
/system/maintenance/maintenance독립 경로

HTTP 메서드도 수정했다. DELETE → POST .../delete, PATCH → POST .../toggle 같은 식으로 BE 스펙에 맞춤. 누락된 엔드포인트 12개 추가, 중복 제거.

PR #1 머지

이 작업들을 모아서 첫 번째 PR을 머지했다. 어드민 프로젝트 첫 PR이니까 감회가 좀 있다.

Claude Code 협업

BE에서 만든 Admin API 가이드 문서를 Claude한테 주고 “이 스펙대로 API 클라이언트 맞춰줘”라고 했다. 엔드포인트 매핑이나 DTO 타입 추출 같은 반복 작업은 빠르게 처리해줬다.

다만 에러 핸들링 쪽은 Claude가 처음에 만든 게 허술했다. success: false인데 error가 null인 케이스를 안 잡고 있었다. “실패 응답이 성공으로 넘어갈 수 있는데?”라고 짚으니까 바로 수정. discriminated union으로 타입을 강화하자는 것도 내가 제안한 부분. 타입 시스템으로 잡을 수 있는 건 런타임까지 미루지 않는다.

오늘의 결과

항목상태
snake_case/camelCase 자동 변환완료
ApiResponse 래퍼 언래핑완료
에러 인터셉터 + 에러 핸들링 통일완료
API 엔드포인트 상수 전면 수정완료
API 서비스 메서드 수정 + 누락 추가완료
PR #1 머지완료

어제 “뼈대 완성, API 연동 미시작”이었는데, 오늘 API 연동 인프라가 깔렸다. 이제 실제 데이터로 화면을 채우는 일만 남았다.
