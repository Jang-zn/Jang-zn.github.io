## 처음 가는 식당에서

메뉴판이 50페이지인 식당이 있다. 앉자마자 두꺼운 바인더가 나온다. 이미 뭘 시킬지 정한 사람은 괜찮다. 처음 온 사람은 다르다. 5분 넘게 넘기다가 그냥 나간다.

직원이 한마디 한다. "오늘 이거 추천이요." 앉는다. 주문한다.

앱도 마찬가지다. 처음 켠 사용자에게 기능 20개를 한 번에 보여주면 아무것도 안 한다. 누군가 "여기 먼저 눌러봐"라고 한마디 해주면 행동이 시작된다. 그게 튜토리얼이다.

---

## 튜토리얼 방식 비교

방법이 하나는 아니다. 각각 장단이 있다.

| 방식 | 설명 | 장점 | 단점 |
|------|------|------|------|
| **온보딩 슬라이드** | 앱 첫 실행 시 3~5장 스와이프 | 전체 기능을 한 번에 설명 | 대부분 스킵한다. 기억도 안 남는다 |
| **빈 상태 메시지** | "아직 레시피가 없어요. + 버튼을 눌러보세요" | 맥락에 맞는 안내 | 특정 UI 요소를 직접 가리킬 수 없다 |
| **코치마크/스포트라이트** | 특정 UI 요소를 하이라이트 + 설명 툴팁 | 맥락 기반, 단계적, 직관적 | 스텝이 많으면 피로. 구현도 복잡 |
| **인터랙티브 튜토리얼** | 유저가 직접 행동해야 다음 단계로 | 학습 효과 최고 | 구현 매우 복잡. 유저 자유도 제한 |

온보딩 슬라이드는 설명서를 읽고 나서 기계를 만지는 것이다. 코치마크는 기계 앞에서 "이 버튼 눌러봐"라고 알려주는 것이다. 어느 쪽이 기억에 남는지는 뻔하다.

## Progressive Disclosure

**정의**: 필요한 정보를 필요한 시점에만 보여주는 설계 원칙. 모든 기능을 첫 화면에 때려넣지 않는다.

**출처**: Nielsen, J. (2006). *"Progressive Disclosure"* — Nielsen Norman Group.

냉장고 문을 열면 냉장실이 보인다. 냉동실은 위에 따로 있다. 김치칸은 아래 서랍을 열어야 나온다. 냉장고 문 열자마자 모든 칸이 한꺼번에 열리면 어디에 뭘 넣어야 할지 모른다.

앱도 동일하다. 첫 화면에 설정, 통계, 필터, 정렬, 검색, 알림을 전부 보여주면 핵심이 묻힌다. 지금 이 시점에 이 사용자에게 필요한 것만 보여준다. 나머지는 필요할 때 꺼낸다.

이전 포스트에서 다룬 [선택피로(Choice Overload)](/posts/진입점-통합과-선택피로/)와 직결되는 개념이다. 선택지를 줄이는 방법 중 하나가 Progressive Disclosure다.

튜토리얼에서 Progressive Disclosure는 이렇게 작동한다:
- 첫 진입 시: 핵심 기능 1~2개만 안내
- 특정 기능 첫 사용 시: 해당 기능의 세부 안내
- 고급 기능은 사용자가 찾을 때까지 설명하지 않는다

## 코치마크 설계 원칙

### 1. 행동 유도 문구

"여기서 레시피를 검색할 수 있어요" — 이건 설명이다. 가능성만 알려준다. 행동이 안 일어난다.

"레시피를 검색해서 원하는 요리를 찾아보세요" — 이건 유도다. 구체적 행동 + 결과가 있다. 차이가 크다.

| 나쁜 예 | 좋은 예 | 차이 |
|---------|---------|------|
| "재료를 추가할 수 있어요" | "+ 버튼을 눌러 냉장고에 재료를 추가해보세요" | 구체적 행동(+ 버튼) + 목적(냉장고에 추가) |
| "여기서 수정 가능해요" | "밥해먹기로 바로 요리를 시작하거나, 수정·삭제할 수 있어요" | 기능명 명시 + 가능한 행동 나열 |
| "체크하세요" | "재료가 준비되었으면 체크하고, 없으면 패스하세요" | 상황(준비됨/없음) + 행동(체크/패스) |

"할 수 있어요"는 관광 안내소에서 팸플릿 주는 것이다. "해보세요"는 직원이 직접 데려다주는 것이다.

### 2. 스텝 수 제한

한 튜토리얼에 스텝이 10개면 아무도 끝까지 안 본다. 가이드라인:

```
1~3 스텝: 이상적. 집중력이 유지된다
4~5 스텝: 허용. 기능이 복잡한 경우에만
6+ 스텝: 주의. 화면/탭 전환으로 자연스럽게 분할되지 않으면 줄여야 한다
```

같은 6스텝이라도 한 화면에서 6개는 지루하다. 2개 화면 x 3개씩은 괜찮다. 자연스러운 구분선이 있으면 체감이 다르다.

### 3. Skip 보장

건너뛰기는 항상 가능해야 한다. 이유:
- 재방문 사용자에게 튜토리얼은 소음이다
- 이미 아는 사람에게 강제하면 짜증만 난다
- 스킵해도 "완료"로 기록해서 다시 안 뜨게 한다

### 4. 맥락 기반 트리거

튜토리얼이 뜨는 타이밍:
- **첫 진입 시**: 앱을 처음 켰을 때, 특정 화면에 처음 들어왔을 때
- **특정 기능 첫 사용 시**: 요리 모드 첫 진입, 재료 추가 첫 시도
- **행동 분기 시**: 공통 재료를 선택했는지, 직접 추가를 선택했는지에 따라 다른 튜토리얼

무작위로 튀어나오면 안 된다. 사용자의 현재 맥락과 맞아야 한다.

### 5. 상태 충돌 방지

두 개의 안내가 동시에 뜨면 둘 다 못 읽는다. 바텀시트 위에 코치마크가 겹치면 혼란만 가중된다.

| 상태 | 튜토리얼 차단? | 이유 |
|------|---------------|------|
| 바텀시트 열림 | 차단 | 스포트라이트가 시트와 겹긴다 |
| 토스트 표시 중 | 허용 | 짧은 알림이라 겹침 무관 |
| 로딩 스피너 | 차단 | 타겟 요소가 가려져 있다 |
| 다른 튜토리얼 진행 중 | 차단 | 두 튜토리얼 동시 진행은 불가 |
| 네트워크 에러 | 차단 | 정상 흐름이 아니다 |

엘리베이터 문이 닫히는 중에 안내 방송을 틀면 아무도 안 듣는다. 문이 닫히고, 조용해지고, 그 다음에 안내한다.

## 맥락 일치 원칙

보이지 않는 걸 설명하면 의미가 없다. "드래그해서 순서를 바꿀 수 있다"는 설명이 재료 확인 탭에서 뜨는데, 드래그 기능은 조리 순서 탭에만 있다. 사용자 눈에는 드래그 핸들이 안 보인다. 뭘 드래그하라는 건지 모른다.

해결: 해당 설명이 나올 때 자동으로 탭을 전환한다. 조리 순서 탭의 실제 UI가 보이는 상태에서 설명한다.

튜토리얼 스텝을 설계할 때 항상 확인해야 하는 것:

> "이 설명이 뜰 때, 사용자 화면에 뭐가 보이는가?"

보이지 않는 요소를 가리키는 건 빈 벽을 가리키며 "저기 문 있어요"라고 하는 거다.

## 안티패턴

잘못 만든 튜토리얼은 안 만든 것보다 나쁘다.

**너무 많은 스텝**: 10스텝 넘으면 이탈률이 급격히 올라간다. 사용자는 튜토리얼을 "공부"하러 온 게 아니다. 빨리 쓰고 싶어서 왔다.

**강제 튜토리얼**: 스킵 버튼이 없다. 끝까지 봐야 앱을 쓸 수 있다. 사용자 입장에서는 입장료를 내는 기분이다. 두 번째 방문부터는 그냥 안 온다.

**추상적 문구**: "환영합니다!" — 그래서 뭘 하라는 건가. "이 앱은 당신의 요리 생활을 도와줍니다" — 알겠다. 그래서 어디를 누르면 되는가. 구체적 행동이 없는 문구는 공간 낭비다.

**다시 볼 수 없는 튜토리얼**: 한 번 봤는데 기억이 안 난다. 다시 보고 싶다. 방법이 없다. 사용자는 "아까 그게 뭐였지?" 상태로 헤맨다. 초기화 수단을 제공하거나, 도움말에서 다시 볼 수 있어야 한다.

## 분기 튜토리얼

모든 사용자가 같은 경로를 거치지 않는다. 공통 재료를 선택한 사용자에게 "카테고리를 선택하세요"라고 안내하면 혼란이다. 카테고리 선택은 직접 추가 경로에서만 필요하다.

사용자 행동에 따라 분기한다:
- 공통 재료 선택 → 수량/단위 입력 안내
- 직접 추가 선택 → 카테고리 선택 → 수량/단위 입력 안내

기획자가 정의해야 하는 건 "어떤 상황에서 어떤 튜토리얼이 나타나야 하는가?"다. 개발자는 이 정의에 따라 조건 분기를 구현한다. 정의가 없으면 개발자가 임의로 만든다. 그러면 사용자가 혼란스러워한다.

## TasteNote 적용

기존 밥해먹자 튜토리얼은 재료 확인 탭에만 있었다. 재료 검색, 조리 단계 안내가 빠져 있었다. 9스텝으로 확장하되, 2개 탭에 걸쳐 분산시켰다. 체감은 4+5. 탭 자동 전환으로 맥락 일치를 보장했다. 상세 구현은 [코치마크 확장과 환경 분리](/posts/코치마크-확장과-환경-분리/) 참고. [넛지 사이클](/posts/넛지사이클과-인라인편집-UX/)에서 다룬 Prompt 설계와도 연결된다 — 코치마크 자체가 Fogg Model의 Prompt인 거다.
