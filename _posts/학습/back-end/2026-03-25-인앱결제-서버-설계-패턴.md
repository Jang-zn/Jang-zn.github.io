## 결제는 한 번만 처리돼야 한다

[TasteNote 인앱결제](/posts/인앱결제-광고-제거-시스템/)를 구현하면서 결제 서버 설계 패턴을 여럿 적용했다. 결제는 돈이 오가는 영역이라 "한 번 더 실행돼도 괜찮은" 설계가 필수다. 작년 학습 과정에서 배운 [멱등성](/posts/멱등성과-중복-방지/)이 여기서 써먹혔다.

## 멱등성 (Idempotency)

같은 요청을 여러 번 보내도 결과가 한 번만 적용되는 성질. FE에서 결제 완료 후 서버에 "이 주문 처리해줘"를 보내는데, 네트워크 에러로 응답을 못 받으면 같은 요청을 재전송한다. 이때 구매가 두 번 생기면 안 된다.

ATM에서 출금 버튼을 두 번 눌러도 돈이 두 번 나오면 안 되는 거다.

### 구현 방식

`orderId`를 고유 키로 사용. `grantProduct()` 진입 시 먼저 같은 `orderId`로 처리된 구매가 있는지 조회.

```
if (이미 처리된 orderId 있으면)
  → 기존 결과 반환 (새로 처리하지 않음)
else
  → 신규 처리
```

DB의 `orderId` 컬럼에 unique 제약을 걸면 동시 요청이 와도 하나만 성공한다. 나머지는 unique 제약 위반으로 실패하고, 재조회해서 기존 결과를 반환한다.

## 서버 측 결제 검증

FE가 "결제 완료"라고 보내도 서버가 직접 확인해야 한다. 결제 금액 변조, 취소된 결제, 존재하지 않는 주문 — FE 요청만 믿으면 이런 걸 걸러낼 수 없다.

Toss Payments API의 `getOrderStatus`를 호출해서 주문 상태와 금액을 검증한다.

| 검증 항목 | 확인 방법 |
|-----------|----------|
| 주문 존재 여부 | Toss API 응답 200 |
| 결제 상태 | `status == DONE` |
| 결제 금액 | 서버 상품 가격과 비교 |

검증 실패 시 자동 환불 처리. "돈은 빠졌는데 상품은 안 나오는" 상태를 만들지 않는다. 마트에서 계산은 됐는데 물건 바코드가 안 찍히면 바로 환불해주는 거다.

## 구매 부여 플로우

전체 플로우를 시퀀스로 보면:

```
FE → 서버: grant(orderId, productType)
  1. 멱등성 체크 (orderId 중복)
  2. Toss API 검증 (상태, 금액)
     → 실패: 자동 환불 + 에러 반환
  3. 기존 동일 상품 타입 구매 확인
     → 있으면: 연장 처리
     → 없으면: 신규 생성
  4. 결과 반환
```

연장 처리는 월간 상품에 해당. 이미 월간 구매가 활성 상태인데 또 결제하면, 기존 만료일에 30일을 더한다. 영구 상품은 연장 개념이 없다.

## 광고 제거 상태 캐시

`getAdFreeStatus()`는 매 화면에서 호출된다. 광고를 보여줄지 말지 결정하는 거라 자주 호출된다. 매번 DB를 조회하면 낭비.

Redis에 5분 캐시. 키는 `ad-free:{userId}`. 캐시 미스 시 DB 조회 후 캐시에 저장.

5분이면 결제 직후 최대 5분간 광고가 계속 보일 수 있다. 하지만 구매 부여 시 캐시를 즉시 갱신하면 실제로는 거의 즉시 반영된다. 캐시 TTL은 "서버 재시작이나 캐시 eviction 후 복구"를 위한 안전장치.

## 월간 구매 만료 스케줄러

월간 상품은 30일 후 만료된다. 만료 처리를 어디서 할 것인가.

| 방식 | 장점 | 단점 |
|------|------|------|
| 조회 시 체크 | 별도 배치 불필요 | 안 열어보면 영영 ACTIVE |
| 스케줄러 배치 | 상태가 항상 정확 | 배치 부하 |

스케줄러를 선택했다. 만료됐는데 ACTIVE로 남아있으면 어드민 대시보드에서 활성 사용자 수가 부풀려진다. 데이터 정합성이 우선.

매시간 실행. 만료 대상을 500건씩 페이징 조회. 전체를 한 번에 가져오면 만료 건이 수만 개일 때 메모리가 터진다. 뷔페에서 접시를 한꺼번에 다 치우지 않고 한 테이블씩 정리하는 거다.

## 환불 플로우

자동 환불이 아니라 수동 승인 방식.

```
사용자 → 환불 요청 → REFUND_REQUESTED
  → Discord 알림 → 어드민 확인
  → 승인 → REFUNDED
```

1인 개발이라 복잡한 환불 정책 엔진을 만들 여력이 없다. Discord 알림이 오면 직접 확인하고 Toss 관리자에서 환불 처리. 환불 가능 조건은 도메인 모델에서 체크한다(영구 상품만, ACTIVE 상태만).

## 정리

| 패턴 | 적용 위치 | 효과 |
|------|----------|------|
| 멱등성 | `grantProduct()` | 중복 결제 방지 |
| 서버 검증 | Toss API 호출 | 변조/취소 방어 |
| 자동 환불 | 검증 실패 시 | 결제-미지급 방지 |
| Redis 캐시 | `getAdFreeStatus()` | DB 부하 감소 |
| 페이징 배치 | 만료 스케줄러 | 메모리 안전 |
| 수동 승인 | 환불 플로우 | 1인 운영 현실적 |

결제 서버는 "정상 플로우"보다 "비정상 플로우"를 더 많이 고민해야 한다. 네트워크 끊김, 중복 요청, 결제 변조, 만료 누락 — 다 "안 터질 것 같지만 터지면 돈이 걸린" 케이스다.
