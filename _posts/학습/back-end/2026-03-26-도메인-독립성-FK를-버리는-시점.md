## FK가 있으면 무조건 좋은 게 아니다

[TasteNote V2에서 Pantry를 독립화](/posts/Pantry-독립화와-재료-연결/)했다. 기존에 `pantry_item.ingredient_id NOT NULL`로 FK를 걸어뒀는데, 이게 오히려 문제였다. FK를 언제 써야 하고 언제 버려야 하는지 기준을 정리한다.

## FK의 역할

FK(Foreign Key)는 DB 레벨에서 참조 무결성을 보장한다. `pantry_item.ingredient_id = 42`면, `ingredient` 테이블에 id=42가 반드시 존재해야 한다. 없는 재료를 참조할 수 없다.

냉장고 선반에 상품 바코드를 붙여놓고, 마트에 없는 바코드는 냉장고에 못 넣게 하는 거다.

## FK 적합성 판단 기준

**핵심 질문: "이 엔티티는 저 엔티티 없이 존재할 수 있는가?"**

| 케이스 | FK 적합? | 이유 |
|--------|---------|------|
| `order_item.product_id` | O | 상품 없는 주문 항목은 무의미 |
| `comment.post_id` | O | 게시글 없는 댓글은 무의미 |
| `pantry_item.ingredient_id` | X | 냉장고는 사용자 것. 재료 마스터와 독립 |
| `cooking_log.recipe_version_id` | O | 버전 없는 요리기록은 무의미 |

`pantry_item`이 `ingredient_id`를 FK로 가지면, "고수"를 냉장고에 넣고 싶어도 `ingredient` 테이블에 "고수"가 없으면 404가 난다. 사용자의 냉장고가 공통 재료 사전에 종속되는 구조다.

## DDD 관점 — Aggregate 경계

DDD에서 Aggregate는 함께 변하는 객체의 묶음이다. Aggregate 간에는 강제 FK 없이 ID 참조만 한다.

- `PantryItem`은 사용자의 냉장고 항목. 독립적인 Aggregate.
- `Ingredient`는 공통 재료 마스터. 별개의 Aggregate.

두 Aggregate는 ID(nullable)로 느슨하게 연결한다. FK로 묶으면 Aggregate 경계가 무너진다. 두 회사가 계약서(FK)로 묶여있으면 한 쪽이 없어지면 계약도 끝난다. 구두로 협력하는(nullable ID) 관계는 한 쪽이 없어져도 다른 쪽이 독립적으로 존재한다.

## 독립화 이후 설계

```
Before: pantry_item { user_id, ingredient_id NOT NULL }
After:  pantry_item { user_id, name NOT NULL, category, ingredient_id NULL }
```

`ingredient_id`를 nullable로 남겨두는 이유: 나중에 선택적으로 연결할 수 있다. `PATCH /api/v1/recipe-ingredients/{id}/link`로 수동 연결. 연결된 재료는 더 정확한 매칭이 가능하고, 연결 안 된 재료도 이름 매칭으로 동작한다.

## 트레이드오프

FK를 버리면 정합성 보장이 DB에서 애플리케이션 레벨로 이동한다.

| | FK 있음 | FK 없음 (독립화) |
|---|---|---|
| 정합성 보장 | DB 레벨 | 앱 레벨 |
| 유연성 | 낮음 (마스터에 종속) | 높음 (자유롭게 추가) |
| 고아 데이터 위험 | 없음 | 있음 (주의 필요) |
| 적합한 경우 | 비즈니스적으로 종속된 관계 | 독립적으로 존재 가능한 관계 |

"DB가 정합성을 보장해주면 좋지 않냐"는 맞다. 하지만 그 정합성 제약이 비즈니스 요구사항과 맞지 않으면 오히려 방해가 된다. 냉장고에 아무 재료나 자유롭게 넣을 수 있어야 하는데, DB가 "마스터에 없는 재료는 못 넣어"라고 막으면 안 된다.
