## 예외 처리가 생겼다면 설계를 의심해라

[TasteNote 요리모드](/posts/요리모드-v2-구현과-UX-개선/)에서 `IngredientAction`을 리팩터링했다. 처음에 `'USED' | 'ADJUSTED' | 'SKIPPED'` 세 가지를 하나의 타입으로 묶었다. `hasModifications()` 계산에서 USED를 제외하는 예외가 필요해졌다.

```typescript
// Bad: 예외 처리가 필요한 구조
const hasModifications = Object.values(ingredientActions)
  .some(a => a !== 'USED'); // USED는 "수정"이 아니라 그냥 체크
```

`a !== 'USED'`라는 예외가 생기는 순간, 설계 문제의 신호다.

## UI 상태 vs 도메인 상태

| | UI 상태 | 도메인 상태 |
|---|---|---|
| 정의 | 화면 표현을 위한 임시 상태 | 비즈니스적으로 의미 있는 변경 |
| 예시 | 모달 열림, 체크 여부, 선택된 탭 | 수량 변경, 재료 제외, 메모 |
| 저장 필요? | X | O (서버 전송) |
| 새로고침 | 사라져도 됨 | 유지돼야 함 |
| 새 버전 반영 | X | O |

"재료를 체크했다"는 UI 상태다. "오늘 요리에서 소금을 뺐다"는 도메인 상태다. 둘이 다르다.

## 판단 기준 4가지

1. **서버에 저장해야 하나?** → 도메인 상태
2. **다음 세션에서도 봐야 하나?** → 도메인 상태
3. **이 화면에서만 의미 있나?** → UI 상태
4. **새로고침하면 없어져도 되나?** → UI 상태

`USED`(체크)는 요리 중 진행 상황 표시. 화면에서만 의미 있고, 새 버전에 반영될 필요가 없다. → **UI 상태**.

`ADJUSTED`(수량 변경), `SKIPPED`(제외)는 실제로 레시피를 다르게 만든 것. 새 버전 저장 시 반영돼야 한다. → **도메인 상태**.

## 분리 후

```typescript
// Before: 하나의 타입에 섞임
ingredientActions: Record<string, 'USED' | 'ADJUSTED' | 'SKIPPED'>

// After: 역할에 따라 분리
checkedIngredientIds: string[]                    // UI 상태
ingredientAdjustments: Record<string, string>     // 도메인 상태 (수량)
skippedIngredientIds: string[]                    // 도메인 상태 (제외)
```

`hasModifications()`가 단순해진다.

```typescript
// Before
some(a => a !== 'USED')

// After
ingredientAdjustments 키가 있거나, skippedIngredientIds가 있으면
```

예외 처리가 사라졌다. 타입이 의미를 명확히 전달한다.

## 교훈

`checkedStepIds`(완료된 단계)는 처음부터 `string[]`로 분리돼 있었다. 재료도 처음부터 이 패턴을 따랐다면 나중에 뜯지 않았을 것이다.

새 상태를 추가할 때 항상 물어봐야 한다: "이건 화면을 위한 건가, 비즈니스 논리를 위한 건가?" 대답이 불명확하면 분리하는 게 낫다. 섞어놓고 나중에 뜯는 게 더 비싸다.
