[학습] Redis 캐시 직렬화 충돌 — 네이밍 전략 바꾸면 캐시가 깨진다

By 코드대장간
Posted Mar 19 Updated Mar 19 3 min read

문제 상황

Jackson 네이밍 전략을 바꿨다. SNAKE_CASE → camelCase. API 응답은 바로 바뀌지만, Redis에 이미 저장된 캐시 데이터는 안 바뀐다.

1
2

Redis에 저장된 값: {"recipe_id":"r-1","main_img_url":"https://..."}
새 ObjectMapper가 기대하는 형식: {"recipeId":"r-1","mainImgUrl":"https://..."}

키 이름이 안 맞으니 역직렬화가 실패한다. 에러가 터지는 게 아니라 warning 로그만 남기고 캐시 미스 처리된다. 모든 요청이 DB를 때린다. 캐시 스탬피드의 조건이 만들어진다.

해결 방법 3가지

방법장점단점
Redis 전체 flush간단모든 캐시 소실, 배포 스크립트 필요
캐시 키 버저닝무중단, 점진적구 캐시가 TTL까지 메모리 차지
캐시 전용 ObjectMapper전략 변경에 독립적ObjectMapper 2개 관리 부담

적용한 방법: 캐시 키 버저닝

앱인토스-통합 Api에서 캐시 키 프리픽스에 버전을 넣었다.

1
2

before: recipe:{recipeId}
after:  recipe:v2:{recipeId}

새 배포 후 새 키(v2)로 캐시가 쌓인다. 구 키(v1)는 TTL(10분)이 지나면 자연 소멸. flush 스크립트 없이 무중단으로 전환된다.

TTL이 짧아서 가능한 방식이다. TTL이 24시간이었으면 구 캐시가 하루 동안 메모리를 잡아먹으니 flush가 나을 수 있다.

캐시 스탬피드는 괜찮았나

배포 직후 v2 키로는 캐시가 하나도 없으니 cold start 상태가 된다. 하지만 이 프로젝트에서는 이미 분산 락 기반 캐시 스탬피드 방지가 적용되어 있다. 동일 레시피에 대해 최대 1개 요청만 DB를 조회하고, 나머지는 락 해제 후 캐시를 읽는다. cold start여도 DB 부하가 폭발하지 않는다.

작년 멘토링에서 들은 것

캐시 직렬화 포맷 변경은 작년 학습 과정에서도 다뤘던 주제다. “캐시 스키마가 바뀌면 버전을 올려라”는 원칙. 그때는 Protobuf 직렬화 얘기였는데, JSON 네이밍 전략 변경도 본질은 같다. 직렬화 포맷이 바뀌면 기존 데이터와 호환이 깨진다.

정리

네이밍 전략 변경의 영향 범위는 API 응답만이 아니다. Redis 캐시, 메시지 큐, 파일 저장소 — JSON으로 직렬화된 모든 곳이 영향권이다. “API 키만 바꾸면 되겠지”라고 생각하면 캐시에서 터진다.
