[학습] 클린 아키텍처 — 인프라를 갈아끼울 수 있게

By 코드대장간
Posted Mar 18 5 min read

처음에는 헷갈렸다

BE 초기 세팅 때 MVC만 하다가 클린 아키텍처를 처음 적용했다. Controller → Service → Repository 흐름에 익숙한데, Port & Adapter 패턴은 흐름이 달라서 “이거 뭔 구조야?” 싶었다.

핵심은 도메인 코드가 인프라를 모르게 하는 것이다. 도메인에는 인터페이스(Port)만 있고, 실제 구현(Adapter)은 인프라 모듈에 있다.

1
2

도메인:  ParseJobService → CachePort (인터페이스)
인프라:                     RedisCacheAdapter (구현체)

ParseJobService는 캐시가 필요하다는 건 알지만, 그게 Redis인지 Memcached인지는 모른다.

왜 이렇게 하는지

1. 기술 교체

Redis를 다른 걸로 바꿀 때 도메인 코드를 한 줄도 안 건드려도 된다. 새 Adapter만 만들면 된다. 실제로 이번 프로젝트에서도 분산 락, 캐시, 작업 큐를 전부 Port/Adapter로 만들었는데, 나중에 Redis 말고 다른 거 쓰고 싶으면 Adapter만 바꾸면 된다.

2. 테스트

이게 제일 체감됐다. Redis 없이 테스트를 돌릴 수 있다. 도메인 테스트에서 FakeCachePort, FakeDistributedLockPort 같은 메모리 구현체를 만들어서 쓴다. ConcurrentHashMap 하나면 충분하다.

1
2

실서비스: ParseJobService → CachePort → Redis
테스트:   ParseJobService → CachePort → ConcurrentHashMap

Redis를 띄우고 끄고 할 필요 없이 그냥 돌린다.

3. 의존성 방향

도메인이 인프라에 의존하면 안 된다. 반대여야 한다.

1
2

❌ domain 코드에 import org.redisson.api.RedissonClient
✅ domain 코드에 import CachePort (내가 만든 인터페이스)

적용한 Port들

Port역할Adapter
CachePort캐시 읽기/쓰기/삭제RedisCacheAdapter
DistributedLockPort분산 락RedisDistributedLockAdapter
JobQueuePort작업 큐에 잡 넣기/빼기RedisJobQueueAdapter
StoragePort이미지 업로드/삭제S3StorageAdapter

전부 Redis나 S3에 의존하는 건데, 도메인 코드에서는 그냥 인터페이스만 쓴다.

Fake 테스트의 장점

Fake 구현체에 의도적 실패를 만들 수 있다. “다음 잠금 획득은 실패하게 해줘” — 이런 테스트를 Redis 없이 할 수 있다.

1
2
3

fakeLockPort.setFailNextLock(true);  // 다음 잠금은 실패
RecipeResponse response = recipeService.getRecipe(recipeId);
// → 잠금 실패해도 서비스가 정상 동작하는지 검증

실제 Redis에서 이런 상황을 만들려면 네트워크를 끊거나 해야 하는데, Fake면 변수 하나 바꾸면 된다.

작년 학습 과정에서부터

작년 백엔드 학습 과정에서 클린 아키텍처는 처음부터 나왔다. 그때 헥사고날 아키텍처 문서를 읽고 패키지 구조를 api/domain/infrastructure로 나누는 과제를 했었다. LockingPort 인터페이스 만들고 InMemory → Redis로 Adapter를 갈아끼우는 것도 했다.

솔직히 그때는 “왜 이렇게 복잡하게 하지?” 싶었다. MVC가 더 직관적이고 빠른데. 근데 수업과 멘토링을 거치면서 DB 락을 Redis 락으로 바꾸고, 캐시를 추가하고, 이벤트 시스템을 붙이는 과정에서 Port/Adapter 덕에 도메인 코드를 안 건드리고 인프라만 교체하는 경험을 했다. “아 이래서 이렇게 하는구나”가 나중에 온 거다.

이번 프로젝트에서는 처음부터 이 구조로 갔다. 작년에 쌓아둔 게 지금 써먹히고 있다. 그때 희미하게 남은 것들이 실전에서 하나씩 선명해지는 느낌.

정리

처음에 “MVC가 아니라 헷갈린다”고 했었는데, 적용하고 나니까 왜 이렇게 하는지 이해됐다. 도메인 로직이 인프라 기술에 오염되지 않는다는 게 핵심이다. 테스트할 때 특히 체감된다.
