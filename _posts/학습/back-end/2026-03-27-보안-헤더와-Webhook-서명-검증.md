## 보안 헤더 = 택배 상자에 붙이는 스티커

택배를 보낼 때 상자에 "깨지기 쉬움", "이 면을 위로" 스티커를 붙인다. 택배 기사는 이 스티커를 보고 상자를 조심히 다룬다. HTTP 보안 헤더도 같은 원리다.

서버가 응답에 "이 페이지를 iframe에 넣지 마라", "Content-Type을 추측하지 마라" 같은 지시를 붙이면, 브라우저는 그 지시에 따라 동작한다. 스티커를 안 붙이면? 택배 기사(브라우저)가 알아서 판단한다. 대부분 관대한 방향으로. 그게 공격의 틈이 된다.

```
HTTP/1.1 200 OK
X-Frame-Options: DENY              ← "iframe에 넣지 마라"
X-Content-Type-Options: nosniff    ← "Content-Type 추측하지 마라"
Strict-Transport-Security: ...     ← "HTTPS만 사용하라"
```

---

## X-Frame-Options — 클릭재킹 방어

클릭재킹(Clickjacking)은 이런 공격이다. 공격자가 자기 사이트에 우리 서비스를 iframe으로 끼워넣는다. 그 위에 투명한 레이어를 덮는다. 사용자는 "무료 쿠폰 받기" 버튼을 누른다고 생각하지만, 실제로는 우리 서비스의 "계정 삭제" 버튼을 누르게 된다.

투명 셀로판 뒤에 다른 버튼이 숨어 있는 거다.

`X-Frame-Options: DENY`를 설정하면 브라우저가 해당 페이지를 iframe에 렌더링하는 것 자체를 거부한다. REST API 서버는 화면을 제공하지 않으므로 `DENY`가 적절하다.

---

## X-Content-Type-Options — MIME 스니핑 방어

서버가 `Content-Type`을 명시하지 않거나 잘못 설정하면, 브라우저가 응답 본문을 "추측"해서 해석한다. MIME 스니핑이라고 한다.

공격자가 `.txt` 확장자로 파일을 업로드하는데, 안에 JavaScript 코드를 넣어둔다. 서버는 `text/plain`으로 응답하지만, 브라우저가 "어? 이거 HTML 같은데?"하고 스크립트를 실행해버린다. 라벨에 "물"이라고 적혀 있는데 냄새 맡아보고 "이거 식초 같은데" 하면서 식초로 쓰는 거다.

`X-Content-Type-Options: nosniff`를 설정하면 브라우저는 서버가 보낸 Content-Type을 무조건 신뢰한다. 추측 금지.

## HSTS — HTTPS 강제

HSTS(HTTP Strict Transport Security)는 브라우저에게 "앞으로 이 도메인은 HTTPS만 써라"라고 명령하는 헤더다.

이게 왜 필요한가. 사용자가 `http://`로 접속하면, 서버가 `https://`로 리다이렉트한다. 그런데 이 리다이렉트 전 구간은 평문 HTTP다. 중간자(MITM) 공격자가 이 리다이렉트를 가로채고 평문 HTTP 통신을 유지시킬 수 있다. SSL Stripping이라고 한다.

HSTS를 한번 수신한 브라우저는 `max-age` 기간 동안 해당 도메인에 `http://`로 접속 자체를 시도하지 않는다. 브라우저 레벨에서 강제로 `https://`로 바꾼다.

```
Strict-Transport-Security: max-age=31536000; includeSubDomains
```

- `max-age=31536000` : 1년간 HTTPS 강제
- `includeSubDomains` : 하위 도메인에도 동일 적용

## CORS 설정

### 동일 출처 정책과 CORS

브라우저에는 Same-Origin Policy가 기본으로 적용된다. `https://a.com`에서 로드된 JavaScript는 `https://b.com`으로 요청을 보낼 수 없다. 도메인이 다르니까. CORS(Cross-Origin Resource Sharing)는 이 정책에 예외를 두는 허가증이다.

Origin은 Protocol + Host + Port 세 가지로 구성된다. 셋 중 하나라도 다르면 "다른 Origin"이다.

### 와일드카드 vs 화이트리스트

| 설정 | 와일드카드 (`*`) | 화이트리스트 (명시) |
|------|-----------------|-------------------|
| Origin | 모든 사이트 허용 | 프로덕션 도메인만 허용 |
| Headers | 모든 헤더 허용 | 필요한 헤더만 허용 |
| Credentials | 사용 불가 (`*`와 호환 안됨) | 사용 가능 |
| 보안 | 취약 | 안전 |

`allowedOrigins("*")`에 `allowCredentials(true)`를 동시에 쓰면 브라우저가 의도적으로 차단한다. 아무 사이트에서나 인증된 요청을 보낼 수 있게 되니까.

### Preflight 요청

브라우저는 실제 요청 전에 `OPTIONS` 메서드로 사전 확인을 한다. "이 요청 보내도 되나?" 확인하는 거다. 매번 OPTIONS를 보내면 낭비니까 `maxAge`로 결과를 캐싱한다.

```
1. 브라우저 → OPTIONS /api/v1/recipes (Preflight)
2. 서버 → 200 OK + CORS 허용 헤더
3. 브라우저 → POST /api/v1/recipes (실제 요청)
```

`maxAge(3600L)` 설정이면 1시간 동안 같은 요청에 대해 Preflight를 생략한다.

### 헤더 와일드카드의 위험

```java
// 이러면 안 된다
config.setAllowedHeaders(List.of("*"));
```

모든 헤더를 허용하면 `X-Forwarded-For`, `X-Real-IP` 같은 헤더를 조작해서 IP 스푸핑이 가능해진다. 필요한 헤더만 명시적으로 열어야 한다.

```java
config.setAllowedHeaders(List.of("Authorization", "Content-Type", "Accept"));
```

## Webhook 서명이 필요한 이유

Webhook 엔드포인트는 외부 서버(PG사 등)가 접근해야 하므로 `permitAll`로 열어야 한다. 문이 열려 있는데 초인종 소리만 듣고 아무나 들여보내는 상황이다.

서명 검증이 없으면 공격자가 `orderId`만 알면 가짜 결제 완료 이벤트를 보낼 수 있다. 돈을 안 내고 상품을 받는 거다. [인앱결제 서버 설계](/posts/인앱결제-서버-설계-패턴/)에서 다룬 멱등성만으로는 부족하다. 멱등성은 "중복 처리 방지"지 "위조 방지"가 아니다.

```java
// 서명 검증 없는 Webhook — 누가 보냈든 무조건 처리
@PostMapping("/webhook")
public ApiResponse<Void> handleWebhook(@RequestBody Map<String, String> payload) {
    orderService.complete(payload.get("orderId"));  // 치명적
    return ApiResponse.ok();
}
```

## HMAC-SHA256 서명 구조

HMAC(Hash-based Message Authentication Code)은 공유 비밀키로 메시지의 무결성과 인증을 동시에 보장하는 방식이다.

편지 봉투에 발신자만 아는 특수 잉크로 봉인 마크를 찍는 거다. 수신자도 같은 잉크를 가지고 있으니 봉인이 진짜인지 확인할 수 있다.

### 검증 흐름

1. PG사와 우리 서버가 **동일한 비밀키(Shared Secret)**를 사전에 공유한다
2. PG사: 요청 body를 비밀키로 해싱 → 서명 생성 → 헤더에 포함하여 전송
3. 우리 서버: 수신한 body를 같은 비밀키로 해싱 → 계산된 서명과 헤더 서명 비교
4. 비밀키를 모르는 공격자는 올바른 서명을 생성할 수 없다

```java
private String generateSignature(String body) {
    Mac mac = Mac.getInstance("HmacSHA256");
    SecretKeySpec keySpec = new SecretKeySpec(
            webhookSecret.getBytes(UTF_8), "HmacSHA256");
    mac.init(keySpec);
    byte[] hash = mac.doFinal(body.getBytes(UTF_8));
    return Base64.getEncoder().encodeToString(hash);
}
```

서명이 일치하면 "PG사가 보낸 진짜 요청"이다. 불일치하면 위조. 401 반환.

### 타임스탬프 검증

서명이 있어도 Replay Attack은 막을 수 없다. 정상 요청을 네트워크에서 캡처해서 그대로 재전송하면 서명은 유효하다. 그래서 타임스탬프를 검증한다. 5분 이상 된 요청은 거부.

```java
long eventTimestamp = event.getTimestamp();
if (Math.abs(System.currentTimeMillis() - eventTimestamp) > 5 * 60 * 1000) {
    throw new BusinessException(ErrorCode.UNAUTHORIZED);
}
```

서명 + 타임스탬프 + 멱등성 키. 이 세 가지가 Webhook의 3단 방어다.

## Timing Attack 방어

서명 검증의 마지막 단계에서 두 문자열을 비교할 때, 비교 방식 자체가 보안 취약점이 된다.

금고 비밀번호 4자리를 맞추는 상황을 생각한다. 첫 자리가 틀리면 0.1초 만에 "틀렸습니다"가 뜨고, 세 자리가 맞으면 0.3초 후에 뜬다면? 공격자는 응답 시간 차이만으로 비밀번호를 한 자리씩 추론할 수 있다. Timing Attack이다.

### `String.equals()` vs `MessageDigest.isEqual()`

| 비교 방식 | 동작 | 보안 |
|-----------|------|------|
| `String.equals()` | 첫 불일치 발견 시 즉시 false 반환 | 취약 — 응답 시간으로 정보 유출 |
| `Arrays.equals()` | 첫 불일치 발견 시 즉시 false 반환 | 취약 — 동일한 단축 평가 |
| `MessageDigest.isEqual()` | 모든 바이트를 끝까지 비교 | 안전 — 일치 여부와 무관하게 일정 시간 |

```java
// 취약 — 절대 사용 금지
if (expectedSignature.equals(receivedSignature)) { ... }

// 안전 — 반드시 이 방식
if (MessageDigest.isEqual(
        expectedSignature.getBytes(UTF_8),
        receivedSignature.getBytes(UTF_8))) { ... }
```

`MessageDigest.isEqual()`은 내부적으로 XOR 기반 상수 시간 비교를 수행한다. 모든 바이트를 순회한 뒤 최종 결과를 반환하므로, 일치 여부와 관계없이 실행 시간이 동일하다. 공격자가 수천 번 요청해서 응답 시간 통계를 내도 아무 정보를 얻을 수 없다.

## 보안 헤더 + CORS 정리

| 보안 헤더 | 방어 대상 | 설정값 |
|----------|----------|-------|
| X-Frame-Options | 클릭재킹 | `DENY` |
| X-Content-Type-Options | MIME 스니핑 | `nosniff` |
| Strict-Transport-Security | SSL 다운그레이드 | `max-age=31536000; includeSubDomains` |

| CORS 설정 | 권장 | 주의 |
|----------|------|------|
| allowedOrigins | 프로덕션 도메인만 명시 | `*`는 credentials와 동시 사용 불가 |
| allowedHeaders | 필요한 헤더만 명시 | `*`는 헤더 조작 공격 가능 |
| maxAge | 3600 (1시간) | Preflight 캐싱으로 성능 절약 |

## Webhook 방어 정리

| 방어 수단 | 목적 | 핵심 |
|---------|------|------|
| HMAC-SHA256 서명 | 위조 방지 | 공유 비밀키로 body 해싱 |
| 타임스탬프 검증 | Replay Attack 방어 | 5분 이상 된 요청 거부 |
| 멱등성 키 | 중복 처리 방지 | eventId로 처리 여부 확인 |
| Timing-Safe 비교 | 서명 정보 유출 방지 | `MessageDigest.isEqual()` 사용 |
